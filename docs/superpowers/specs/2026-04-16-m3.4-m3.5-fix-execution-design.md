# M3.4 & M3.5 设计：Issue 自动修复与 PR 创建

> 日期：2026-04-16
> 状态：已确认
> 依赖：M3.1-M3.3（Issue 分析，已完成）、M3.3.1（REST API + dtw，已完成）

## 1. 概述

Phase 3 第二轮：在 M3.1-M3.3 的只读分析基础上，引入容器写权限，实现 Issue 自动修复并创建 PR 的完整闭环。

**里程碑切分策略**：按层切分（非按阶段），M3.4 纯基础设施，M3.5 纯业务逻辑。

| 里程碑 | 职责 | 核心交付 |
|--------|------|---------|
| M3.4 | 基础设施层 | 两级镜像 + Pool 多镜像 + TaskType 拆分 + entrypoint 写模式 + Webhook 扩展 |
| M3.5 | 业务逻辑层 | 修复 Prompt + 容器执行 + PR 创建 + 通知 + 失败处理 |

## 2. 关键设计决策

### 2.1 两级镜像策略

新建 `Dockerfile.worker-full`，在现有 worker 镜像基础上叠加 JDK + Maven/Gradle 编译环境。

- 目标仓库为全栈项目（Java + Vue），修复需要运行 `mvn test` / `npm test` 验证
- 不引入 Go 工具链
- 两级共享 Node.js + Claude CLI 基础层，Docker layer cache 友好

镜像选择映射：

| TaskType | 镜像 | 原因 |
|----------|------|------|
| `review_pr` | worker（轻量） | 只读分析，无需编译环境 |
| `analyze_issue` | worker（轻量） | 只读分析，无需编译环境 |
| `fix_issue` | worker-full（执行） | 需要编译运行测试 |
| `gen_tests` | worker-full（执行） | 需要编译运行测试（Phase 4） |

### 2.2 标签触发机制

两个独立标签，语义分离：

| 标签 | 触发的 TaskType | 行为 |
|------|----------------|------|
| `auto-fix` | `analyze_issue` | 只读分析，输出根因报告到 Issue 评论 |
| `fix-to-pr` | `fix_issue` | 修复代码 + 运行测试 + push + 创建 PR |

**优先级**：`fix-to-pr` > `auto-fix`。两标签并存时，添加 `fix-to-pr` 只触发修复，不重复触发分析。

### 2.3 TaskType 拆分

将现有 `TaskTypeFixIssue`（实际做分析）拆分为语义正确的两个类型：

```go
TaskTypeAnalyzeIssue TaskType = "analyze_issue"  // 新增，M3.1-M3.3 的分析流程
TaskTypeFixIssue     TaskType = "fix_issue"       // 含义变更，M3.4-M3.5 的修复流程
```

历史数据库中的 `fix_issue` 记录不回填，保持原样。

### 2.4 单容器全托管执行模型

Claude 在一次容器调用中完成全流程：读 Issue 评论 → 分析/复用前序分析 → 修复代码 → 写测试 → 运行测试 → 创建分支 + commit + push。

选择单容器而非双容器的原因：
- Claude Code CLI 的核心优势是自主探索和迭代，测试失败时自行修正
- 保持上下文连续性，避免双容器间的上下文断裂
- 与 review/analyze 的 Service 模式架构一致
- Token 利用率更高

### 2.5 前序分析结果引用

修复容器中的 Claude 通过阅读 Issue 评论获取前序分析报告（如有），而非从数据库读取。理由：
- Issue 评论是公开可见信息，不依赖内部实现
- Claude 可自主判断前序分析是否仍然适用

无前序分析时，Claude 隐式执行分析再修复。

### 2.6 PR 创建策略（混合模式）

容器内完成代码修改 + commit + `git push`；容器外由 `fix.Service` 通过 Gitea API 创建 PR。

- 仅需保留 Git push 凭证，不需要给容器 Gitea API Token
- PR 描述模板化（结构化操作更可控）
- 修复 PR 自动评审：Gitea PR created Webhook 自然触发 Phase 2 评审流程，无需额外代码

### 3.2 Pool 多镜像支持

```go
type PoolConfig struct {
```

```go
func (p *Pool) resolveImage(taskType model.TaskType) string {
    switch taskType {
    case model.TaskTypeFixIssue, model.TaskTypeGenTests:
        if p.config.ImageFull != "" {
            return p.config.ImageFull

    return p.config.Image
```

3. `ImageExists` 检查使用同一个 resolved 镜像名

- `ImageFull` 为空时所有任务用 `Image`（向后兼容）
- 配置校验：`ImageFull` 非空时检查格式合法性
- `PoolConfig.Validate()` 不强制 `ImageFull` 必填

配置示例：


### 3.4 entrypoint.sh 修复模式

```bash
fix_issue)
    if [ -n "${ISSUE_REF:-}" ]; then
        log "checkout 到关联 ref: ${ISSUE_REF}"
        git fetch origin "${ISSUE_REF}" 2>&1 >&2
        git checkout FETCH_HEAD 2>&1 >&2
    fi
    # 修复模式：配置凭证缓存（内存模式，1 小时超时），允许 push
    git config --global credential.helper 'cache --timeout=3600'
    git config --global user.name "DTWorkflow Bot"
    git config --global user.email "dtworkflow-bot@noreply.local"
    # Maven/Gradle 缓存重定向到 /workspace（避免 /tmp tmpfs 溢出）
    export MAVEN_OPTS="${MAVEN_OPTS:--Dmaven.repo.local=/workspace/.m2/repository}"
    export GRADLE_USER_HOME="${GRADLE_USER_HOME:-/workspace/.gradle}"
    log "修复模式已启用（credential cache + git identity + build cache redirect）"
    ;;

analyze_issue)
    # 只读分析模式（从原 fix_issue 搬迁）
    if [ -n "${ISSUE_REF:-}" ]; then
        log "checkout 到关联 ref: ${ISSUE_REF}"
        git fetch origin "${ISSUE_REF}" 2>&1 >&2
        git checkout FETCH_HEAD 2>&1 >&2
    fi
    ;;
```

**凭证安全模型（已接受风险）**：

修复模式使用 `credential.helper cache`（内存缓存，非 `store` 文件模式），凭证不持久化到磁盘文件。但需要明确的安全边界：

- **已接受风险**：Claude 在修复模式下拥有 Bash 执行能力（需运行测试），理论上可通过 `echo "protocol=https\nhost=..." | git credential fill` 从 cache daemon 提取凭证。这是修复模式授予 push 权限的固有攻击面——任何允许 push 的凭证机制都无法对拥有 Bash 的进程完全隔离
- **缓解措施**：
  1. `credential.helper cache`（非 `store`）：凭证仅在内存中，不写入磁盘文件，降低意外泄露风险
  2. 环境变量照常清除：clone 后 `unset GITEA_TOKEN / AUTH_URL / GITEA_URL / REPO_CLONE_URL`
  3. **建议使用受限 Token**：为修复任务配置专用 Gitea Token，仅授予目标仓库的 push 权限，限制爆炸半径
  4. 容器隔离：ReadonlyRootfs + 独立网络 + PID 限制 + CPU/内存限制仍然生效
- **不设 pre-commit hook / 不锁 push URL**：与 review 模式的关键区别
- **Git identity**：Bot 账户（`DTWorkflow Bot`），commit 可溯源

**入队分流逻辑**（`HandleIssueLabel`）：

```go
// queue/enqueue.go
switch {
```

## 4. M3.5 详细设计：业务逻辑层

### 4.1 修复 Prompt 设计

通过 stdin 传入容器（同 review/analyze 安全实践）。

**四段式结构**：

```
[上下文段]
- 仓库 {repo}，Issue #{id}，标题：{title}
- 当前代码基于 ref: {ref}
- 请先阅读该 Issue 的评论，了解是否有之前的分析报告可以参考

[修复指令段]
1. 理解问题：阅读 Issue 描述和评论，定位根因
2. 修复代码：创建分支 auto-fix/issue-{id}，修改最小必要范围
3. 补充测试：为修复点添加/修改单元测试，验证修复有效
4. 运行测试：执行 mvn test 或 npm test，确保全部通过
5. 如果测试失败，根据错误信息修正后重试（最多 3 轮）
6. 全部通过后 commit + push

[约束段]
- 不修改与 Issue 无关的文件
- 不删除现有测试
- commit message 格式：fix: #{issue_id} {简述}
- 分支名：auto-fix/issue-{id}

[输出格式段]
JSON schema（FixOutput，见 4.2）
```

安全约束：fix 模式不加 `--disallowedTools` 的 Bash 限制（需运行测试），保留 prompt 层网络访问禁令。

### 4.2 修复输出 JSON Schema

```go
type FixOutput struct {
    Success        bool         `json:"success"`
    InfoSufficient bool         `json:"info_sufficient"`
    MissingInfo    []string     `json:"missing_info,omitempty"`
    BranchName     string       `json:"branch_name"`
    CommitSHA      string       `json:"commit_sha"`
    ModifiedFiles  []string     `json:"modified_files"`
    TestResults    *TestResults `json:"test_results,omitempty"`
    Analysis       string       `json:"analysis"`
    FixApproach    string       `json:"fix_approach"`
}

type TestResults struct {
    Passed    int  `json:"passed"`
    Failed    int  `json:"failed"`
    Skipped   int  `json:"skipped"`
    AllPassed bool `json:"all_passed"`
}
```

- `InfoSufficient=false` 时 `Success` 必为 `false`，Claude 不执行修复
- `Success=true` 必须伴随非空 `BranchName` 和 `CommitSHA`

### 4.3 fix.Service 扩展

同一个 `Execute` 方法，按 `payload.TaskType` 内部路由。当前 `service.go:94` 的 `Execute` 直接走分析流程，需在方法入口添加 switch：

```go
func (s *Service) Execute(ctx context.Context, payload model.TaskPayload) (*FixResult, error) {
    switch payload.TaskType {
    case model.TaskTypeAnalyzeIssue:
        return s.executeAnalysis(ctx, payload) // 现有分析流程搬入
    case model.TaskTypeFixIssue:
        return s.executeFix(ctx, payload)      // 新修复流程
    default:
        return nil, fmt.Errorf("fix.Service 不支持任务类型: %s", payload.TaskType)
    }
}
```

**修复流程 `executeFix`**：

```
executeFix(payload)
  1. 前置校验（Issue open + Ref 有效）
  2. Ref 类型检查：validateRef 区分分支 vs tag
  3. "信息不足"前置检查（查 DB 最新 analyze_issue 结果）
  4. 采集 Issue 上下文（复用 collectContext）
  5. 构造修复 prompt + 容器执行（full image）
  6. 双层 JSON 解析：CLI 信封 → FixOutput
  7. FixOutput 校验：success=true 时 BranchName 和 CommitSHA 不可为空
  8. 按结果分流：
     - success=true → 创建 PR + Issue 评论
     - info_sufficient=false → Issue 评论提醒
     - success=false → Issue 评论报告失败
```

### 4.4 "信息不足"前置检查

容器启动前查数据库：

```
最新 analyze_issue 任务存在 && result 中 info_sufficient == false
  → Issue 评论："之前的分析表明信息不足（缺少：{missing_info}），
     建议补充信息后重新添加 auto-fix 标签更新分析，再添加 fix-to-pr 标签触发修复。"
  → SkipRetry，不启动容器
```

- 无前序分析 → 不阻拦，Claude 隐式分析
- 用户补充信息后 → 需先加 `auto-fix` 更新分析结论，再加 `fix-to-pr`

### 4.5 PR 创建

容器执行成功（`success=true` + `branch_name` 非空）后：

```go
pr, _, err := s.prClient.CreatePullRequest(ctx, owner, repo, gitea.CreatePullRequestOption{
    Head:   fixOutput.BranchName,         // auto-fix/issue-{id}
    Base:   resolvedBase,                  // 分支名（见下方 Tag 处理）
    Title:  fmt.Sprintf("fix: #%d %s", issueNum, issueTitle),
    Body:   buildFixPRBody(fixOutput, issueNum),
})
```

**Tag 作为 Ref 的边界处理**：Gitea `CreatePullRequest` API 的 `Base` 字段不支持 tag（只支持分支名）。`validateRef` 需要在校验 ref 有效性时记录 ref 类型（分支 vs tag）：
- **Ref 是分支**：`Base` 直接使用分支名
- **Ref 是 tag**：`Base` 使用仓库默认分支（通过 Gitea API `GetRepo` 获取 `DefaultBranch`），prompt 中仍注入 tag 信息让 Claude 基于正确的代码版本修复。注意：此时 PR 的 base 与 Claude 实际 checkout 的代码可能不同（tag commit vs 默认分支 HEAD），需在 PR 描述中注明

```markdown
fixes #{issue_id}

{analysis}
```

### 4.7 失败处理

| 场景 | 行为 |
|------|------|
| Issue 已关闭 / Ref 无效 / Ref 缺失 | 同分析模式，Issue 评论提醒 + SkipRetry |
| DB 前置检查：前序分析"信息不足" | Issue 评论提醒补充信息 + SkipRetry |
| 容器执行失败（进程错误） | Issue 评论报告错误 + 允许 asynq 重试 |
| Claude 返回 `info_sufficient=false` | Issue 评论列出缺失信息 + SkipRetry |
| Claude 返回 `success=false`（测试失败等） | Issue 评论报告失败原因 + SkipRetry |
| Push 成功但 PR 创建失败 | Issue 评论："分支已推送但 PR 创建失败" + 允许重试 |
| 分支已存在（前次修复残留） | prompt 中指示 Claude：分支已存在时 force push 覆盖（`git push --force`），确保重试幂等 |
| 修复 PR 自动评审 | 无额外代码——Gitea Webhook 自然触发 Phase 2 |

### 4.9 边界场景与风险

**分支冲突处理**：`auto-fix/issue-{id}` 分支可能因前次修复失败而残留。prompt 中指示 Claude 使用 `git push --force-with-lease`（优于 `--force`，可检测他人并发 push）。若 push 仍失败，返回 `success=false` 由失败处理链上报。

**并发修复同一 Issue**：两个 `fix-to-pr` 事件几乎同时到达时，参照 M2.4 Cancel-and-Replace 机制——新任务入队后取消同一 Issue 的旧 `fix_issue` 活跃任务。需在 `store` 层新增 `FindActiveIssueTasks(repoFullName, issueNumber, taskType)` 查询方法。

**FixOutput 校验不变量**：`fix.Service` 解析 `FixOutput` 后校验：`success=true` 时 `BranchName` 和 `CommitSHA` 必须非空，否则视为解析异常走降级流程（Issue 评论展示原始输出）。

**`--output-format json` 参数**：修复模式的 `buildCommand` 必须包含 `--output-format json`（或 stream-json），确保容器输出可被双层 JSON 解析。与分析模式一致。

**dtw 瘦客户端适配**：`dtw fix-issue` 命令新增 `--fix` flag 区分触发修复（默认仍为分析）。`EnqueueManualFix` 函数根据 flag 使用 `TaskTypeAnalyzeIssue` 或 `TaskTypeFixIssue`。
