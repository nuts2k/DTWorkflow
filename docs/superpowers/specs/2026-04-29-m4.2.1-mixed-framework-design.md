# M4.2.1 混合仓库多框架支持设计文档

## 背景

当前 `test_gen.test_framework` 只能设单个值（`junit5` / `vitest`），混合仓库（前后端一体）
在整仓模式（`--module` 留空）下触发 gen_tests 时，`resolveFramework` 只检查仓库根目录的
`pom.xml` / `package.json`：

- 根目录两者都没有 → `ErrNoFrameworkDetected`（实际子目录存在可测试模块）
- 根目录两者都有 → `ErrAmbiguousFramework`

用户目前的唯一选择是手动分 module 多次触发，效率低。

## 目标

1. 整仓模式自动发现 depth=1 子目录中的可测试模块
2. 多模块自动拆分为独立 gen_tests 任务，各自产出独立 PR
3. 为 M4.3（变更驱动测试生成）铺路——拆分管线可复用

## 设计决策

以下决策在 brainstorming 中确认：

1. **入队层拆分**：扫描和拆分逻辑放在 `EnqueueManualGenTests`，`test.Service.Execute` 不变。每个子任务复用现有完整管线。
2. **完全独立任务**：拆分后的子任务互不感知，各自有独立的 Cancel-and-Replace 生命周期、飞书通知、分支和 PR。不引入父子任务模型。
3. **仅整仓模式触发扫描**：`--module` 非空时不做 depth=1 扫描，保持现有向上回溯逻辑。
4. **每个子目录一个任务**：同框架的多个子目录不合并（`service-a` + `service-b` 各自独立任务），粒度最细。
5. **整仓触发 = 全量清理 + 重新拆分**：整仓拆分入队前清理该仓库所有 `auto-test/*` 旧资源（关闭 PR + 删除远程分支 + 取消活跃任务），解决拆分策略变化后旧资源成为孤儿的问题。
6. **清理只涉及 Gitea 临时资源**：不合并任何 PR、不碰 `auto-test/*` 以外的分支、不修改仓库已有测试文件。清理操作前记录 dry-run 日志用于审计。
7. **显式 module + ambiguous 仍报错**：`--module` 非空时遇到 `ErrAmbiguousFramework` 保持 SkipRetry，提示用户指定 `--framework` 或细化 module。
8. **不新增配置项**：扫描为自动行为，depth=1 为固定策略，全量清理为整仓触发的固有语义。

## 目标仓库结构

确认用户仓库为 A 类型——根下直接有若干子目录，每个子目录是独立的 Maven 模块或 npm 项目：

```
repo-root/
  backend/
    pom.xml
    src/main/java/...
  frontend/
    package.json
    src/components/...
```

depth=1 扫描即可覆盖。

## 详细设计

### 1. 多模块扫描（internal/test/scan.go，新建）

#### 1.1 类型定义

```go
// DiscoveredModule 扫描发现的可测试模块。
type DiscoveredModule struct {
    Path      string    // 子目录相对路径（如 "backend"、"frontend"）
    Framework Framework // 检测到的框架
}
```

#### 1.2 RepoFileChecker 接口扩展

```go
type RepoFileChecker interface {
    HasFile(ctx context.Context, owner, repo, ref, module, relPath string) (bool, error)
    ListDir(ctx context.Context, owner, repo, ref, dir string) ([]string, error) // 新增
}
```

`ListDir` 返回指定目录下的子目录名列表（不含文件，不递归）。生产实现对接
Gitea Contents API（`GET /repos/{owner}/{repo}/contents/{path}?ref={ref}`，
过滤 `type=="dir"` 的条目）。

#### 1.3 ScanRepoModules 函数

```go
func ScanRepoModules(ctx context.Context, checker RepoFileChecker,
    owner, repo, ref string) ([]DiscoveredModule, error)
```

扫描逻辑：

1. 先检查仓库根（`dir=""`）的 `pom.xml` / `package.json`
   - 只有一个 → 返回单条 `{Path: "", Framework: ...}`（根即项目，不做 depth=1）
   - 两个都有 → 返回两条 `{Path: "", Framework: junit5}` + `{Path: "", Framework: vitest}`
   - 都没有 → 继续 depth=1 扫描
2. 调用 `ListDir(ctx, owner, repo, ref, "")` 获取根下子目录列表
3. 对每个子目录检查 `pom.xml` / `package.json`：
   - 只有 `pom.xml` → 追加 `{Path: subdir, Framework: junit5}`
   - 只有 `package.json` → 追加 `{Path: subdir, Framework: vitest}`
   - 两者都有 → 追加两条（同目录双框架拆分）
   - 都没有 → 跳过
4. 结果为空 → 返回 `ErrNoFrameworkDetected`

错误处理：
- `ListDir` 失败 → 返回 error，上层 fail-open 回退到现有单任务逻辑
- 单个子目录 `HasFile` 失败 → 跳过该子目录（warn 日志），继续扫描其他
- 子目录数量超过上限（`maxScanDirs = 30`）→ 截断并记 warn 日志，只处理前 30 个

已知限制：
- depth=1 仅覆盖根下直接子目录；嵌套结构（如 `apps/backend/`）需用户手动指定
  `--module apps/backend`。后续可按需扩展为可配置 depth。

### 2. 入队层拆分（internal/queue/enqueue.go）

#### 2.1 EnqueueManualGenTests 签名变更

```go
// 返回值从 (string, error) 改为 ([]string, error)
func (h *EnqueueHandler) EnqueueManualGenTests(
    ctx context.Context, payload model.TaskPayload, triggeredBy string,
) ([]string, error)
```

#### 2.2 拆分流程（整仓模式，module 空 + framework 空）

前置条件：`payload.BaseRef` 必须在调用 `EnqueueManualGenTests` 前由上层填充
（API handler 通过 Gitea `GetRepo` 查询默认分支后设置；CLI handler 从 `--ref` flag 或
同样查询 Gitea 后设置）。`payload.BaseRef` 为空时跳过扫描，走现有单任务逻辑（fail-open）。

```
payload.Module == "" && payload.Framework == ""
  │
  ├─ payload.BaseRef 为空？ → 跳过扫描，走现有单任务逻辑
  ├─ ScanRepoModules(ctx, checker, owner, repo, payload.BaseRef)
  │
  ├─ err != nil → warn + 回退到现有单任务逻辑（fail-open）
  ├─ len == 0  → ErrNoFrameworkDetected
  └─ len >= 1  → 拆分入队
       │
       ├─ len >= 2 时先执行 cleanupAllAutoTestBranches(repo)
       │  （单模块不需要全量清理——不存在策略变化冲突）
       │
       └─ for _, mod := range discovered:
            subPayload := payload (浅拷贝)
            subPayload.Module = mod.Path
            subPayload.Framework = string(mod.Framework)
            taskID, err := h.enqueueSingleGenTests(ctx, subPayload, triggeredBy)
            // enqueueSingleGenTests = 提取的现有单任务入队逻辑
```

`enqueueSingleGenTests` 封装现有的单任务入队逻辑（Cancel-and-Replace by branch key +
入队 + 日志），避免拆分循环中重复代码。

#### 2.3 cleanupAllAutoTestBranches

新增方法，整仓拆分入队前调用：

```go
func (h *EnqueueHandler) cleanupAllAutoTestBranches(
    ctx context.Context, owner, repo, repoFullName string,
) error
```

步骤：
1. 查询该仓库所有活跃 gen_tests module 列表（`store.ListActiveGenTestsModules(ctx, repoFullName)`），
   再逐个调用 `store.FindActiveGenTestsTasks(ctx, repoFullName, module)` 取回任务记录 → 全部 cancel。
   此模式与现有 `listActiveGenTestsTasksByBranchKey` 一致，无需新增 store 方法。
2. 通过 PRClient 调用 `ListRepoPullRequests(ctx, owner, repo, {State: "open"})` **一次性**
   列出所有 open PR，在内存中按 `auto-test/*` head ref 前缀过滤，逐个关闭 PR + 删除远程分支。
   避免逐分支重复调用列表 API。
3. 每步执行前记录 dry-run 日志：即将取消的任务 ID、即将关闭的 PR 编号、即将删除的分支名
4. 任一步骤失败仅 warn，不阻断后续入队

实现方式：在 `cleanupAllAutoTestBranches` 内部直接调用 `PRClient.ListRepoPullRequests` +
`PRClient.ClosePullRequest` + `BranchCleaner.DeleteBranch`（或 Gitea DeleteBranch API），
不新增 BranchCleaner 接口方法。`EnqueueHandler` 已持有 `branchCleaner`，新增对
`PRClient` 的引用（通过 `WithPRClient` option 注入，或复用现有 branchCleaner 内部的
Gitea client）。

#### 2.4 EnqueueHandler 新增依赖注入

```go
type EnqueueHandler struct {
    // ... 现有字段
    moduleScanner RepoFileChecker // 新增：用于 ScanRepoModules
}

func WithModuleScanner(c test.RepoFileChecker) EnqueueOption {
    return func(h *EnqueueHandler) { h.moduleScanner = c }
}
```

未注入时整仓模式走现有逻辑（不扫描，不拆分）。

### 3. 分支命名扩展

#### 3.1 BuildAutoTestBranchName 变更

```go
func BuildAutoTestBranchName(module string, framework ...string) string {
    key := ModuleKey(module)
    if len(framework) > 0 && framework[0] != "" {
        key = key + "-" + framework[0]
    }
    return "auto-test/" + key
}
```

追加 framework 后缀的条件：
- framework 参数非空（由入队层在多框架拆分场景显式传入）
- module 为空时同样追加（根级双框架场景需要 `auto-test/all-junit5` / `auto-test/all-vitest` 区分）

**单框架场景不传 framework 参数（variadic 为空）**，分支名保持不变（`auto-test/backend`、
`auto-test/all`）。只有多框架拆分时入队层才显式传入 framework，触发后缀追加。

#### 3.2 分支名示例

```
场景                              分支名
─────────────────────────────     ────────────────────────────
单框架、module="backend"          auto-test/backend          （无 framework 参数）
单框架、module=""                 auto-test/all              （无 framework 参数）
整仓扫描拆出 backend + frontend   auto-test/backend          （各自单框架，无后缀）
                                  auto-test/frontend
根级双框架（根有 pom+pkg）        auto-test/all-junit5       （多框架拆分，有后缀）
                                  auto-test/all-vitest
同目录双框架、module="mono"       auto-test/mono-junit5
                                  auto-test/mono-vitest
```

#### 3.3 调用点适配

- `internal/worker/container.go`：`moduleKeyForContainer` 需同步适配 framework 后缀逻辑。
  入队层在 `payload` 中同时设置了 `Module` 和 `Framework`，container.go 可据此判断是否
  追加后缀。新增 helper：
  ```go
  func moduleKeyForContainerWithFramework(module, framework string) string
  ```
  framework 非空时追加后缀；其他场景保持现有行为。
  `buildContainerEnv` 中 `MODULE_SANITIZED` 注入改为调用此函数。

  **一致性测试要求**：`internal/worker/container_test.go` 中新增对照测试，
  assert `moduleKeyForContainerWithFramework(m, f) == strings.TrimPrefix(test.BuildAutoTestBranchName(m, f), "auto-test/")`
  对各种输入组合成立（空 module、空 framework、双框架、特殊字符等）。
  这是维护 worker ↔ test 两份实现一致性的最低保障（worker 不能 import internal/test，
  但测试文件可以）。

- `build/docker/entrypoint.sh`：`MODULE_SANITIZED` 环境变量由 container.go 注入，
  entrypoint 直接使用 `auto-test/${MODULE_SANITIZED}`，无需改动 entrypoint 自身逻辑。

- `internal/test/prompt.go` `writeHeader`：当前调用
  `BuildAutoTestBranchName(ctx.Module)`，双框架场景会显示错误的分支名。修复：
  - `PromptContext` 新增 `Framework string` 字段
  - `writeHeader` 改为 `BuildAutoTestBranchName(ctx.Module, ctx.Framework)`
  - `service.go` Execute 中构造 `PromptContext` 时填充 `Framework: string(framework)`
  - 单框架场景 `payload.Framework` 为空，variadic 不生效，行为不变

- Cancel-and-Replace branch key 调用点（`internal/queue/enqueue.go`）：
  `test.BuildAutoTestBranchName(payload.Module)` 需改为
  `test.BuildAutoTestBranchName(payload.Module, payload.Framework)` 以正确匹配
  framework 后缀分支。涉及的具体位置：
  - `enqueueSingleGenTests`（提取后的函数）内部 Cancel-and-Replace 的 branch key 计算
  - `CleanupAutoTestBranch` 调用处的分支名构造
  - `listActiveGenTestsTasksByBranchKey` 中 `test.ModuleKey(module)` 的 target key
    计算——需扩展为考虑 framework 维度：当 `payload.Framework` 非空时 target key 为
    `ModuleKey(module) + "-" + framework`

### 4. resolveFramework 适配

**M4.2.1 不改动 `resolveFramework`**。`ErrAmbiguousFramework` 保持原样。

理由：`ScanRepoModules` 直接通过 `HasFile` 检测框架并拆分为多条 `DiscoveredModule`，
不经过 `resolveFramework`。子任务入队时 `payload.Framework` 已显式设置，`Execute` 中
`resolveFramework` 走"显式配置优先"分支直接返回。

显式 `--module mono` 且 mono 目录下双框架的场景，仍返回 `ErrAmbiguousFramework` →
SkipRetry，提示用户指定 `--framework` 或细化 module。行为不变。

`MultiFrameworkError`（携带结构化框架列表的错误类型）推迟到 M4.3——M4.3 的变更驱动
可能需要在显式 module 场景也做自动拆分，届时再引入该类型并消费其 `Frameworks` 字段。

### 5. 通知与用户反馈

#### 5.1 飞书通知

不新增事件类型。每个子任务各自走现有 Started / Done / Failed 三事件通知。
M4.2.1 不做拆分汇总通知，后续按需补充。

#### 5.2 CLI / API 返回值

API JSON 响应：
```json
{
  "split": true,
  "tasks": [
    {"task_id": "abc123", "module": "backend", "framework": "junit5"},
    {"task_id": "def456", "module": "frontend", "framework": "vitest"}
  ]
}
```

单任务场景（未拆分）：
```json
{
  "split": false,
  "tasks": [
    {"task_id": "abc123", "module": "", "framework": ""}
  ]
}
```

CLI 人类可读输出（拆分场景）：
```
整仓扫描发现 2 个可测试模块，已拆分入队：
  [1] module=backend  framework=junit5  task_id=abc123
  [2] module=frontend framework=vitest  task_id=def456
```

#### 5.3 dtw 瘦客户端

- `--no-wait` 模式：打印拆分信息后退出
- 非 `--no-wait` 模式：提示"已拆分为 N 个子任务，请使用 `dtw task list` 查看各任务进度"
  （不逐个轮询）

### 6. 向后兼容性

| 场景 | M4.2.1 前 | M4.2.1 后 |
|------|-----------|-----------|
| `--module backend`（单框架） | 正常执行 | 不变 |
| `--module ""`，根有 `pom.xml` | 正常执行 | 不变（根级检测走 ScanRepoModules 步骤 1，返回单条记录） |
| `--module ""`，根无标记文件 | `ErrNoFrameworkDetected` | 自动扫描 depth=1 → 拆分入队 |
| `--module "" --framework junit5` | 正常执行（显式框架） | 不变（跳过扫描） |
| `--module ""`，根有 `pom.xml` + `package.json` | `ErrAmbiguousFramework` (SkipRetry) | 自动拆分为两个子任务（`auto-test/all-junit5` + `auto-test/all-vitest`） |
| `--module mono`，双框架 | `ErrAmbiguousFramework` | 仍报错 SkipRetry（行为不变） |

仅"整仓 + 根级双框架"场景存在行为变更（从报错变为自动拆分），属正面改进。其他场景不存在行为回退。

### 7. 安全约束

- 全量清理仅操作 `auto-test/*` 前缀的 PR 和分支，不触碰其他 PR 或分支
- 全量清理是"关闭"PR 而非"合并"，不会将代码写入主分支
- 不修改仓库中任何已有文件（现有 prompt `noOverwriteInstruction` 约束不变）
- 清理操作前记录 dry-run 日志（即将操作的 PR 编号和分支名），便于事后审计
- 清理任一步骤失败仅 warn，不阻断后续入队

## M4.3 复用路径

M4.3（变更驱动测试生成）的输入是 PR merged 事件的变更文件列表。复用路径：

1. 解析变更文件 → 按目录归组 → 对每组调用 `resolveFramework` 确定框架
2. 构造 `[]DiscoveredModule`（与 `ScanRepoModules` 输出格式相同）
3. 调用同一套拆分入队逻辑

M4.2.1 建设的"检测 → 拆分 → 多任务入队"管线，M4.3 只需替换输入源（从目录扫描
换为变更文件分析），无需重建拆分基础设施。
