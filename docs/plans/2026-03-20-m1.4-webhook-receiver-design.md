# M1.4 Webhook 接收器 - 设计文档

> 日期：2026-03-20
> 状态：已评审
> 里程碑：M1.4 Webhook 接收器
> 依赖：M1.2 CLI 框架（已完成）、M1.3 Gitea API 客户端（已完成）

---

## 1. 目标

在 `internal/webhook/` 包中实现 Gitea Webhook 接收器，支持统一接收入口、请求签名校验、PR / Issue 事件解析，以及调用内部 handler 完成后续处理。M1.4 的交付目标是打通“HTTP 接收 → 验签 → 事件解析 → 内部处理器”这条链路，为 M1.5 任务队列、M2 PR 自动评审、M3 Issue 自动修复提供稳定入口。

本阶段**不直接创建任务、不接入 Redis/asynq、不实现完整业务处理**，而是聚焦 Webhook 接入边界和事件标准化。

---

## 2. 设计决策记录

| # | 决策项 | 结论 | 理由 |
|---|--------|------|------|
| 1 | Webhook 对外入口形式 | **单一入口** `/webhooks/gitea` | Gitea 配置简单，部署与文档成本低；内部再按领域分发即可 |
| 2 | 整体架构风格 | **轻量模块化分层**（Receiver / Verifier / Parser / Handler） | 比直接写在 `serve` 中更易测试和扩展；又避免过早抽象成事件总线 |
| 3 | M1.4 完成边界 | **到内部 handler 为止**，不直接入队 | 保持 M1.4 可独立交付，同时为 M1.5 替换为入队 handler 预留清晰边界 |
| 4 | 不支持事件的处理方式 | **区分处理**：坏请求返回 `4xx`，不支持事件返回 `2xx` + 日志 | 避免 Gitea 因无须处理的合法事件反复重试；同时让配置或安全问题尽早暴露 |
| 5 | 验签实现方式 | **独立 HMAC verifier**，基于原始请求体校验 | 安全项是 M1.4 明确范围；独立模块便于单测和未来兼容不同签名头格式 |
| 6 | 事件建模方式 | **内部标准事件模型**，不直接暴露原始 Gitea payload | 降低后续模块对 Gitea payload 细节的耦合；保留当前阶段真正需要的字段 |
| 7 | Pull Request 支持范围 | **仅支持** `pull_request.opened`、`pull_request.synchronized` | 满足 M2 触发需求，避免在 M1.4 扩展过多动作处理 |
| 8 | Issue 支持范围 | **仅支持** `issues.labeled`、`issues.unlabeled` | 满足 M3 `auto-fix` 标签触发场景；后续需要时再补更多动作 |
| 9 | Issue 标签策略 | 保持通用标签变更事件，同时**额外识别 `auto-fix` 增删** | 兼顾事件通用性与 M3 的直接消费需求 |
| 10 | 幂等策略 | M1.4 **不做持久化去重**，仅记录 `delivery_id` | 去重更适合在 M1.5 队列/持久化层实现；当前先保留可观测性 |
| 11 | 测试策略 | verifier / parser 细粒度单测 + receiver 路由测试 | 安全、解析和 HTTP 行为分别验证，便于定位问题 |
| 12 | 配置方式 | Webhook secret 先通过 `serve` 集成点注入，详细配置管理留到 M1.8 | 避免提前引入 Viper；先让接收器具备明确依赖接口 |

## 3. 架构概览

```text
Gitea Webhook              internal/cmd              internal/webhook
┌──────────────┐          ┌──────────────┐         ┌──────────────────────┐
│  Gitea       │ POST     │ serve.go     │         │ router.go            │
│  Webhook     │─────────>│ Gin router   │────────>│ RegisterRoutes()     │
└──────────────┘          └──────────────┘         ├──────────────────────┤
                                                    │ receiver.go          │
                                                    │ - 读取请求/响应映射  │
                                                    ├──────────────────────┤
                                                    │ verifier.go          │
                                                    │ - HMAC 验签          │
                                                    ├──────────────────────┤
                                                    │ parser.go            │
                                                    │ - 事件识别/标准化    │
                                                    ├──────────────────────┤
                                                    │ handler.go           │
                                                    │ - 调用内部处理器     │
                                                    └──────────────────────┘
                                                               │
                                                               ▼
                                                   当前：日志 handler / 占位 handler
                                                   后续：M1.5 入队 handler
```

### 核心原则

- **对外简单**：Gitea 只需配置一个 Webhook URL
- **对内清晰**：HTTP 接入、安全校验、事件解析、后续处理解耦
- **便于替换**：未来将默认日志 handler 替换为任务队列 handler 时，不需要重写接收层
- **可测试**：每一层都可以独立验证，不依赖真实 Gitea 环境

## 4. 模块设计

### 4.1 router.go

职责：
- 向 Gin 注册 `/webhooks/gitea` 路由
- 组装 receiver 依赖（verifier、parser、handler）
- 为 `serve.go` 提供单一接入点

建议入口：

```go
type Config struct {
    Secret  string
    Handler Handler
}

func RegisterRoutes(router gin.IRouter, cfg Config)
```

设计要点：
- `RegisterRoutes` 只做装配，不写业务判断
- 若未显式传入 handler，可使用默认日志 handler
- secret 为空时仍允许注册，但运行时请求会因认证要求失败，避免“表面可用、实际未启用安全”

---

### 4.2 receiver.go

职责：
- 接收 `/webhooks/gitea` 的 HTTP 请求
- 读取 header 和原始 body
- 调用 verifier 做签名校验
- 调用 parser 解析为内部标准事件
- 调用 handler 执行后续处理
- 将内部错误映射为明确的 HTTP 状态码

建议流程：

1. 校验 HTTP 方法与基本请求格式
2. 读取原始 body
3. 调用 verifier 校验签名
4. 读取 event type、delivery ID 等 header
5. 调用 parser 解析为内部事件
6. 调用相应 handler
7. 返回对应 HTTP 响应

设计要点：
- receiver 只负责协议层和流程编排，不持有业务状态
- receiver 不直接关心 Redis、SQLite 或 Gitea API 客户端
- receiver 不直接依赖 Gin 之外的框架特性，便于单测

---

### 4.3 verifier.go

职责：
- 基于 **原始请求体字节** + secret + Gitea 签名头做 HMAC 校验
- 返回是否通过以及失败原因

建议接口：

```go
type Verifier interface {
    Verify(signature string, body []byte) error
}
```

建议错误分类：
- 缺少签名头
- secret 未配置
- 签名格式非法
- HMAC 不匹配

设计要点：
- 必须使用**原始 body** 校验，不能反序列化后再重新编码
- 采用常量时间比较避免时序侧信道问题
- 将签名头解析逻辑封装在 verifier 内，避免 receiver 散落安全细节

---

### 4.4 parser.go

职责：
- 根据 header 中的事件类型和 payload 中的 action 字段解析请求
- 将原始 Gitea payload 转换为内部标准事件模型
- 区分“无效请求”和“合法但忽略的事件”

建议接口：

```go
type Parser interface {
    Parse(eventType, deliveryID string, body []byte) (Event, error)
}
```

其中 `Event` 为内部事件接口：

```go
type Event interface {
    Name() string
}
```

设计要点：
- parser 只负责语义解析，不关心 HTTP 状态码
- parser 识别到不支持事件时返回特定错误，由 receiver 映射为 `200`
- parser 对 payload 必要字段做最小校验，避免后续 handler 再做重复防御性判断

---

### 4.5 handler.go

职责：
- 定义内部处理器接口
- 提供当前阶段默认日志实现

建议接口：

```go
type Handler interface {
    HandlePullRequest(ctx context.Context, event PullRequestEvent) error
    HandleIssueLabel(ctx context.Context, event IssueLabelEvent) error
}
```

默认实现：
- 使用 `slog` 打印结构化日志
- 记录关键字段：`delivery_id`、`event_type`、`action`、`repo`、`pr_number` / `issue_number`、`label`、`auto_fix_changed`

设计要点：
- M1.4 默认实现只做日志，不访问外部系统
- M1.5 可替换为 enqueue handler，接口无需变化
- handler 不依赖 Gin `Context`，只使用标准 `context.Context`

## 5. 事件模型

M1.4 采用**轻量标准化事件模型**，隔离 Gitea 原始 payload 与后续业务逻辑。

### 5.1 公共引用类型

```go
type RepositoryRef struct {
    Owner         string
    Name          string
    FullName      string
    CloneURL      string
    DefaultBranch string
}

type UserRef struct {
    Login    string
    FullName string
}

type LabelRef struct {
    Name  string
    Color string
}
```

### 5.2 PullRequestEvent

```go
type PullRequestEvent struct {
    DeliveryID string
    EventType  string // pull_request
    Action     string // opened, synchronized

    Repository RepositoryRef
    PullRequest PullRequestRef
    Sender      UserRef
    OccurredAt  time.Time
}

type PullRequestRef struct {
    Number  int64
    Title   string
    Body    string
    HTMLURL string

    BaseRef string
    HeadRef string
    BaseSHA string
    HeadSHA string
}
```

保留字段理由：
- M2 评审任务创建需要仓库、PR 编号、分支与 SHA
- 这些字段足以支撑后续获取 diff、拉取文件、创建评审等行为
- 不暴露整个 Gitea payload，减少未来业务层对第三方字段的耦合

### 5.3 IssueLabelEvent

```go
type IssueLabelEvent struct {
    DeliveryID string
    EventType  string // issues
    Action     string // labeled, unlabeled

    Repository RepositoryRef
    Issue      IssueRef
    Label      LabelRef
    Sender     UserRef

    AutoFixChanged bool
    AutoFixAdded   bool
    AutoFixRemoved bool

    OccurredAt time.Time
}

type IssueRef struct {
    Number  int64
    Title   string
    Body    string
    HTMLURL string
    State   string
}
```

保留字段理由：
- M3 触发修复任务需要知道仓库、Issue 编号、当前标签变化
- 保持为通用标签事件，同时显式识别 `auto-fix` 是否发生增删
- 后续 handler 可直接基于 `AutoFixAdded` / `AutoFixRemoved` 决策，无需重复做标签判断

## 6. 请求处理流程

### 6.1 Header 约定

M1.4 假设 Gitea Webhook 会提供以下关键信息：
- 事件类型 header（如 `pull_request`、`issues`）
- delivery ID header（用于日志追踪）
- 签名 header（用于 HMAC 校验）
- `Content-Type: application/json`

具体 header 名称在实现时与 Gitea 文档保持一致，但设计上将其封装在接收层和 verifier 中，避免业务层感知细节。

### 6.2 处理顺序

```text
1. 接收 POST /webhooks/gitea
2. 校验方法、必要 header、Content-Type、body 非空
3. 读取原始 body
4. 基于 secret 和签名 header 做 HMAC 验签
5. 识别 event type
6. 解析 payload 中的 action 与实体信息
7. 转换为内部标准事件
8. 调用内部 handler
9. 返回 HTTP 响应
```

### 6.3 事件支持范围

#### Pull Request
- 支持：`opened`
- 支持：`synchronized`
- 忽略：其他 action（如 `closed`、`reopened`）

#### Issues
- 支持：`labeled`
- 支持：`unlabeled`
- 忽略：其他 action（如 `edited`、`closed`）

#### 其他 event type
- 直接忽略并记录日志

### 6.4 HTTP 响应策略

| 场景 | HTTP 状态码 | 说明 |
|------|-------------|------|
| 成功处理事件 | `200 OK` | handler 成功完成 |
| 合法但不关心的事件 | `200 OK` | 避免 Gitea 重试 |
| 请求格式错误 | `400 Bad Request` | 方法、header、JSON、payload 缺字段等问题 |
| 验签失败 | `401 Unauthorized` | secret / 签名问题 |
| handler 执行失败 | `500 Internal Server Error` | 内部错误，允许上游重试 |

## 7. 错误处理策略

### 7.1 错误类别

建议定义专用错误，供 receiver 做统一映射：

```go
var (
    ErrUnsupportedEvent  = errors.New("unsupported event")
    ErrUnsupportedAction = errors.New("unsupported action")
    ErrInvalidPayload    = errors.New("invalid payload")
    ErrSignatureMismatch = errors.New("signature mismatch")
    ErrMissingSignature  = errors.New("missing signature")
    ErrSecretNotConfigured = errors.New("secret not configured")
)
```

### 7.2 映射规则

- `ErrUnsupportedEvent` / `ErrUnsupportedAction`
  - 返回 `200 OK`
  - 记录 `ignored=true`
- `ErrInvalidPayload`
  - 返回 `400 Bad Request`
- `ErrMissingSignature` / `ErrSignatureMismatch` / `ErrSecretNotConfigured`
  - 返回 `401 Unauthorized`
- 其他错误
  - 返回 `500 Internal Server Error`

### 7.3 设计原则

- **不要把“忽略事件”当成服务失败**
- **安全和格式问题必须明确拒绝**
- **内部执行失败应让上游可感知并重试**

## 8. Gitea 配置指南

M1.4 设计文档需同时定义推荐的 Gitea 配置方式，避免服务端与 Gitea 控制台配置不一致。

### 8.1 Webhook URL

```text
POST /webhooks/gitea
```

部署示例：

```text
https://dtworkflow.example.com/webhooks/gitea
```

### 8.2 Content Type

- 选择 `application/json`

### 8.3 Secret

- 必填
- 服务端与 Gitea 必须配置相同 secret
- 不允许以空 secret 进入生产环境

### 8.4 勾选事件

建议当前只启用：
- Pull Request
- Issues

当前服务端实际处理动作：
- PR：`opened`、`synchronized`
- Issue：`labeled`、`unlabeled`

### 8.5 重试预期

- `2xx`：视为投递成功
- `4xx`：表示请求或配置错误，应检查 Webhook 配置和 secret
- `5xx`：表示服务内部异常，可触发 Gitea 重试

## 9. 文件组织

```text
internal/webhook/
├── router.go            # 路由注册与依赖组装
├── receiver.go          # HTTP 接入、错误映射、调用 verifier/parser/handler
├── verifier.go          # Gitea Webhook HMAC 验签
├── parser.go            # 事件类型识别、payload 解析、标准事件构造
├── event.go             # 内部标准事件模型与引用类型
├── handler.go           # Handler 接口与默认日志实现
├── gitea_types.go       # M1.4 所需的最小 Gitea Webhook payload 类型
├── receiver_test.go     # 路由与 HTTP 行为测试
├── verifier_test.go     # 验签测试
├── parser_test.go       # 事件解析与标准化测试
└── testdata/            # Webhook payload fixture
    ├── pull_request_opened.json
    ├── pull_request_synchronized.json
    ├── issue_labeled_auto_fix.json
    ├── issue_unlabeled_auto_fix.json
    └── issue_labeled_other.json
```

设计要点：
- **接收层、解析层、安全层分文件**，便于独立测试
- **原始 payload 类型独立隔离**，避免污染内部标准事件模型
- **fixture 化测试数据**，降低测试中的 JSON 嵌入噪音

## 10. 测试策略

### 10.1 verifier 单元测试

覆盖：
- 正确签名通过
- 错误签名失败
- 缺少签名 header 失败
- 空 secret 失败
- 使用原始 body 校验而不是重编码 JSON

### 10.2 parser 单元测试

#### Pull Request
- `opened` 成功解析
- `synchronized` 成功解析
- `closed` 识别为忽略事件
- payload 缺少 PR/仓库关键字段时报错

#### Issues
- `labeled` 且标签为 `auto-fix` → `AutoFixAdded=true`
- `unlabeled` 且标签为 `auto-fix` → `AutoFixRemoved=true`
- `labeled` 但不是 `auto-fix` → 仍成功解析为通用标签事件
- 不支持 action → 识别为忽略事件
- payload 非法 / 缺字段 → 报错

### 10.3 receiver 路由测试

基于 `httptest` + Gin：
- 正常 PR 事件返回 `200`
- 正常 Issue label 事件返回 `200`
- 不支持事件返回 `200`
- 签名错误返回 `401`
- 非法 JSON 返回 `400`
- handler 返回错误时返回 `500`

同时验证：
- handler 是否被调用
- 调用次数与事件类型是否正确
- 关键字段是否被正确传递

### 10.4 与现有 serve 的集成测试

在 `internal/cmd/serve_test.go` 基础上补充：
- 启动服务后访问 `/webhooks/gitea`
- 验证路由已注册
- 保持 `/healthz` 现有行为不变

## 13. 实现顺序建议

```text
Step 1: 定义模型与接口
  event.go       → RepositoryRef / PullRequestEvent / IssueLabelEvent
  handler.go     → Handler 接口 + 默认日志实现
  gitea_types.go → 最小 Gitea payload 类型

Step 2: 实现安全与解析
  verifier.go    → HMAC 校验
  parser.go      → 事件识别 + payload 解析 + 标准事件构造
  verifier_test.go / parser_test.go

Step 3: 接入 HTTP 层
  receiver.go    → 请求校验、错误映射、handler 调用
  router.go      → Gin 路由注册
  receiver_test.go

Step 4: 集成到 serve
  internal/cmd/serve.go 中注册 webhook 路由
  补充 serve_test.go 覆盖路由注册

Step 5: 文档与收尾
  补充 Gitea 配置指南
  运行 lint / test / build-linux 验证
```
