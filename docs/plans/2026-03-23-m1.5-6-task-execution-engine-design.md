## 1. 合并可行性分析

### 1.1 为什么合并

M1.5（任务队列）和 M1.6（Docker Worker 池）在架构上是**生产者-消费者**关系的两端：

- M1.5 负责"入队"：Webhook 事件 -> 创建任务 -> 推入 Redis 队列
- M1.6 负责"消费"：从队列取任务 -> 启动 Docker 容器 -> 执行 Claude Code CLI

**单独交付的问题**：
- M1.5 单独交付只能入队但无法执行，无法端到端验证
- M1.6 单独交付只有 Worker 但无任务来源，同样无法端到端验证
- Phase 1 的验证标准明确要求"Webhook -> 创建任务 -> 启动容器 -> Claude Code 执行 -> 结果回写"全链路打通

**合并的价值**：
- 一次交付即可端到端验证：Webhook -> 入队 -> Worker 消费 -> 容器执行 -> 结果持久化
- 减少集成阶段的返工风险
- 接口在同一个里程碑内设计，天然契合

### 1.2 合并后的里程碑定义

**合并里程碑名称**：M1.5-6 任务执行引擎（Task Execution Engine）

**范围**：
- 任务队列（asynq + Redis）集成与任务生命周期管理
- 任务状态持久化（SQLite）
- Docker Worker 池管理与容器生命周期
- Claude Code CLI Docker 镜像构建
- CLI 命令实现（`task status/list/retry`）
- Webhook Handler 替换（LogHandler -> EnqueueHandler）

**明确不做**：
- PR 评审业务逻辑（M2）
- Issue 修复业务逻辑（M3）
- Viper 配置管理（M1.8）
- 通知框架（M1.7）
- Web 管理界面
- **review-pr/fix-issue --sync 模式**（推迟到 M2/M3，见设计决策 #16）

### 1.3 并行开发可行性

合并后可拆分为 **3 条并行开发轨道（Track）**，通过预先定义的接口契约解耦：

```
Track A: 任务队列层        Track B: Docker Worker 层      Track C: 持久化与 CLI
(internal/queue/)          (internal/worker/)             (internal/store/ + cmd/)

  asynq 集成                 Docker SDK 集成                SQLite schema
  任务类型定义               容器生命周期管理               任务 CRUD
  EnqueueHandler             Worker 池管理                  CLI task 命令
  任务路由与分发             Claude Code 镜像构建           状态查询 API

        |                          |                           |
        +----------+---------------+---------------------------+
                   |
            集成层（串行）
          asynq Handler 调用 Worker
          Worker 结果写入 Store
          serve.go 组装所有依赖
```

**关键解耦点**：Track A 和 Track B 通过 `model.TaskPayload` 通信，Track C 通过 `store.Store` 接口与两者解耦。三条轨道可由不同开发者/agent 并行推进，最后在集成层汇合。

### 3.3 TaskPayload 设计哲学：最小定位符

**决策**：TaskPayload 是**任务定位符**，不是完整数据快照。

**理由**：
1. **数据新鲜度**：PR 从入队到执行可能有时间差（队列积压、重试），此时 PR diff 可能已更新。如果 payload 是快照，Worker 会基于过时数据执行评审。
2. **payload 体积**：asynq 将 payload 序列化存入 Redis，完整 diff/issue body 可能很大，浪费 Redis 内存。
3. **幂等兼容**：定位符不变（repo + PR number），但实际数据可以变。重试时自动获取最新状态。

**当前 Processor 执行流程**：
```
1. 从 asynq 反序列化 TaskPayload（仅含 repo_owner, repo_name, pr_number 等定位信息）
2. 更新任务状态并将 payload 直接传入 Worker 容器执行链路
3. 根据执行结果更新最终状态，并在最终状态落库后触发后续副作用（如通知）
```

因此 TaskPayload 当前仍以“最小定位符 + 基础上下文字段”为主，为后续业务阶段补充更完整的数据获取语义预留空间。

### 3.4 并发控制归属：asynq 主控，Pool 管容器

**问题**：asynq.Config{Concurrency: N} 与 Worker Pool 若同时做并发控制会形成双层限制，语义冲突。

**方案**：

```
asynq.Config{Concurrency: N}   -- 唯一的并发控制点
         |
         v
    Processor.ProcessTask()     -- 每个并发 goroutine 调用
         |
         v
    Pool.Run(ctx, payload)      -- Pool 不做并发限制，只负责：
                                     1. 创建容器
                                     2. 启动容器
                                     3. 等待执行完成
                                     4. 收集结果
                                     5. 销毁容器
```

**关键设计**：
- Pool 从"信号量 + Acquire/Release"模式改为 `Run(ctx, payload) (*ExecutionResult, error)` 单方法模式
- Pool 内部不维护 buffered channel 做并发控制
- `Pool.Stats()` 通过 atomic counter 统计活跃/已完成容器数
- asynq `Concurrency` 配置值与 `--max-workers` flag 保持一致（默认 3）

### 3.5 SQLite-Redis 双写一致性

**问题**：EnqueueHandler 同步执行 "创建 TaskRecord + 入队 asynq"，如果 Redis 入队失败但 SQLite 已写入，任务变孤儿。

**写入流程设计**：

```go
func (h *EnqueueHandler) enqueueTask(ctx context.Context, payload model.TaskPayload, priority model.TaskPriority) error {
    // Step 1: 幂等检查
    existing, err := h.store.FindByDeliveryID(ctx, payload.DeliveryID, payload.TaskType)
    if err != nil { return err }
    if existing != nil { return nil } // 已存在，跳过

    // Step 2: 创建任务记录（status = pending）
    record := &model.TaskRecord{
        ID:        uuid.NewString(),
        TaskType:  payload.TaskType,
        Status:    model.TaskStatusPending,
        Priority:  priority,
        Payload:   payload,
        // ...
    }
    if err := h.store.CreateTask(ctx, record); err != nil {
        return fmt.Errorf("创建任务记录失败: %w", err)
    }

    // Step 3: 入队 Redis
    asynqID, err := h.client.Enqueue(ctx, payload, EnqueueOptions{
        Priority: priority,
        TaskID:   record.ID,
    })
    if err != nil {
        // Redis 入队失败，任务保持 pending 状态，由 recovery goroutine 后续处理
        slog.Error("Redis 入队失败，任务保留为 pending 等待恢复",
            "task_id", record.ID, "error", err)
        return fmt.Errorf("入队失败: %w", err)
    }

    // Step 4: 更新状态为 queued
    record.AsynqID = asynqID
    record.Status = model.TaskStatusQueued
    if err := h.store.UpdateTask(ctx, record); err != nil {
        // SQLite 更新失败但 Redis 已入队——任务会被执行，Processor 会更新状态
        slog.Warn("更新任务状态为 queued 失败，Processor 执行时将修正",
            "task_id", record.ID, "error", err)
    }

    return nil
}
```

**Recovery Goroutine 设计**：

```go
// RecoveryLoop 定期扫描 pending 状态的孤儿任务并重新入队
// 在 serve 启动时作为后台 goroutine 运行
type RecoveryLoop struct {
    store    store.Store
    client   *Client
    interval time.Duration // 默认 60s
    maxAge   time.Duration // pending 超过此时长视为孤儿，默认 120s
}

func (r *RecoveryLoop) Run(ctx context.Context) {
    ticker := time.NewTicker(r.interval)
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            r.recoverOrphanTasks(ctx)
        }
    }
}

func (r *RecoveryLoop) recoverOrphanTasks(ctx context.Context) {
    orphans, err := r.store.ListOrphanTasks(ctx, r.maxAge)
    if err != nil {
        slog.Error("查询孤儿任务失败", "error", err)
        return
    }
    for _, task := range orphans {
        asynqID, err := r.client.Enqueue(ctx, task.Payload, EnqueueOptions{
            Priority: task.Priority,
            TaskID:   task.ID,
        })
        if err != nil {
            // 注意：如果 asynq 返回 UniqueViolation，说明任务已在 Redis 中，
            // 应视为成功并更新状态为 queued
            slog.Error("恢复孤儿任务入队失败", "task_id", task.ID, "error", err)
            continue
        }
        task.AsynqID = asynqID
        task.Status = model.TaskStatusQueued
        _ = r.store.UpdateTask(ctx, task)
        slog.Info("恢复孤儿任务成功", "task_id", task.ID)
    }
}
```

### 3.6 包依赖方向

```
                internal/model
              (TaskPayload, TaskRecord, TaskType, TaskStatus 等共享数据类型)
               ^         ^         ^
              /          |          \
             /           |           \
webhook     queue       store       worker
(Handler)   (Enqueue,   (Store,     (Pool,
             Processor,  SQLite)     Container)
             Recovery)

queue --> webhook  (实现 Handler 接口，导入事件类型)
queue --> store    (调用 Store 接口)
queue --> worker   (调用 Pool.Run)
queue --> model    (使用共享数据类型)
store --> model    (使用共享数据类型)
worker --> model   (使用 TaskPayload)
```

**确认**：
- `queue -> webhook` 单向，webhook 不导入 queue
- `queue -> store` 单向，store 不导入 queue
- `queue -> worker` 单向，worker 不导入 queue
- 所有包共同导入 `model`，但 `model` 不导入任何内部包
- **无循环依赖**（决策 #26 引入 `internal/model/` 解决了 store <-> queue 潜在循环）

### 3.7 serve 命令依赖注入设计

**问题**：当前 `serve.go` 使用包级全局变量（`serveHost`, `servePort`, `serveWebhookSecret`），无法注入 EnqueueHandler/Store/Pool 等依赖。

**解决方案**：引入 `ServiceDeps` struct 封装所有依赖，`runServe` 内部构建并组装。

**runServe 核心流程**：

```go
func runServeWithConfig(cfg serveConfig, stopCh <-chan struct{}) error {
    deps, cleanup, err := BuildServiceDeps(cfg)
    if err != nil {
        return err
    }
    defer cleanup()

    gin.SetMode(gin.ReleaseMode)
```

### 3.8 优雅关闭编排

**分层关闭设计**：

```go
func gracefulShutdown(server *http.Server, deps *ServiceDeps, cancelRecovery context.CancelFunc) error {
    var firstErr error
    recordErr := func(layer string, err error) {
        slog.Error("关闭失败", "layer", layer, "error", err)
        if firstErr == nil { firstErr = err }
    }

    // Layer 1: HTTP Server（5s）
    slog.Info("关闭 Layer 1: HTTP Server...")
    httpCtx, httpCancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer httpCancel()
    if err := server.Shutdown(httpCtx); err != nil { recordErr("HTTP", err) }

    // Layer 2: asynq Processor（等待活跃 handler 完成）
    slog.Info("关闭 Layer 2: asynq Processor...")
    deps.AsynqServer.Shutdown()

    // Layer 3: Recovery goroutine
    slog.Info("关闭 Layer 3: Recovery goroutine...")
    cancelRecovery()

    // Layer 4: Worker Pool（30s，清理残留容器）
    slog.Info("关闭 Layer 4: Worker Pool...")
    poolCtx, poolCancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer poolCancel()
    if err := deps.Pool.Shutdown(poolCtx); err != nil { recordErr("WorkerPool", err) }

    // Layer 5: Store（关闭 SQLite 连接）
    slog.Info("关闭 Layer 5: Store...")
    if err := deps.Store.Close(); err != nil { recordErr("Store", err) }

    // Layer 6: asynq Client（关闭 Redis 连接）
    slog.Info("关闭 Layer 6: asynq Client...")
    if err := deps.QueueClient.Close(); err != nil { recordErr("QueueClient", err) }

    if firstErr != nil {
        return fmt.Errorf("关闭过程中出现错误: %w", firstErr)
    }
    slog.Info("所有组件已优雅关闭")
    return nil
}
```

**各层超时值**：

| 层级 | 组件 | 超时值 | 理由 |
|------|------|--------|------|
| L1 | HTTP Server | 5s | Webhook 请求处理很快（< 1s） |
| L2 | asynq Processor | 由 asynq ShutdownTimeout 控制 | 等待活跃 handler 完成；最长任务 30min |
| L3 | Recovery goroutine | 立即取消 | 无状态，随时可停 |
| L4 | Worker Pool | 30s | 清理容器资源（stop + remove） |
| L5 | Store (SQLite) | 即时 | Close() 同步 |
| L6 | asynq Client (Redis) | 即时 | Close() 同步 |

**实现注意**：`defer cleanup()` 会在 `gracefulShutdown` 之后再次执行，依赖各 Close() 方法的幂等性。实现时可通过标志位避免双重关闭。

## 12. RALPLAN-DR 摘要

### Principles

1. **接口驱动并行**：先定义接口契约，再并行实现，最后集成验证
2. **端到端可验证**：合并里程碑确保一次交付即可全链路验证
3. **单一并发控制点**：asynq 控制并发，Worker Pool 只管容器生命周期
4. **防御性双写**：SQLite-Redis 三步写入 + recovery goroutine 防止孤儿任务
5. **最小外部依赖**：不提前引入 Viper，配置通过 flags/env 管理

### Decision Drivers（Top 3）

1. **端到端验证需求**：Phase 1 验证标准要求全链路打通
2. **一致性保证**：SQLite + Redis 双存储必须有明确的写入顺序和故障恢复
3. **关闭安全性**：Worker 执行可达 30 分钟，分层关闭是生产可用前提

### ADR

**Decision**：合并 M1.5 + M1.6 为 "M1.5-6 任务执行引擎"，3 条并行 Track，引入 model 包解耦、ServiceDeps 依赖注入、三步双写 + recovery、分层关闭编排。

**Alternatives considered**：方案 B（独立里程碑）因延迟端到端验证和适配返工被排除。

**Consequences**：阶段一接口契约约 0.5 天串行；交付周期约 4-6 天；需 Docker daemon。

**Follow-ups**：M1.7 通知 / M1.8 配置 / M2 PR 评审 / M3 Issue 修复均可在现有基础设施上增量实现。

### 实现注意事项
