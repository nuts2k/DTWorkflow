## 1. 目标

在 M1.1 骨架基础上，搭建完整的 CLI 命令结构，注册所有业务命令（空壳）和管理命令，实现 `serve` 子命令启动 HTTP 服务，建立统一的输出层和错误处理模式。交付一个可 `--help` 浏览全部功能、可 `serve` 运行的 CLI 框架。

---

## 2. 设计决策记录

| 决策项 | 结论 | 理由 |
|--------|------|------|
| 命令命名风格 | 混合式：业务命令扁平，管理命令层级 | 业务命令（review-pr、fix-issue）是单一动作，扁平更直接；task 有多个子操作（status/list/retry），层级更合理 |
| CLI 执行模型 | 默认走队列，`--sync` 同步直接执行 | 自动化场景走队列（Webhook 触发），手动调试走 `--sync` 方便排查 |
| 输出格式化 | 统一 Render 层（`PrintResult` 函数） | 一个小函数解决两种输出模式，不过度抽象，避免每个命令重复 `if jsonOutput` |
| serve 范围 | 最小骨架：Gin + `/healthz` + 优雅关闭 | 验证 Gin 集成链路，不预注册空路由（YAGNI） |
| 参数校验 | `PreRunE` 手动校验，中文提示 | Cobra `MarkFlagRequired` 输出不友好，手动校验可控 |
| JSON 错误输出 | stdout 输出结构化 JSON，stderr 不输出 | 调用方只需解析 stdout，退出码判断成功/失败 |
| M1.2 新增依赖 | 仅 Gin | Viper（M1.8）、asynq（M1.5）等严格按需引入 |

## 4. 统一输出层（Render）

新增 `internal/cmd/output.go`，提供统一的输出工具函数：

### PrintResult

```go
// PrintResult 根据 --json 标志选择输出模式
// data: 可 JSON 序列化的结构体
// humanFn: 人类可读格式化函数
func PrintResult(data any, humanFn func(any) string) {
    if jsonOutput {
        enc := json.NewEncoder(os.Stdout)
        enc.SetIndent("", "  ")
        enc.Encode(data)
        return
    }
    fmt.Print(humanFn(data))
}
```

### PrintError

```go
// PrintError 统一错误输出
// JSON 模式：stdout 输出 {"error": "...", "exitCode": N}
// 人类模式：stderr 输出错误信息
func PrintError(err error) {
    code := ExitCode(err)
    if jsonOutput {
        enc := json.NewEncoder(os.Stdout)
        enc.SetIndent("", "  ")
        enc.Encode(map[string]any{
            "error":    err.Error(),
            "exitCode": code,
        })
        return
    }
    fmt.Fprintln(os.Stderr, err)
}
```

### 设计要点

- 命令代码不再写 `if jsonOutput`，统一通过 `PrintResult` 处理
- JSON 模式依赖 struct tag 控制字段名
- 人类格式化通过闭包传入，每个命令自定义展示
- 错误输出：JSON 模式走 stdout（结构化），人类模式走 stderr
- `main.go` 中的错误处理改为调用 `PrintError`
- 现有 `version` 命令重构为使用 `PrintResult`

### 命令使用示例

```go
// review-pr 命令（未来 M2 实现时）
result := &ReviewResult{Issues: 3, Critical: 1}
PrintResult(result, func(v any) string {
    r := v.(*ReviewResult)
    return fmt.Sprintf("评审完成：%d 个问题（%d 个严重）\n", r.Issues, r.Critical)
})
```

## 5. serve 命令设计

### 命令定义

```go
// internal/cmd/serve.go

var serveCmd = &cobra.Command{
    Use:   "serve",
    Short: "启动 HTTP 服务（API + Webhook 接收器）",
    RunE:  runServe,
}
```

### M1.2 路由

仅注册一个端点：

```
GET  /healthz    → {"status": "ok", "version": "..."}
```

不注册 `/api/v1/`、`/webhooks/` 等路由。后续里程碑通过调用各模块的路由注册函数扩展，例如 M1.4 加 `webhook.RegisterRoutes(router)`。

### 优雅关闭流程

```
1. 创建 http.Server（非 gin.Run，需要控制 server 生命周期）
2. goroutine 中启动 server.ListenAndServe()
3. 主 goroutine 监听 SIGINT / SIGTERM（signal.Notify）
4. 收到信号 → ctx, cancel := context.WithTimeout(5s)
5. server.Shutdown(ctx) → 等待在途请求完成
6. slog.Info("服务已停止")
7. return nil（退出码 0）
```

### 日志

使用 `slog`，启动时输出：

```
INFO 服务启动 host=0.0.0.0 port=8080
```

M1.2 不做请求日志中间件（仅 healthz 无需日志）。

### 新增依赖

`github.com/gin-gonic/gin`

## 6. 命令文件组织

```
internal/cmd/
├── root.go            # root 命令 + 全局 flags + Execute()
├── output.go          # 统一输出层（PrintResult / PrintError）
├── version.go         # version 子命令（重构为用 PrintResult）
├── serve.go           # serve 子命令
├── review_pr.go       # review-pr 子命令（空壳）
├── fix_issue.go       # fix-issue 子命令（空壳）
├── gen_tests.go       # gen-tests 子命令（空壳）
└── task.go            # task 父命令 + status/list/retry 子命令
```

### 设计要点

- **一个文件一个顶层命令**，文件名与命令名对应（连字符转下划线：`review-pr` → `review_pr.go`）
- **task 子命令同文件**：`task.go` 定义 `taskCmd`、`taskStatusCmd`、`taskListCmd`、`taskRetryCmd`。它们共享上下文且体量小（空壳），无需拆分
- **命令注册**：每个文件的 `init()` 中调用 `rootCmd.AddCommand()`，与现有 `version.go` 一致。task 子命令在 `init()` 中注册到 `taskCmd`

## 7. 错误处理与退出码

### 退出码语义

| 退出码 | 含义 | 触发场景 |
|--------|------|----------|
| 0 | 成功 | 命令正常完成 |
| 1 | 失败 | 参数错误、执行失败、未实现命令 |
| 2 | 部分成功 | 批量操作部分失败（如大型 PR 分批评审部分批次失败） |

### 必填参数校验

使用 `PreRunE` 手动校验，不用 Cobra 的 `MarkFlagRequired`：

```go
PreRunE: func(cmd *cobra.Command, args []string) error {
    if repo == "" {
        return fmt.Errorf("必须指定仓库：--repo <仓库名>")
    }
    if prNumber <= 0 {
        return fmt.Errorf("必须指定 PR 编号：--pr <编号>")
    }
    return nil
},
```

### 错误链路

```
命令 RunE 返回 error
  → main.go 收到 error
  → cmd.PrintError(err)（根据 --json 模式选择输出方式）
  → os.Exit(cmd.ExitCode(err))
```

### --json 模式下的错误输出

```json
{"error": "必须指定仓库：--repo <仓库名>", "exitCode": 1}
```

JSON 模式下错误输出到 stdout（结构化），stderr 不输出。调用方只需解析 stdout + 检查退出码。

## 8. --sync 执行模式预留

M1.2 注册 `--sync` flag，不实现队列逻辑（队列在 M1.5）。

### 命令内部结构预留

```go
// review_pr.go（展示最终结构，M1.2 为空壳）
RunE: func(cmd *cobra.Command, args []string) error {
    req := &ReviewRequest{Repo: repo, PR: prNumber}

    if syncMode {
        // 直接执行：调用核心逻辑，阻塞等结果
        return executeReviewPR(cmd.Context(), req)
    }
    // 队列模式：提交任务，返回 task ID
    return enqueueReviewPR(cmd.Context(), req)
},
```

### M1.2 实际行为

- `--sync` 和默认模式都返回"未实现"提示
- flag 已注册，`--help` 中可见

### Request 结构体

M1.2 在各命令文件中定义请求结构体（如 `ReviewRequest`），作为文件内的私有类型。等 M1.5 引入队列时，迁移到 `internal/` 对应业务包（如 `internal/review/`）。

### 关键原则

M1.2 不引入任何队列依赖（无 asynq、无 Redis），纯粹是 CLI 框架层面的工作。
