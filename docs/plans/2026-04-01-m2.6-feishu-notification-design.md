# M2.6 飞书通知渠道接入设计

> 为 DTWorkflow 接入飞书通知渠道，支持 PR 评审等事件的实时状态推送。

## 1. 需求概述

### 1.1 背景

当前 DTWorkflow 的通知框架仅支持 Gitea 评论回写（`GiteaNotifier`）。团队日常沟通使用飞书，需要将关键事件（PR 评审开始、完成、失败等）推送到飞书群组，实现即时感知。

### 1.2 目标

- 接入飞书自定义机器人（Webhook），向群组发送通知
- **通知时机至少两次**：事件接收/开始处理时一次，完成后状态同步一次
- 消息内容包含评审摘要（结论 + 关键 issue 统计）+ 跳转链接
- 接口设计预留应用机器人（App Bot）扩展空间

### 1.3 不做

- 本期不实现应用机器人（App Bot）
- 不实现个人消息推送（需 App Bot）
- 不实现卡片交互回调（需 App Bot）
- 不实现飞书 @用户（需 App Bot 查询 open_id）

## 2. 方案选型

采用 **方案 C：Webhook 先行，预留应用机器人扩展**。

| 维度 | Webhook 机器人 | 应用机器人 |
|------|---------------|-----------|
| 接入成本 | 5 分钟，群设置添加 | 需创建应用、管理员审批 |
| 发群消息 | ✅ | ✅ |
| 发个人消息 | ❌ | ✅ |
| 交互卡片回调 | 仅跳转链接 | 支持按钮回调 |
| Rate Limit | 100次/分钟 | 1000次/分钟 |

选择理由：
1. Webhook 零门槛，不需要等管理员审批
2. 10 人团队 + 10 仓库的体量，群通知够用
3. 接口层按 `Notifier` 接口设计，后续替换为 App Bot 只是内部实现变化

### 3.2 通知时机

当前只在任务完成后发一次通知（`sendCompletionNotification`）。需新增任务开始时的通知：

| 时机 | EventType | 触发点 | 消息内容 |
|------|-----------|--------|---------|
| 任务开始处理 | `pr.review.started` | `Processor.handleReviewPR` 开头 | "正在评审 PR #N，请稍候..." |
| 评审成功完成 | `pr.review.done` | 现有 `sendCompletionNotification` | 评审摘要 + 结论 + issue 统计 + 链接 |
| 评审失败 | `system.error` | 现有 `sendCompletionNotification` | 失败原因 + PR 链接 |

后续扩展 Issue 修复等任务类型时，同样遵循"开始 + 完成"双通知模式。

## 4. 详细设计

### 4.1 FeishuNotifier

```go
// feishu_notifier.go
package notify

type FeishuNotifier struct {
    webhookURL string
    secret     string     // 签名密钥（可选，推荐开启）
    httpClient *http.Client
    logger     *slog.Logger
}

func NewFeishuNotifier(webhookURL string, opts ...FeishuOption) (*FeishuNotifier, error)

func (n *FeishuNotifier) Name() string { return "feishu" }
func (n *FeishuNotifier) Send(ctx context.Context, msg Message) error
```

**签名校验**（HMAC-SHA256，飞书自定义机器人标准）：

飞书签名算法：以 `timestamp + "\n" + secret` 作为 HMAC-SHA256 的 **key**，对空消息体计算摘要，再 Base64 编码。
参考：[飞书自定义机器人安全设置](https://open.feishu.cn/document/client-docs/bot-v3/add-custom-bot#f62e72d5)

```go
func genSign(secret string, timestamp int64) (string, error) {
    stringToSign := fmt.Sprintf("%d\n%s", timestamp, secret)
    h := hmac.New(sha256.New, []byte(stringToSign))
    h.Write([]byte{}) // data 为空
    return base64.StdEncoding.EncodeToString(h.Sum(nil)), nil
}
```

签名和 timestamp 放在请求体顶层：
```json
{
  "timestamp": "1617000000",
  "sign": "xxx",
  "msg_type": "interactive",
  "card": { ... }
}
```

**HTTP 调用**：
- 直接使用 `net/http` POST 到 Webhook URL，不引入飞书 SDK（Webhook 模式无需 SDK）
- 超时设置 10 秒
- 非 2xx 响应返回 `ErrSendFailed`

格式化逻辑：

| Key | 说明 | 示例 |

| `pr_url` | PR 页面链接 | `https://gitea.example.com/org/repo/pulls/42` |

### 4.7 Processor 层改动

#### 数据来源说明

当前 `TaskPayload` 没有 `PRTitle` 字段，`TaskRecord` 没有 `ReviewVerdict` / `IssueSummary` 字段。
Metadata 中这些信息的获取方式：

| Metadata Key | 数据来源 | 说明 |
|-------------|---------|------|
| `pr_title` | Gitea API（review.Service.Execute 中已获取 PR 对象） | 需在 Processor 调用 review 后从 PR 标题透传；或新增 `TaskPayload.PRTitle` 字段，在 Webhook handler 构造 payload 时填入 |
| `verdict` | `review.ReviewResult.Review.Verdict` | Execute 返回值已包含，需在 Processor 层记录到通知 Metadata |
| `issue_summary` | `review.ReviewResult.Review.Issues` | 在 Processor 层从 ReviewResult 统计 issue 数量并格式化 |

**推荐方案**：新增 `TaskPayload.PRTitle` 字段（Webhook handler 中从事件体提取 PR 标题填入），
verdict 和 issue_summary 在 Processor 执行完 review 后从 `ReviewResult` 提取并传入通知 Metadata。

#### 新增"开始"通知

`sendStartNotification` 的调用位置：在 `ProcessTask` 中，**review-enabled 检查通过后、实际执行前**调用。
即在当前代码 `processor.go` 的 review-enabled 检查（约 L179）之后、task 执行（约 L183）之前插入。
这避免了对已禁用评审的仓库发送无意义的"开始"通知。

```go
// processor.go

func (p *Processor) sendStartNotification(ctx context.Context, payload model.TaskPayload) {
    if p.notifier == nil {
        return
    }
    msg, ok := p.buildStartMessage(payload)
    if !ok {
        return
    }
    if err := p.notifier.Send(ctx, msg); err != nil {
        p.logger.ErrorContext(ctx, "发送任务开始通知失败",
            "task_type", payload.TaskType,
            "error", err,
        )
        // 开始通知失败不阻断任务执行
    }
}

func (p *Processor) buildStartMessage(payload model.TaskPayload) (notify.Message, bool) {
    if payload.RepoOwner == "" || payload.RepoName == "" {
        return notify.Message{}, false
    }

    switch payload.TaskType {
    case model.TaskTypeReviewPR:
        if payload.PRNumber <= 0 {
            return notify.Message{}, false
        }
        metadata := map[string]string{
            "pr_url": buildPRURL(p.giteaBaseURL, payload),
        }
        if payload.PRTitle != "" {
            metadata["pr_title"] = payload.PRTitle
        }
        return notify.Message{
            EventType: notify.EventPRReviewStarted,
            Severity:  notify.SeverityInfo,
            Target: notify.Target{
                Owner:  payload.RepoOwner,
                Repo:   payload.RepoName,
                Number: payload.PRNumber,
                IsPR:   true,
            },
            Title:    "PR 自动评审开始",
            Body:     fmt.Sprintf("正在评审 PR #%d\n\n仓库：%s", payload.PRNumber, payload.RepoFullName),
            Metadata: metadata,
        }, true
    default:
        return notify.Message{}, false
    }
}
```

#### 完成通知补充 Metadata

在 `buildNotificationMessage`（完成通知）中，verdict 和 issue_summary 从 `ReviewResult` 解析：

```go
// Processor 执行 review 后，将结果摘要注入通知 Metadata
func buildReviewMetadata(giteaBaseURL string, payload model.TaskPayload, reviewResult *review.ReviewResult) map[string]string {
    metadata := map[string]string{
        "pr_url": buildPRURL(giteaBaseURL, payload),
    }
    if payload.PRTitle != "" {
        metadata["pr_title"] = payload.PRTitle
    }
    if reviewResult != nil && reviewResult.Review != nil {
        metadata["verdict"] = string(reviewResult.Review.Verdict)
        metadata["issue_summary"] = formatIssueSummary(reviewResult.Review.Issues)
    }
    return metadata
}
```

### 4.8 PR URL 构造

需要 Gitea 的 base URL 来构造 PR 跳转链接：

```go
// buildPRURL 基于 Gitea 配置构造 PR 页面链接
func buildPRURL(giteaBaseURL string, payload model.TaskPayload) string {
    return fmt.Sprintf("%s/%s/%s/pulls/%d",
        strings.TrimRight(giteaBaseURL, "/"),
        payload.RepoOwner,
        payload.RepoName,
        payload.PRNumber,
    )
}
```

`giteaBaseURL` 从 `config.GiteaConfig.URL` 获取，在 `Processor` 初始化时注入。建议沿用现有的 Option 注入模式：

```go
type Processor struct {
    // ...
    giteaBaseURL string
}

func WithGiteaBaseURL(url string) ProcessorOption {
    return func(p *Processor) {
        p.giteaBaseURL = strings.TrimRight(url, "/")
    }
}
```

装配层调用示例：

```go
processor := queue.NewProcessor(
    deps.Pool,
    deps.Store,
    deps.Notifier,
    slog.Default(),
    queue.WithGiteaBaseURL(cfg.AppCfg.Gitea.URL),
    // ... other opts
)
```

## 5. 错误处理

| 场景 | 处理方式 |
|------|---------|
| 飞书 Webhook 返回非 2xx | 返回 `ErrSendFailed`，Router 记录日志，不影响其他渠道 |
| 飞书 Webhook 超时 | HTTP Client 10 秒超时，返回错误 |
| Webhook URL 无效 | 启动时配置校验拦截 |
| 签名计算失败 | 返回 `ErrSendFailed` |
| 开始通知失败 | 仅日志，不阻断任务执行 |
| 飞书 Rate Limit (429) | 当前不做重试（100次/分钟对 10 仓库绰绰有余），但显式检测 429 状态码并用 `slog.Warn` 记录，便于运维提前感知接近限流 |

关键原则：**飞书通知失败绝不影响 Gitea 回写和任务执行**。Router 的多渠道错误聚合机制已保证这一点。
