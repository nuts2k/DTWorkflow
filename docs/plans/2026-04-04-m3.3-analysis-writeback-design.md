# M3.3 分析结果解析与 Issue 评论回写 — 设计文档

> Phase 3 第一轮最后一个里程碑。将 Claude 输出的结构化分析结果格式化为 Markdown，回写为 Gitea Issue 评论。

## 1. 目标

- 将 M3.2 容器执行产出的 `AnalysisOutput` 格式化为人类可读的 Issue 评论
- 覆盖三种场景：正常分析、信息不足追问、降级（解析失败）
- 回写失败不影响任务最终状态（与 Phase 2 review 一致）
- 安全防护：Markdown 注入防护、评论长度限制

## 2. 关键设计决策

| 决策 | 结论 | 理由 |
|------|------|------|
| 分析结果持久化 | **暂不做** | ROADMAP 标注"可选"；Issue 评论本身即持久化；TaskRecord.result 已保存原始输出；无消费者 |
| 回写代码组织 | **轻量方案** | 新增 `formatter.go` + `Execute()` 内直接调用 `CreateIssueComment`；不新建 `Writer` 结构体；逻辑仅"格式化→发评论"两步，不值得额外抽象 |
| 与 review.Writer 的关系 | **不复用** | Issue 评论无行级 comment、无 diff 映射、无 staleness check、无持久化；复用反而引入不必要的耦合 |

## 4. 代码改动清单

### 4.1 `internal/fix/formatter.go`（新建）

格式化函数，将 `FixResult` 转换为 Markdown 评论正文。

```go
// FormatAnalysisComment 根据分析结果生成 Issue 评论 Markdown。
// 三场景分支：正常 / 信息不足 / 降级。
func FormatAnalysisComment(result *FixResult) string

// formatNormalReport 信息充分时的完整分析报告
func formatNormalReport(analysis *AnalysisOutput, durationSec, costUSD float64) string

// formatInsufficientInfo 信息不足时的追问评论
func formatInsufficientInfo(analysis *AnalysisOutput, durationSec, costUSD float64) string

// formatFallback 解析失败时的降级评论
func formatFallback(rawOutput string, durationSec, costUSD float64) string
```

工具函数复用 review 包的模式（独立实现，避免跨包依赖）：
- `escapeMarkdown(s string) string` — 转义 Markdown 特殊字符
- `truncateString(s string, maxBytes int) string` — UTF-8 安全截断
- `bodyMaxLen = 60000` — Gitea 评论长度上限

### 4.2 `internal/fix/service.go`（修改）

1. **扩展 `IssueClient` 接口**：

```go
type IssueClient interface {
    GetIssue(ctx context.Context, owner, repo string, index int64) (*gitea.Issue, *gitea.Response, error)
    ListIssueComments(ctx context.Context, owner, repo string, index int64, opts gitea.ListOptions) ([]*gitea.Comment, *gitea.Response, error)
    CreateIssueComment(ctx context.Context, owner, repo string, index int64, opts gitea.CreateIssueCommentOption) (*gitea.Comment, *gitea.Response, error)
}
```

2. **`Execute()` 末尾插入回写**：

```go
// 6. 回写 Issue 评论
comment := FormatAnalysisComment(result)
if _, _, err := s.gitea.CreateIssueComment(ctx, owner, repo, issueNum,
    gitea.CreateIssueCommentOption{Body: comment}); err != nil {
    s.logger.ErrorContext(ctx, "回写分析评论失败",
        "issue", issueNum, "error", err)
    result.WritebackError = fmt.Errorf("回写分析评论失败: %w", err)
}
```

### 4.3 `internal/fix/result.go`（修改）

添加 `WritebackError` 字段：

```go
type FixResult struct {
    IssueContext   *IssueContext
    RawOutput      string
    CLIMeta        *model.CLIMeta
    Analysis       *AnalysisOutput
    ParseError     error
    WritebackError error  // 新增：回写失败时非 nil
}
```

### 4.4 `internal/queue/processor.go`（修改）

调整 `adaptFixResult`：当回写已完成（无论正常还是降级），`ParseError` 不再导致 ExitCode=1。

逻辑变更：
- 容器执行本身失败（`Execute` 返回 error）→ 任务 failed
- `CLIMeta.IsError == true` → 任务 failed（Claude CLI 本身报错，虽降级评论已发但属于执行异常；与 review 包一致）
- `ParseError != nil` 且 `CLIMeta.IsError == false` → 任务 succeeded（仅解析失败，降级评论已发）
- `WritebackError != nil` → 任务 succeeded（回写失败不影响任务状态，仅记录日志）

具体改动：
1. 移除 `adaptFixResult` 中 `ParseError` 导致 ExitCode=1 的逻辑，改为仅将错误信息保留到 `Error` 字段供调试；`CLIMeta.IsError` 仍保留 ExitCode=1。与 review 包的 `adaptReviewResult` 行为对齐。
2. 删除 `processor.go` 中 `_ = fixResult // M3.3 回写时使用` 占位行（回写已在 `fix.Service.Execute()` 内部完成，Processor 不再需要直接使用 fixResult）。

### 4.5 无需改动的文件

- `cmd/serve.go`：`gitea.Client` 已实现 `CreateIssueComment`，接口扩展后自动满足
- `internal/gitea/issues.go`：`CreateIssueComment` 已实现
- `internal/notify/`：通知逻辑无需改动（Processor 已有 fix_issue 完成通知）

## 5. 错误处理与降级链

```
Execute() 返回
  ├── error != nil → 任务 failed（容器执行失败，不发评论；与 review 包行为一致）
  └── error == nil → 进入回写
        ├── Analysis != nil && InfoSufficient == true → 正常报告
        ├── Analysis != nil && InfoSufficient == false → 信息不足追问
        ├── Analysis == nil || ParseError != nil → 降级（原始输出）
        └── CreateIssueComment 失败 → WritebackError，任务仍 succeeded
```

关键原则：**用户至少能看到一条评论**。即使解析失败，降级评论（原始输出）也要发出去。

## 6. 安全防护

| 防护措施 | 说明 |
|---------|------|
| Markdown 转义 | `escapeMarkdown()` 转义 `[]()!<>` 等字符，防止注入钓鱼链接 |
| 代码块包裹 | 降级场景的原始输出包裹在 ``` 代码块内，阻止 Markdown 渲染 |
| 长度截断 | 评论正文不超过 60000 字节（Gitea 限制），UTF-8 安全截断 |
| 内部错误不泄露 | `ParseError` 详情仅写服务端日志，不暴露到 Issue 评论 |
