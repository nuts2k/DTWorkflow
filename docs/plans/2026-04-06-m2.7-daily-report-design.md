# M2.7 每日评审统计报告 — 设计文档

> 日期：2026-04-06
> 状态：已确认，待实施
> 依赖：M2.3（review_results 持久化）、M2.6（飞书通知渠道）

## 1. 概述

每日定时通过飞书通知发送前一天的 PR 评审统计摘要。一条聚合消息涵盖所有仓库，内部按仓库分组展示。

## 2. 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 定时调度 | asynq Scheduler | 已是项目依赖，天然支持 cron + 分布式去重 |
| 趋势数据 | 实时查询两天对比 | 数据量小（<10 仓库），避免额外存储 |
| 空数据处理 | 默认发送简短通知 | 充当系统心跳，团队可确认服务正常 |
| 发送粒度 | 一条聚合消息 | <10 仓库，一张卡片足够；避免消息轰炸 |
| 扩展架构 | 最小接口（StatCollector） | 成本极低，为 fix_issue/gen_tests 统计预留 |
| 架构模式 | 独立 Handler（Scheduler 入队 + 独立 Handler 处理） | 日报无 payload、无 delivery_id，与现有 Processor 的 TaskPayload 生命周期不兼容；独立 Handler 更简洁 |

## 3. 包结构

```
internal/report/
  collector.go       # StatCollector 接口 + ReviewStatCollector 实现
  generator.go       # ReportGenerator 编排层（收集 -> 格式化 -> 发送）
  handler.go         # DailyReportHandler（实现 asynq.Handler，独立于 Processor）
  feishu_card.go     # 统计专用飞书卡片格式化
  model.go           # 统计数据模型（DailyStats / RepoStats）
```

## 4. 端到端数据流

```
serve.go 启动
  |
  |  asynq.NewScheduler(redisOpt, Location=配置时区)
  |  scheduler.Register(cron, "dtworkflow:gen_daily_report")
  |  scheduler.Run()
  |
  v  cron 触发
asynq.Server (ServeMux 路由到 DailyReportHandler)
  |
  |  DailyReportHandler.ProcessTask(ctx, task):
  |    1. 创建 TaskRecord（类型 gen_daily_report，状态 running）
  |    2. reportGenerator.Generate(ctx)
  |    3. 更新 TaskRecord（succeeded / failed）
  |
  v
ReportGenerator.Generate()
  |
  |  1. 计算时间窗口（昨天 00:00 ~ 今天 00:00，按配置时区）
  |  2. collector.Collect(ctx, yesterday)    -> DailyStats
  |  3. collector.Collect(ctx, dayBefore)    -> DailyStats（趋势对比）
  |  4. FormatDailyReportCard(stats, prevStats)
  |  5. feishuNotifier.Send(ctx, cardJSON)
  v
飞书群收到统计卡片
```

关键点：
- **不经过 Docker 容器**：`gen_daily_report` 由独立的 `DailyReportHandler` 直接执行，不走 `Processor.ProcessTask`
- **不经过现有 Processor**：现有 `Processor.ProcessTask` 假设 payload 包含 `TaskPayload`（带 delivery_id、repo 信息），且通过 `findRecord` 查找已有 TaskRecord。日报任务无 payload、无预创建的 TaskRecord，生命周期完全不同，因此使用独立 Handler
- **独立 FeishuNotifier 实例**：日报使用 `DailyReportConfig.FeishuWebhook` 构造专属的 `notify.FeishuNotifier`，不经过 `configDrivenNotifier` / `notify.Router`
- **不经过 notify.Router**：日报有独立的 Webhook 配置，直接构造 FeishuNotifier 发送
- **时区处理**：时间窗口按配置的 timezone 计算本地日期边界，再转 UTC 查询数据库。实现必须按本地日历日递推边界（如 `todayStart.AddDate(0, 0, -1)`），不能用 `Add(-24*time.Hour)` 反推昨天，否则 DST 切换日会错位 1 小时。`Asia/Shanghai` 无 DST，但仍统一使用日历边界算法，避免跨时区部署踩坑。

## 5. 数据层

### 5.1 Store 新增方法

```go
// store.go 接口新增
ListReviewResultsByTimeRange(ctx context.Context, start, end time.Time) ([]*model.ReviewRecord, error)
```

- 查询 `julianday(created_at) >= julianday(start) AND julianday(created_at) < julianday(end)`，返回所有仓库的记录
- 按 `created_at DESC` 排序
- 硬上限 2000 条（防止异常数据导致 OOM，正常日均量远低于此）
- 利用已有的 `idx_review_results_created` 索引

### 5.4 统计数据模型

```go
// model.go

type DailyStats struct {
    Date      string          // "2026-04-05"
    RepoStats []RepoStats     // 按仓库分组
    Total     AggregatedStats // 跨仓库汇总
}

type RepoStats struct {
    RepoFullName string
    Stats        AggregatedStats
    TopPRs       []TopPR
}

type AggregatedStats struct {
    ReviewCount    int     // 评审总数
    ApproveCount   int     // verdict="approve" 的次数
    RequestChanges int     // verdict="request_changes" 的次数
    CommentCount   int     // verdict="comment" 的次数（含解析失败降级为 comment 的场景）
    FailedCount    int     // parse_failed=true 的次数（解析失败，是 CommentCount 的子集）
    CriticalCount  int     // CRITICAL issue 总数（各记录 critical_count 之和）
    ErrorCount     int     // ERROR issue 总数（各记录 error_count 之和）
    WarningCount   int     // WARNING issue 总数（各记录 warning_count 之和）
    InfoCount      int     // INFO issue 总数（各记录 info_count 之和）
    AvgDurationMs  int64   // 平均评审耗时（各记录 duration_ms 的平均值）
    TotalCostUSD   float64 // 总费用（各记录 cost_usd 之和）
}

type TopPR struct {
    PRNumber    int64
    RepoName    string
    ReviewCount int    // 该 PR 被评审次数
    IssueCount  int    // 该 PR 的 issue 总数
    Reason      string // "review_count" 或 "issue_count"
}
```

**Verdict 判定逻辑**：直接读取 `ReviewRecord.Verdict` 字段值（`"approve"` / `"request_changes"` / `"comment"`），按字符串匹配归类。解析失败的降级记录 `Verdict="comment"` 且 `ParseFailed=true`，同时计入 `CommentCount` 和 `FailedCount`。

### 5.5 关键 PR 高亮逻辑

- **评审次数最多的 PR**：同一 `(repo, pr_number)` 出现次数最多（反映频繁 push 重新评审）
- **issue 数量最多的 PR**：`issue_count` 最大的记录（反映代码质量问题集中）
- 每种各取 Top 1，去重后最多 2 条

### 6.3 独立 Handler（report/handler.go）

`gen_daily_report` 使用独立的 `asynq.Handler` 实现，**不走现有的 `Processor.ProcessTask`**。

原因：现有 Processor 的 `ProcessTask` 假设所有任务都有 `TaskPayload` JSON 作为 payload（含 delivery_id、repo 信息等），并通过 `findRecord` 查找预创建的 TaskRecord。日报任务无 payload、无 delivery_id、无预创建 TaskRecord，生命周期完全不同。

```go
// internal/report/handler.go

type DailyReportHandler struct {
    store     store.Store
    generator *ReportGenerator
    logger    *slog.Logger
}

func (h *DailyReportHandler) ProcessTask(ctx context.Context, task *asynq.Task) error {
    // 1. 自行创建 TaskRecord
    record := &model.TaskRecord{
        ID:        "daily-report-" + uuid.NewString(),
        TaskType:  model.TaskTypeGenDailyReport,
        Status:    model.TaskStatusRunning,
        CreatedAt: time.Now(),
    }
    now := record.CreatedAt
    record.StartedAt = &now
    _ = h.store.CreateTask(ctx, record)

    // 2. 执行报告生成
    err := h.generator.Generate(ctx)

    // 3. 更新 TaskRecord
    completedAt := time.Now()
    record.CompletedAt = &completedAt
    record.UpdatedAt = completedAt
    if err != nil {
        record.Status = model.TaskStatusFailed
        record.Error = err.Error()
    } else {
        record.Status = model.TaskStatusSucceeded
    }
    _ = h.store.UpdateTask(ctx, record)

    return err // 返回 error 让 asynq 处理重试
}
```

### 6.4 ServeMux 注册（serve.go）

```go
// 在现有的 mux.Handle 注册之后
mux.Handle(queue.AsynqTypeGenDailyReport, dailyReportHandler)
```

日报 Handler 与现有的 Processor 并列注册在同一个 ServeMux 上，由 asynq 按任务类型路由。

### 6.5 关键设计点

- **独立 Handler**：`DailyReportHandler` 实现 `asynq.Handler`，独立于 `Processor`，避免侵入现有 Processor 的复杂逻辑
- **无 payload**：时间窗口由 ReportGenerator 根据当前时间和配置时区动态计算
- **重试**：显式注册 `MaxRetry(TaskMaxRetry())` 与日报默认超时，保持与项目其他任务一致；`ProcessTask` 返回 error 即触发 asynq 重试
- **分布式安全**：asynq Scheduler 基于 Redis 去重，多实例不会重复触发
- **优雅退出**：`scheduler.Shutdown()` 在 `gracefulShutdown` 的 Layer 2（asynq Server Shutdown）之后调用。Scheduler 需作为参数传入 `gracefulShutdown`，或通过 `ServiceDeps` 承载

## 7. 配置

### 7.1 Config 结构体

```go
// config.go 新增字段
type Config struct {
    // ...现有字段...
    DailyReport DailyReportConfig `mapstructure:"daily_report"`
}

type DailyReportConfig struct {
    Enabled       bool   `mapstructure:"enabled"`        // 默认 false
    Cron          string `mapstructure:"cron"`           // 默认 "0 9 * * *"
    Timezone      string `mapstructure:"timezone"`       // 默认 "Asia/Shanghai"
    SkipEmpty     bool   `mapstructure:"skip_empty"`     // 默认 false（无数据也发送）
    FeishuWebhook string `mapstructure:"feishu_webhook"` // 独立 Webhook URL
    FeishuSecret  string `mapstructure:"feishu_secret"`  // 签名密钥（可选）
}
```

**Clone() 说明**：`DailyReportConfig` 所有字段均为值类型（bool / string），`Config.Clone()` 的值拷贝已自动覆盖，无需额外深拷贝逻辑。

### 7.2 默认值

```go
viper.SetDefault("daily_report.enabled", false)
viper.SetDefault("daily_report.cron", "0 9 * * *")
viper.SetDefault("daily_report.timezone", "Asia/Shanghai")
viper.SetDefault("daily_report.skip_empty", false)
```

### 7.3 配置校验（仅 enabled=true 时）

- cron 表达式语法校验（`robfig/cron/v3` 解析；该包已作为 asynq 的间接依赖存在，校验时升级为直接依赖）
- timezone 合法性（`time.LoadLocation`）
- feishu_webhook 必填且 URL 格式校验（复用 `validate.go` 现有的 URL 校验模式）

### 7.4 示例配置

```yaml
daily_report:
  enabled: false
  cron: "0 9 * * *"
  timezone: "Asia/Shanghai"
  skip_empty: false
  feishu_webhook: ""
  feishu_secret: ""
```

### 7.5 热加载边界

`daily_report` 配置变更**需要重启**生效（Scheduler 在启动时注册 cron，运行时不重新注册）。

### 8.4 与 M2.6 卡片关系

- **复用**：飞书卡片 JSON 结构模式（`msg_type: interactive` + `card`）、签名算法（`genSign`）、HTTP 发送逻辑（`FeishuNotifier`）
- **独立实例**：日报使用 `DailyReportConfig.FeishuWebhook` 单独构造一个 `notify.FeishuNotifier` 实例，与 M2.6 事件通知的飞书 Notifier（来自 `notify.channels.feishu`）互相独立，两者可以指向不同的飞书群
- **新建**：`FormatDailyReportCard` 在 `internal/report/feishu_card.go` 中独立实现，布局与单条事件通知完全不同。它接受 `DailyStats` + `*DailyStats`（前日对比），返回 `map[string]any` 卡片结构

## 9. 错误处理

| 场景 | 处理 |
|------|------|
| Store 查询失败 | Handler 返回 error，TaskRecord 标记 failed，asynq 自动重试 |
| 飞书 Webhook 超时 | Handler 返回 error，TaskRecord 标记 failed，asynq 自动重试（最多 3 次） |
| 飞书返回非 0 code | Handler 返回 error，TaskRecord 标记 failed，记录错误详情 |
| 时区加载失败 | 启动时 Validate 校验拦截，不会运行到此 |
| 单日数据量超过硬上限（2000 条） | 截断处理，日志输出 warning，报告基于截断数据生成 |
