# M2.7 每日评审统计报告 — 设计文档

> 日期：2026-04-06
> 状态：已确认，待实施
> 依赖：M2.3（review_results 持久化）、M2.6（飞书通知渠道）

## 1. 概述

每日定时通过飞书通知发送前一天的 PR 评审统计摘要。一条聚合消息涵盖所有仓库，内部按仓库分组展示。

## 2. 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 定时调度 | asynq Scheduler | 已是项目依赖，天然支持 cron + 分布式去重 |
| 趋势数据 | 实时查询两天对比 | 数据量小（<10 仓库），避免额外存储 |
| 空数据处理 | 默认发送简短通知 | 充当系统心跳，团队可确认服务正常 |
| 发送粒度 | 一条聚合消息 | <10 仓库，一张卡片足够；避免消息轰炸 |
| 扩展架构 | 最小接口（StatCollector） | 成本极低，为 fix_issue/gen_tests 统计预留 |
| 架构模式 | 任务集成（Scheduler 入队 + Processor 处理） | 完整可观测性 + 自动重试 + 与现有模式一致 |

## 3. 包结构

```
internal/report/
  collector.go       # StatCollector 接口 + ReviewStatCollector 实现
  generator.go       # ReportGenerator 编排层（收集 -> 格式化 -> 发送）
  feishu_card.go     # 统计专用飞书卡片格式化
  model.go           # 统计数据模型（DailyStats / RepoStats）
```

## 4. 端到端数据流

```
serve.go 启动
  |
  |  asynq.NewScheduler(redisOpt, Location=配置时区)
  |  scheduler.Register(cron, "dtworkflow:gen_daily_report")
  |  scheduler.Run()
  |
  v  cron 触发
asynq.Server (Processor)
  |
  |  case "gen_daily_report":
  |    1. 创建 TaskRecord（类型 gen_daily_report，状态 processing）
  |    2. reportGenerator.Generate(ctx)
  |    3. 更新 TaskRecord（succeeded / failed）
  |
  v
ReportGenerator.Generate()
  |
  |  1. 计算时间窗口（昨天 00:00 ~ 今天 00:00，按配置时区）
  |  2. collector.Collect(ctx, yesterday)    -> DailyStats
  |  3. collector.Collect(ctx, dayBefore)    -> DailyStats（趋势对比）
  |  4. FormatDailyReportCard(stats, prevStats)
  |  5. feishuNotifier.Send(ctx, msg)
  v
飞书群收到统计卡片
```

关键点：
- **不经过 Docker 容器**：`gen_daily_report` 在 Processor 内直接执行
- **不经过 notify.Router**：日报有独立的 Webhook 配置，直接构造 FeishuNotifier 发送
- **时区处理**：时间窗口按配置的 timezone 计算本地日期边界，再转 UTC 查询数据库

## 5. 数据层

### 5.1 Store 新增方法

```go
// store.go 接口新增
ListReviewResultsByTimeRange(ctx context.Context, start, end time.Time) ([]*model.ReviewRecord, error)
```

- 查询 `created_at >= start AND created_at < end`，返回所有仓库的记录
- 按 `created_at DESC` 排序
- 无分页（日均评审量可控，单日数据全量加载到内存）
- 利用已有的 `idx_review_results_created` 索引

### 5.2 StatCollector 接口

```go
// collector.go

type TimeRange struct {
    Start time.Time  // inclusive
    End   time.Time  // exclusive
}

type StatCollector interface {
    Collect(ctx context.Context, tr TimeRange) (*DailyStats, error)
}
```

M2.7 只实现 `ReviewStatCollector`，未来可插拔新增 fix_issue / gen_tests 等收集器。

### 5.4 统计数据模型

```go
// model.go

type DailyStats struct {
    Date      string          // "2026-04-05"
    RepoStats []RepoStats     // 按仓库分组
    Total     AggregatedStats // 跨仓库汇总
}

type RepoStats struct {
    RepoFullName string
    Stats        AggregatedStats
    TopPRs       []TopPR
}

type AggregatedStats struct {
    ReviewCount    int     // 评审总数
    ApproveCount   int     // approve 次数
    RequestChanges int     // request_changes 次数
    CommentCount   int     // comment 次数（降级场景）
    FailedCount    int     // 解析失败次数
    CriticalCount  int     // CRITICAL issue 总数
    ErrorCount     int     // ERROR issue 总数
    WarningCount   int     // WARNING issue 总数
    InfoCount      int     // INFO issue 总数
    AvgDurationMs  int64   // 平均评审耗时
    TotalCostUSD   float64 // 总费用
}

type TopPR struct {
    PRNumber    int64
    RepoName    string
    ReviewCount int    // 该 PR 被评审次数
    IssueCount  int    // 该 PR 的 issue 总数
    Reason      string // "review_count" 或 "issue_count"
}
```

### 5.5 关键 PR 高亮逻辑

- **评审次数最多的 PR**：同一 `(repo, pr_number)` 出现次数最多（反映频繁 push 重新评审）
- **issue 数量最多的 PR**：`issue_count` 最大的记录（反映代码质量问题集中）
- 每种各取 Top 1，去重后最多 2 条

## 6. asynq Scheduler 集成

### 6.1 任务类型

```go
// queue/client.go
const AsynqTypeGenDailyReport = "dtworkflow:gen_daily_report"

// model/task.go
const TaskTypeGenDailyReport TaskType = "gen_daily_report"
```

### 6.2 Scheduler 初始化（serve.go）

```go
if cfg.DailyReport.Enabled {
    loc, _ := time.LoadLocation(cfg.DailyReport.Timezone) // 已在 Validate 校验
    scheduler := asynq.NewScheduler(redisOpt, &asynq.SchedulerOpts{
        Location: loc,
    })

    task := asynq.NewTask(queue.AsynqTypeGenDailyReport, nil) // 无 payload
    _, err := scheduler.Register(cfg.DailyReport.Cron, task)
    // ...错误处理

    go func() { _ = scheduler.Run() }()
    defer scheduler.Shutdown()
}
```

### 6.3 Processor 路由

```go
// queue/processor.go ProcessTask 方法内

case AsynqTypeGenDailyReport:
    // 不需要 Docker 容器，直接在 Processor 内执行
    record := &model.TaskRecord{
        ID:        uuid.New().String(),
        TaskType:  model.TaskTypeGenDailyReport,
        Status:    model.TaskStatusProcessing,
        CreatedAt: time.Now(),
    }
    p.store.CreateTask(ctx, record)

    err := p.reportGenerator.Generate(ctx)

    if err != nil {
        record.Status = model.TaskStatusFailed
        record.Result = err.Error()
    } else {
        record.Status = model.TaskStatusSucceeded
    }
    p.store.UpdateTask(ctx, record)
```

### 6.4 关键设计点

- **ProcessorOption 注入**：`ReportGenerator` 通过现有 `ProcessorOption` 模式注入
- **无 payload**：时间窗口由 ReportGenerator 根据当前时间和配置时区动态计算
- **重试**：复用 asynq 默认重试（指数退避），最多 3 次
- **分布式安全**：asynq Scheduler 基于 Redis 去重，多实例不会重复触发
- **优雅退出**：`scheduler.Shutdown()` 在 serve 退出路径中调用

## 7. 配置

### 7.1 Config 结构体

```go
type DailyReportConfig struct {
    Enabled       bool   `mapstructure:"enabled"`        // 默认 false
    Cron          string `mapstructure:"cron"`           // 默认 "0 9 * * *"
    Timezone      string `mapstructure:"timezone"`       // 默认 "Asia/Shanghai"
    SkipEmpty     bool   `mapstructure:"skip_empty"`     // 默认 false（无数据也发送）
    FeishuWebhook string `mapstructure:"feishu_webhook"` // 独立 Webhook URL
    FeishuSecret  string `mapstructure:"feishu_secret"`  // 签名密钥（可选）
}
```

### 7.2 默认值

```go
viper.SetDefault("daily_report.enabled", false)
viper.SetDefault("daily_report.cron", "0 9 * * *")
viper.SetDefault("daily_report.timezone", "Asia/Shanghai")
viper.SetDefault("daily_report.skip_empty", false)
```

### 7.3 配置校验（仅 enabled=true 时）

- cron 表达式语法校验（robfig/cron/v3 解析）
- timezone 合法性（time.LoadLocation）
- feishu_webhook 必填且 URL 格式校验

### 7.4 示例配置

```yaml
daily_report:
  enabled: false
  cron: "0 9 * * *"
  timezone: "Asia/Shanghai"
  skip_empty: false
  feishu_webhook: ""
  feishu_secret: ""
```

### 7.5 热加载边界

`daily_report` 配置变更**需要重启**生效（Scheduler 在启动时注册 cron，运行时不重新注册）。

### 8.3 趋势指示

| 变化 | 符号 | 说明 |
|------|------|------|
| 增加 | 趋势+N | 较前日增加 N |
| 减少 | 趋势-N | 较前日减少 N |
| 持平 | 持平 | 数值不变 |
| 无前日数据 | 不显示 | 首次报告或前日无数据时省略 |

### 8.4 与 M2.6 卡片关系

- **复用**：飞书卡片 JSON 结构、签名算法、HTTP 发送逻辑（FeishuNotifier）
- **新建**：`FormatDailyReportCard` 独立实现，布局与单条事件通知完全不同

## 9. 错误处理

| 场景 | 处理 |
|------|------|
| Store 查询失败 | 任务标记 failed，asynq 自动重试 |
| 飞书 Webhook 超时 | 任务标记 failed，asynq 自动重试（最多 3 次） |
| 飞书返回非 0 code | 任务标记 failed，记录错误详情 |
| 时区加载失败 | 启动时 Validate 校验拦截，不会运行到此 |
