# M2.4 重新评审机制设计文档

> 日期：2026-03-26
> 状态：已完成实施（2026-03-26），含两轮审核修复
> 依赖：M2.1-M2.3 已完成
>
> ### 实施偏差记录
>
> 以下为实施过程中相对原始设计的调整（提交 `a27f66f` 修复 + `eb0b08c` 审核修复）：
>
> 1. **取消时序修正**：原设计 Section 6.2 中"先取消再入队"改为"先入队再取消"。
>    `cancelActivePRTasks` 拆分为 `listActivePRTasks`（纯查询）+ `cancelTasks`（纯执行），
>    在 `enqueueTask` 成功后才执行取消，避免新任务创建失败导致 PR 无可用评审。
> 2. **cancelTask 返回值**：原设计中 `cancelTask` 无返回值，实现改为返回 `bool`。
>    asynq 操作失败时 `return false` 并保留旧任务原始状态，避免 SQLite 与 asynq 状态不一致。
>    SQLite UpdateTask 失败时也返回 false。
> 3. **cancelTasks 可观测性**：新增失败计数日志，记录取消失败的任务数量。
> 4. **Processor pre-cancelled 检查**：原设计 Section 7 未涵盖。新增 `record.Status == cancelled`
>    提前返回检查，防止已取消但仍被 asynq 分发的任务被执行。
> 5. **ErrStaleReview 传播链**：原设计 Section 8 中 Writer 返回 ErrStaleReview 后，
>    Service 和 Processor 均需识别该错误。Service.Execute 新增 `errors.Is(wbErr, ErrStaleReview)` 提前返回，
>    Processor 新增 `markTaskCancelled` 处理 ErrStaleReview。
> 6. **HasNewerReviewTask 状态过滤**：原设计 Section 5.2 SQL 不过滤任务状态，
>    审核后新增 `AND status NOT IN (cancelled, failed)` 条件，避免终态任务误判 staleness。
> 7. **SQL 参数化**：FindActivePRTasks 和 HasNewerReviewTask 的 SQL 从硬编码字符串改为
>    引用 `model.TaskStatus*` / `model.TaskType*` 常量，降低维护风险。
> 8. **ErrStaleReview 声明**：从 `fmt.Errorf` 改为 `errors.New`（哨兵错误惯例）。

## 1. 目标

实现 PR 更新（新 push）时的自动重新评审机制，核心解决三个问题：

1. **自动触发**：PR 有新 push 时自动触发新一轮评审
2. **防抖/去重**：短时间内多次 push 不产生冗余评审
3. **旧任务管理**：新评审入队时取消旧评审，避免资源浪费

### 不在本轮范围

- **增量评审**：每次 push 仍为全量评审（Claude Code 在完整仓库中执行），增量 prompt 优化属于 M2.5
- **评审配置精细化**：文件过滤、维度开关等属于 M2.5

## 2. 策略选型

### Cancel-and-Replace（取消并替代）

每次 `synchronized` 事件正常入队，但新任务入队前主动取消同一 PR 尚未完成的旧评审任务。

**对比其他方案**：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 入队层防抖（延迟入队） | 只执行一次评审 | 需要定时器 + 异步状态管理，增加响应延迟 | 不采用 |
| Cancel-and-Replace | 实现直接，响应快，与 asynq 原生能力契合 | running 任务取消有资源浪费 | **采用** |
| asynq UniqueKey | 利用原生能力 | 不支持替换语义，对 running 无效 | 不采用 |

### 旧评审 Gitea 侧处理

不删除或 dismiss 旧评审。新评审 summary 开头标注"本评审替代了之前的评审"，用户可看到评审演进历史。

## 3. 整体数据流

```
Gitea PR push
  -> Webhook (X-Gitea-Event: pull_request, action: synchronized)
  -> Receiver.Handle -> Parser.parsePullRequest
  -> EnqueueHandler.HandlePullRequest
      |-- 1. 幂等检查（delivery_id，现有逻辑不变）
      |-- 2. [新增] 查找同一 PR 的活跃旧任务 (pending/queued/running)
      |-- 3. [新增] 逐一取消旧任务
      |       |-- pending/queued -> asynq DeleteTask + SQLite status=cancelled
      |       +-- running -> asynq CancelProcessing + SQLite status=cancelled
      |-- 4. 构建新任务 payload（含 pr_number 冗余列 + 替代信息）
      +-- 5. enqueueTask（现有逻辑不变）

新任务执行：
  -> Processor.ProcessTask -> review.Service.Execute
      |-- ... 现有评审流程 ...
      +-- Writer.Write
            |-- [新增] staleness check：查询是否存在更新的同 PR 评审任务
            |     +-- 存在 -> 跳过回写，记录日志
            |-- [新增] summary 中标注"替代旧评审"信息
            +-- ... 现有回写逻辑 ...
```

**关键约束**：
- 取消操作是 best-effort（取消失败不阻断新任务入队）
- running 任务的取消依赖 asynq 的 `CancelProcessing`（向 handler 的 ctx 发送取消信号）
- 回写 staleness check 是最终安全网，确保不会出现旧评审覆盖新评审

## 5. Store 层变更

### 5.1 新增 FindActivePRTasks

```go
// FindActivePRTasks 查找同一 PR 的活跃评审任务（pending/queued/running）
// 返回按 created_at 升序排列的任务列表（最旧的在前）
FindActivePRTasks(ctx context.Context, repoFullName string, prNumber int64, taskType model.TaskType) ([]*model.TaskRecord, error)
```

SQL 实现：

```sql
SELECT {taskColumns}
FROM tasks
WHERE repo_full_name = ?
  AND pr_number = ?
  AND task_type = ?
  AND status IN ('pending', 'queued', 'running')
ORDER BY created_at ASC
```

命中复合索引 `idx_tasks_repo_pr`，查询高效。同一 PR 的活跃任务数量极少（正常 1-2 个），无需 LIMIT。

### 5.2 新增 HasNewerReviewTask

```go
// HasNewerReviewTask 检查是否存在比指定时间更新的同 PR 评审任务
// 用于回写前的 staleness 检查
HasNewerReviewTask(ctx context.Context, repoFullName string, prNumber int64, afterCreatedAt time.Time) (bool, error)
```

SQL 实现：

```sql
SELECT EXISTS(
  SELECT 1 FROM tasks
  WHERE repo_full_name = ?
    AND pr_number = ?
    AND task_type = 'review_pr'
    AND created_at > ?
)
```

> **已知限制**：`created_at` 使用 RFC3339Nano 字符串比较，极端情况下两个任务在同一纳秒内创建可能无法区分。该概率极低（需要 Go 的 `time.Now()` 在两次顺序调用中返回相同纳秒），可接受。

## 6. EnqueueHandler 取消逻辑

### 6.1 接口设计

遵循接口隔离原则，新增独立的 `TaskCanceller` 接口，避免扩展 `Enqueuer` 导致语义模糊：

```go
// TaskCanceller 任务取消能力接口（与 Enqueuer 分离）
type TaskCanceller interface {
    // Delete 从队列中删除 pending/queued 任务。
    // queueName 通过 PriorityToQueue(record.Priority) 推导。
    Delete(ctx context.Context, queueName, taskID string) error
    // CancelProcessing 向正在执行的任务 handler 发送取消信号
    CancelProcessing(ctx context.Context, taskID string) error
}
```

`Client` 需持有 `asynq.Inspector` 实例（与 `asynq.Client` 共享同一 `RedisConnOpt`），底层调用 `Inspector.DeleteTask(queueName, taskID)` 和 `Inspector.CancelProcessing(taskID)`。

> **注意**：`Inspector.DeleteTask` 需要 queue name 参数。取消时从 `record.Priority` 通过已有的 `PriorityToQueue()` 函数推导，无需新增存储字段。

`EnqueueHandler` 新增 `canceller TaskCanceller` 依赖：

```go
type EnqueueHandler struct {
    client    Enqueuer
    canceller TaskCanceller // 新增
    store     store.Store
    logger    *slog.Logger
}
```

> **pending 状态的 AsynqID 为空**：任务停留在 pending 状态（入队失败等待 RecoveryLoop）时没有 AsynqID，此时 cancelTask 中的 asynq 操作会被 `task.AsynqID != ""` 条件跳过，仅更新 SQLite 状态。

### 6.2 HandlePullRequest 改动

在幂等检查之后、构建 payload 之前，插入取消逻辑：

```go
func (h *EnqueueHandler) HandlePullRequest(ctx context.Context, event webhook.PullRequestEvent) error {

}
```

### 6.3 cancelActivePRTasks 实现

```go
// SupersededInfo 取消旧任务后的汇总信息
type SupersededInfo struct {
    Count       int    // 取消的旧任务数量
    LastHeadSHA string // 最近一个旧任务的 HeadSHA
}

func (h *EnqueueHandler) cancelActivePRTasks(ctx context.Context, repoFullName string, prNumber int64) SupersededInfo {
    tasks, err := h.store.FindActivePRTasks(ctx, repoFullName, prNumber, model.TaskTypeReviewPR)
    if err != nil {
        h.logger.WarnContext(ctx, "查找活跃旧任务失败，跳过取消",
            "repo", repoFullName, "pr", prNumber, "error", err)
        return SupersededInfo{}
    }
    if len(tasks) == 0 {
        return SupersededInfo{}
    }

    info := SupersededInfo{}
    for _, task := range tasks {
        h.cancelTask(ctx, task)
        info.Count++
        if task.Payload.HeadSHA != "" {
            info.LastHeadSHA = task.Payload.HeadSHA
        }
    }
    return info
}
```

### 6.4 cancelTask 实现

```go
func (h *EnqueueHandler) cancelTask(ctx context.Context, task *model.TaskRecord) {
    prevStatus := task.Status

    if task.AsynqID != "" {
        switch task.Status {
        case model.TaskStatusPending, model.TaskStatusQueued:
            // 从 asynq 队列中删除，需要 queue name
            queueName := PriorityToQueue(task.Priority)
            if err := h.canceller.Delete(ctx, queueName, task.AsynqID); err != nil {
                h.logger.WarnContext(ctx, "从 asynq 删除任务失败",
                    "task_id", task.ID, "asynq_id", task.AsynqID, "error", err)
            }
        case model.TaskStatusRunning:
            if err := h.canceller.CancelProcessing(ctx, task.AsynqID); err != nil {
                h.logger.WarnContext(ctx, "取消运行中任务失败",
                    "task_id", task.ID, "asynq_id", task.AsynqID, "error", err)
            }
        }
    }
    // AsynqID 为空时（pending 未入队），仅更新 SQLite 状态

    now := time.Now()
    task.Status = model.TaskStatusCancelled
    task.Error = "被同一 PR 的新评审任务取代"
    task.UpdatedAt = now
    task.CompletedAt = &now
    if err := h.store.UpdateTask(ctx, task); err != nil {
        h.logger.WarnContext(ctx, "更新旧任务状态为 cancelled 失败",
            "task_id", task.ID, "error", err)
    } else {
        h.logger.InfoContext(ctx, "已取消旧评审任务",
            "task_id", task.ID, "prev_status", prevStatus, "pr", task.PRNumber)
    }
}
```

## 7. Processor 层 context 取消响应

### 7.1 Worker Pool — 无需改动

经代码验证，现有的 ctx 取消链路已经完整：

1. `pool.go` 中 `waitCtx` 派生自调用方 `ctx`：`waitCtx, waitCancel := context.WithTimeout(ctx, 30*time.Minute)`
2. `docker.go` 的 `WaitContainer` 内部已有 `select { case <-ctx.Done() }` 分支，ctx 取消时返回 `ctx.Err()`
3. `pool.go` 的 `defer RemoveContainer` 使用 `context.Background()` 独立清理容器，不受调用方 ctx 取消影响

因此，当 asynq `CancelProcessing` 取消 Processor 的 ctx 时：
- `waitCtx` 被取消 → `WaitContainer` 返回 `context.Canceled`
- `pool.runContainer` 返回错误 → `defer RemoveContainer(context.Background(), ...)` 强制删除容器（含停止）
- 错误传播回 Processor

**`internal/worker/pool.go` 无需改动。**

### 7.2 Processor.ProcessTask

新增 `context.Canceled` 错误分支（在现有 `ErrPRNotOpen` 判断之前）：

```go
if runErr != nil {
    if errors.Is(runErr, context.Canceled) {

        record.CompletedAt = &completedAt

            }
        }
        return fmt.Errorf("任务被取消: %w", asynq.SkipRetry)
    }
    // ... 现有的 ErrPRNotOpen / shouldRetry 逻辑 ...
}
```

## 8. 回写 Staleness Check

### 8.1 StaleChecker 接口

```go
// StaleChecker 检查评审任务是否已过时
type StaleChecker interface {
    HasNewerReviewTask(ctx context.Context, repoFullName string, prNumber int64, afterCreatedAt time.Time) (bool, error)
}
```

SQLiteStore 已实现此方法（Section 5.2），直接满足接口。

### 8.2 Writer 改动

```go
type Writer struct {
    gitea        WritebackClient
    store        ReviewStore
    staleChecker StaleChecker // 新增，可选（nil 时跳过检查）
    logger       *slog.Logger
}

type WritebackInput struct {
    TaskID          string
    Owner           string
    Repo            string
    PRNumber        int64
    HeadSHA         string
    Result          *ReviewResult
    TaskCreatedAt   time.Time // 新增：staleness 比较基准
    SupersededCount int       // 新增：替代标注
    PreviousHeadSHA string    // 新增：替代标注

```

在 `Write` 方法中，`CreatePullReview` 调用之前插入 staleness check：

- staleChecker 为 nil → 跳过检查
- check 失败 → 记日志，继续回写（fail-open）
- 判定过时 → 跳过回写，返回错误

### 8.3 Summary 替代标注

当前 `formatReviewBody` 已有 6 个参数，再添加 superseded 信息会导致签名过长。重构为接受 `FormatOptions` 结构体：

```go
// FormatOptions formatReviewBody 的输入参数
type FormatOptions struct {
    Review          *ReviewOutput
    Unmapped        []ReviewIssue
    ParseFailed     bool
    RawOutput       string
    DurationSec     float64
    CostUSD         float64
    SupersededCount int    // M2.4 新增
    PreviousHeadSHA string // M2.4 新增
}

func formatReviewBody(opts FormatOptions) string { ... }
```

当 `SupersededCount > 0` 时，在 summary 开头插入：

```
> 本评审基于最新提交，替代了之前基于 `abc1234` 的评审。
```

无 PreviousHeadSHA 时简化为：

```
> 本评审基于最新提交，替代了之前的评审。
```

### 8.4 TaskCreatedAt 传递路径

`Service.Execute` 需要 `TaskCreatedAt` 以传递给 `WritebackInput`。传递路径：

1. `EnqueueHandler.enqueueTask` 设置 `payload.CreatedAt = record.CreatedAt`（TaskPayload 新增 `CreatedAt` 字段，见 Section 4.4）
2. `Processor.ProcessTask` 从 SQLite 查出 TaskRecord 后，补充 `payload.CreatedAt = record.CreatedAt`（覆盖，确保一致）
3. `review.Service.Execute` 从 `payload.CreatedAt` 读取，填入 `WritebackInput.TaskCreatedAt`

## 10. 错误处理与边界情况

| 场景 | 处理方式 |
|------|----------|
| 首次评审（opened），无旧任务 | cancelActivePRTasks 返回空，正常入队，不显示替代标注 |
| asynq DeleteTask 失败（任务已被消费） | 记日志，SQLite 仍标记 cancelled，staleness check 兜底 |
| asynq CancelProcessing 失败（任务已完成或已取消） | 记日志，不影响新任务入队 |
| 取消信号与容器完成竞争 | 容器先完成则走回写 + staleness check；取消先到则容器停止 |
| 多次快速 push（A->B->C） | B 取消 A，C 取消 B（和可能残留的 A），最终只有 C 回写 |
| FindActivePRTasks 查询失败 | 记日志，跳过取消，新任务正常入队 |
| staleness check 查询失败 | 记日志，继续回写（fail-open） |
| 旧 running 任务的 Processor 和 EnqueueHandler 并发更新同一记录 | Processor 可能将已被标记为 cancelled 的记录覆盖为 running/succeeded（`UpdateTask` 无乐观锁）。这是可接受的：staleness check 在回写时兜底，确保过时结果不会发布到 Gitea。SQLite 中的最终状态可能不是 cancelled，但业务语义正确。 |
| pending 状态任务无 AsynqID（入队失败等待 RecoveryLoop） | cancelTask 中 `task.AsynqID != ""` 条件跳过 asynq 操作，仅更新 SQLite 状态为 cancelled。RecoveryLoop 后续扫描时发现状态非 pending，不会重新入队。 |
| PR 被关闭后收到 synchronized 事件 | Service.Execute 中 ErrPRNotOpen 校验兜底 |

**错误处理原则**：

1. 取消路径全部 best-effort：任何失败只记日志，不阻断新任务
2. 回写保护 fail-open：staleness check 失败时继续回写
3. 状态更新幂等：多个地方可能标记同一任务为 cancelled，均安全
