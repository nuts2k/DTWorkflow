# M5.1 E2E 基础设施与执行引擎 — 设计文档

> 日期：2026-05-09
> 状态：设计确认
> 依赖：Phase 1（基础设施）+ Phase 3（Issue auto-fix）+ Phase 4（镜像策略）

## 1. 概述

### 1.1 目标

为 DTWorkflow 新增 E2E 测试执行能力：用户在仓库中编写 E2E 用例（`case.yaml` + 脚本），通过 CLI/API 手动触发，DTWorkflow 在容器内由 Claude Code CLI 编排执行 Playwright 浏览器测试和 SQL/JS 数据脚本，产出结构化测试报告。

### 1.2 职责边界

- **人负责**：编写 case.yaml（业务意图）+ Playwright/SQL/JS 脚本（技术实现）
- **平台负责**：执行、报告、失败分析（M5.2 扩展 Issue 创建 + auto-fix 联动）

### 1.3 不做什么

- 不做测试用例自动生成（场景定义需要业务知识）
- 不做 YAML → 脚本翻译
- 不做测试环境部署/管理（对接现有部署流程）
- 不做容器内实时脚本修复（失败后由 M5.2 创建 Issue 走 Phase 3 链路）
- 不做 `suite.yaml`（M5.1 由 Claude 按目录约定自动发现用例；M5.3 模块拆分并行时如需声明子用例执行顺序和公共配置，届时引入 `suite.yaml`）

### 1.4 关键设计决策

| # | 决策 | 理由 |
|---|------|------|
| D1 | Claude 全编排（非 shell/node runner） | 与 Phase 2-4 架构一致；Claude 拥有完整执行上下文，失败分析质量更高 |
| D2 | 镜像基于轻量 worker（非 worker-full） | E2E 不需要 JDK/Maven；被测应用在外部已部署 |
| D3 | 命名环境 + 名称引用 | 凭证只存 config 不经 CLI/API；热加载支持动态增减环境 |
| D4 | case.yaml = 业务意图 + 执行清单 | 合一文件降低维护成本；expectations 段是 auto-fix 的修复基准 |
| D5 | 用例间完全独立 | 每个用例自包含 setup/teardown，牺牲少量数据初始化成本换取可靠性 |
| D6 | 截图通过宿主机 Volume 收集 | JSON 输出保持轻量；二进制文件可靠持久化；M5.2 创建 Issue 时直接读取 |
| D7 | M5.1 用例发现由 Claude 容器内完成 | 宿主侧不解析仓库 e2e/ 结构；M5.3 并行拆分时再加宿主侧模块发现 |
| D8 | 触发粒度：模块 + 用例级 | 模块级覆盖常见场景；用例级方便开发者调试脚本 |

## 2. 镜像与容器基础设施

### 2.1 三级镜像策略

新增 `build/Dockerfile.worker-e2e`，基于轻量 worker 镜像：

```
Dockerfile.worker        → Node.js + Claude CLI + Git        (review_pr / analyze_issue)
Dockerfile.worker-full   → + JDK + Maven                     (fix_issue / gen_tests)
Dockerfile.worker-e2e    → worker + Playwright + Chromium     (run_e2e)
                           + mysql-client
```

镜像内关键安装步骤：

- `npx playwright install --with-deps chromium`（预装浏览器 + 系统依赖）
- `apt-get install default-mysql-client`
- Playwright 浏览器预装在镜像层，运行时不再下载（ReadonlyRootfs 兼容）

### 2.2 容器配置

`PoolConfig` 新增 `ImageE2E` 字段，`resolveImage` 按 `TaskTypeRunE2E` 选择镜像：

| 任务类型 | 镜像 |
|----------|------|
| `review_pr` / `analyze_issue` | `Image`（轻量） |
| `fix_issue` / `gen_tests` | `ImageFull` |
| `run_e2e` | `ImageE2E` |

E2E 容器特殊配置（`runContainer` 中按 TaskType 条件设置）：

- **ReadonlyRootfs**：保持启用
- **`/dev/shm`**：512MB（`ShmSize`，Chromium 渲染需求）
- **`/workspace` tmpfs**：4GB（仓库 + Playwright 缓存 + npm）
- **Artifact Volume**：宿主机 `{data_dir}/e2e-artifacts/{task_id}/` 挂载到容器 `/workspace/artifacts/`
- **网络**：复用 `dtworkflow-net` bridge（访问测试环境 + Claude API + Gitea）

配置段扩展：

```yaml
worker:
  image_e2e: "dtworkflow-worker-e2e:latest"
  timeouts:
    run_e2e: 60m    # E2E 多用例串行，默认超时需大于单用例超时之和
```

`ImageE2E` 空值语义：为空时 `resolveImage` 回退到 `Image`（与 `ImageFull` 一致）。`PoolConfig.Validate()` 需同步新增 `ImageE2E` 格式校验（非空时不含空格，与 `ImageFull` 对齐）。

### 2.3 entrypoint.sh `run_e2e` 分支

新增 `run_e2e)` case，行为类似 `analyze_issue`（只读模式）：

- Clone 仓库 + checkout 指定 ref（默认分支或 `--ref` 指定）
- **不需要** credential helper / git push 能力（E2E 不产生代码变更）
- **不需要** pre-push hook
- 注入 E2E 环境变量：`E2E_BASE_URL`、`E2E_DB_*`、`E2E_ACCOUNT_*`
- 从 `E2E_DB_*` 生成 `/tmp/.my.cnf`（权限 600），之后 `unset E2E_DB_PASSWORD`
- 设置 Playwright 输出目录：`PLAYWRIGHT_OUTPUT_DIR=/workspace/artifacts/test-results`、`PLAYWRIGHT_REPORT_DIR=/workspace/artifacts/reports`
- 敏感环境变量清除同现有模式（`unset GITEA_TOKEN` 等）
- 账号密码处理：将 `E2E_ACCOUNT_*` 写入 `/tmp/.e2e-accounts.json`（权限 600），Playwright 脚本通过读取该文件获取账号信息，之后 `unset E2E_ACCOUNT_*_PASSWORD`。这与 `E2E_DB_PASSWORD` 的处理方式一致（凭证不保留在环境变量中）。如用户仓库已有通过 `process.env` 读取账号的 Playwright 脚本，可在 `playwright.config.ts` 中加载该文件并注入 `process.env`，脚本无需修改
- **安全风险登记**：测试账号凭证通过文件传递而非环境变量，降低了 Claude 直接读取密码的风险。测试账号本身应为专用测试环境账号（非生产凭证），容器隔离 + 销毁后消失提供额外保障。此风险登记于 `docs/SECURITY.md`

### 2.4 Makefile

新增 `build-worker-e2e` 构建目标（依赖 `build-worker`）。

## 3. 用例模型与目录结构

### 3.1 目录约定

```
repo/
└── e2e/
    ├── playwright.config.ts            # 可选，Playwright 全局配置
    ├── order/                          # 模块 = 业务领域
    │   └── cases/
    │       ├── create-order/           # 用例目录
    │       │   ├── case.yaml           # 意图 + 执行清单
    │       │   ├── setup.sql
    │       │   ├── seed-data.js
    │       │   ├── create-order.spec.ts
    │       │   └── teardown.sql
    │       └── cancel-order/
    │           ├── case.yaml
    │           ├── cancel-order.spec.ts
    │           └── teardown.sql
    └── auth/                           # 另一个模块
        └── cases/
            └── login/
                ├── case.yaml
                └── login.spec.ts
```

约定规则：

- `e2e/` 为固定根目录
- 第一级子目录 = 模块（M5.3 模块拆分的发现依据）
- 模块下固定 `cases/` 子目录
- `cases/` 下每个子目录 = 一个用例，必须包含 `case.yaml`
- `playwright.config.ts` 可选，放在 `e2e/` 根目录，所有模块共享

### 3.2 `case.yaml` Schema

```yaml
# 必填字段
name: "创建工单后列表页正确展示"

# 可选字段
description: "验证完整的工单创建和列表展示流程"
timeout: 5m                            # 默认 5m，合法范围 [30s, 30m]
tags: [order, crud, smoke]             # 预留字段，M5.1 暂不做过滤

# 业务意图（auto-fix 的修复基准）
expectations:
  - step: "以 admin 账号登录系统"
    expect: "登录成功，跳转到首页"
  - step: "导航到工单创建页，填写标题'测试工单'，优先级选择'高'，点击提交"
    expect: "页面提示创建成功"
  - step: "导航到工单列表页"
    expect: "列表中包含标题为'测试工单'的条目，状态显示'待处理'，优先级显示'高'"

# 执行清单（有序数组，按声明顺序执行）
setup: [setup.sql, seed-data.js]       # 数据准备，可选
test: [create-order.spec.ts]           # 测试脚本，必填且非空
teardown: [teardown.sql]               # 清理脚本，可选
```

Schema 校验规则（`e2e.Service` 中校验）：

- `name` 必填，非空
- `test` 必填，至少一个脚本
- `setup` / `teardown` 可选
- `timeout` 合法范围 `[30s, 30m]`，默认 5 分钟。注：单用例 timeout 由 Claude 在容器内控制（prompt 约束），容器级 `worker.timeouts.run_e2e`（默认 60m）是硬上限；多用例串行场景下容器超时应 > 各用例 timeout 之和
- 引用的脚本文件必须在同目录下存在（Claude 执行前校验并报错）
- `expectations` 可选但强烈建议填写

### 3.3 脚本类型与执行约定

Claude 在 prompt 中被指引按文件扩展名选择执行方式：

| 扩展名 | 执行命令 | 说明 |
|--------|---------|------|
| `.sql` | `mysql --defaults-extra-file=/tmp/.my.cnf < script.sql` | 连接信息从 env 写入 `.my.cnf` |
| `.js` | `node script.js` | Node.js 脚本 |
| `.ts`（非 `.spec.ts`） | `npx tsx script.ts` | TypeScript 脚本 |
| `.spec.ts` | `npx playwright test script.spec.ts --reporter=json` | Playwright 测试 |

### 3.4 Playwright 输出目录（环境变量 fallback 模式）

`playwright.config.ts` 使用环境变量实现本地/自动化双环境兼容：

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  outputDir: process.env.PLAYWRIGHT_OUTPUT_DIR || './test-results',
  use: {
    baseURL: process.env.E2E_BASE_URL || 'http://localhost:3000',
    screenshot: 'only-on-failure',
    trace: 'on-first-retry',
  },
  reporter: [
    ['list'],
    ['json', {
      outputFile: `${process.env.PLAYWRIGHT_REPORT_DIR || './playwright-report'}/results.json`
    }],
  ],
});
```

| 运行环境 | 环境变量 | 截图输出 | 报告输出 |
|----------|---------|---------|---------|
| 本地开发 | 不设（用默认值） | `./test-results/` | `./playwright-report/` |
| DTWorkflow 容器 | entrypoint.sh 注入 | `/workspace/artifacts/test-results/` | `/workspace/artifacts/reports/` |

### 3.5 Artifact 宿主机清理

可配置保留天数（`e2e.artifact_retention_days`，默认 7 天，范围 `[1, 90]`），由 dtworkflow 定期清理过期 artifact。

## 4. 测试环境配置

### 4.1 命名环境配置结构

```yaml
e2e:
  enabled: true
  default_env: staging
  artifact_retention_days: 7

  environments:
    staging:
      base_url: "https://staging.example.com"
      db:
        host: "staging-db.internal"
        port: 3306
        user: "e2e_test"
        password: "xxx"
        database: "app_test"
      accounts:
        admin:
          username: "admin"
          password: "admin123"
        normal:
          username: "testuser01"
          password: "test123"

    dev:
      base_url: "https://dev.example.com"
      db:
        host: "dev-db.internal"
        port: 3306
        user: "e2e_test"
        password: "yyy"
        database: "app_dev"
      accounts:
        admin:
          username: "admin"
          password: "admin456"
```

### 4.2 仓库级覆盖

```yaml
repos:
  "changyu/backend":
    e2e:
      enabled: true
      default_env: staging
  "changyu/admin":
    e2e:
      enabled: true
      default_env: dev
```

`E2EOverride` 结构：

- `Enabled *bool`（nil = 未覆盖，跟随全局；`*false` = 显式禁用）
- `DefaultEnv string`（空 = 跟随全局）

`ResolveE2EConfig` 合并逻辑与 `ResolveReviewConfig` / `ResolveTestGenConfig` 一致。

### 4.3 环境变量注入容器

`buildContainerEnv` 中 `run_e2e` case 将命名环境序列化为环境变量：

```
E2E_BASE_URL=https://staging.example.com
E2E_DB_HOST=staging-db.internal
E2E_DB_PORT=3306
E2E_DB_USER=e2e_test
E2E_DB_PASSWORD=xxx
E2E_DB_DATABASE=app_test
E2E_ACCOUNT_ADMIN_USERNAME=admin
E2E_ACCOUNT_ADMIN_PASSWORD=admin123
E2E_ACCOUNT_NORMAL_USERNAME=testuser01
E2E_ACCOUNT_NORMAL_PASSWORD=test123
```

entrypoint.sh 生成 `/tmp/.my.cnf` 后 `unset E2E_DB_PASSWORD`。账号环境变量保留（Playwright 脚本需通过 `process.env` 读取）。

### 4.4 CLI 非敏感字段覆盖

| 参数 | 说明 |
|------|------|
| `--env` | 选择命名环境 |
| `--base-url` | 覆盖 base_url（临时环境场景） |
| `--ref` | 指定代码基线 |

`db.*` / `accounts.*` 等凭证字段**不接受** CLI/API 覆盖，尝试传入返回 400。

### 4.5 环境解析优先级

`resolveEnvironment` 查找顺序：

1. `payload.Environment`（CLI `--env` 或 API `env` 字段）
2. 仓库级 `e2e.default_env`
3. 全局 `e2e.default_env`
4. 均为空 → `ErrEnvironmentNotFound`

### 4.6 配置校验

- `e2e.enabled=true` 时 `environments` 至少有一个环境
- `default_env` 必须指向已定义的环境名
- 每个环境 `base_url` 必填且为合法 URL
- `db.host` / `db.user` / `db.database` 必填（`db` 段整体可选——纯 API 测试不需要数据库）
- `accounts` 可选
- 仓库级 `default_env` 若非空必须指向已定义的环境名
- `artifact_retention_days` 范围 `[1, 90]`，默认 7

## 5. e2e.Service 骨架与任务路由

### 5.1 任务类型

```go
// internal/model/task.go
TaskTypeRunE2E = "run_e2e"

// internal/queue/client.go
AsynqTypeRunE2E = "task:run_e2e"
```

同步扩展：

- `TaskType.IsValid()` 需包含 `TaskTypeRunE2E`
- 任务优先级：`PriorityNormal`（E2E 通常是用户主动触发、在等结果）
- `selectGiteaToken`：M5.1 无需扩展（`run_e2e` 回退到默认 review token，不创建 PR/Issue）；M5.2 引入 `IssueCreator` 时需评估是否需要独立 token

### 5.2 窄接口

```go
// internal/e2e/service.go

// E2EPoolRunner 容器执行接口。
type E2EPoolRunner interface {
    RunWithCommandAndStdin(ctx context.Context, payload model.TaskPayload,
        cmd []string, stdinData []byte) (*worker.ExecutionResult, error)
}

// E2EConfigProvider 配置访问接口。
// 与 review/test 的 ConfigProvider 保持一致：按值返回、无 error。
// 环境查找逻辑（含 ErrEnvironmentNotFound）放在 Service.Execute 内部的
// resolveEnvironment 步骤中，不暴露到接口层。
type E2EConfigProvider interface {
    ResolveE2EConfig(repoFullName string) config.E2EOverride
    GetE2EEnvironments() map[string]config.E2EEnvironment
    GetClaudeModel() string
    GetClaudeEffort() string
}

// RepoClient 仓库信息查询。
type RepoClient interface {
    GetRepo(ctx context.Context, owner, repo string) (*gitea.Repository, *gitea.Response, error)
    GetBranch(ctx context.Context, owner, repo, branch string) (*gitea.Branch, *gitea.Response, error)
}

// IssueCreator M5.2 预留：失败时创建 Issue（M5.1 声明接口但不实装）。
type IssueCreator interface {
    CreateIssue(ctx context.Context, owner, repo string,
        opts gitea.CreateIssueOption) (*gitea.Issue, *gitea.Response, error)
}
```

### 5.3 Service 结构

```go
type Service struct {
    gitea        RepoClient
    pool         E2EPoolRunner
    cfgProv      E2EConfigProvider
    issueCreator IssueCreator      // M5.2 注入，M5.1 为 nil
    logger       *slog.Logger
}

type ServiceOption func(*Service)

func WithIssueCreator(c IssueCreator) ServiceOption { ... }
func WithServiceLogger(logger *slog.Logger) ServiceOption { ... }
```

### 5.4 Execute 六步流程

```go
func (s *Service) Execute(ctx context.Context, payload model.TaskPayload) (*E2EResult, error)
```

1. **前置校验**：`Enabled` 检查 → `resolveBaseRef` → `resolveEnvironment` → `--base-url` 覆盖
2. **用例发现**：由 prompt 指引 Claude 在容器内按目录约定发现（宿主侧不解析仓库结构）
3. **构建 Prompt + 容器执行**：`buildE2EPrompt` + `pool.RunWithCommandAndStdin`
4. **解析结果**：双层 JSON（CLI 信封 → E2EOutput）+ `validateE2EOutput`
5. **结果持久化**：M5.1 仅记录到 TaskRecord.result（现有机制）
6. **返回结果**：成功 / 部分失败 / 全部失败

### 5.5 Sentinel 错误

```go
var (
    ErrE2EDisabled         // e2e.enabled=false，SkipRetry
    ErrEnvironmentNotFound // 命名环境不存在，SkipRetry
    ErrInvalidRef          // 指定 ref 不存在，SkipRetry
    ErrNoCasesFound        // 指定范围内无 case.yaml，SkipRetry
    ErrE2EParseFailure     // E2EOutput 解析失败，允许 asynq 重试
)
```

### 5.6 Processor 集成

```go
// ProcessorOption 注入
func WithE2EService(svc E2EExecutor) ProcessorOption { ... }

// switch 路由
case model.TaskTypeRunE2E:
    if p.e2eService != nil {
        e2eResult, runErr = p.e2eService.Execute(ctx, payload)
    }
```

`adaptE2EResult` 适配器将 `E2EResult` 映射到通知 metadata。

### 5.7 装配层

`serve.go`：

```go
e2eSvc := e2e.NewService(giteaClient, pool, cfgAdapter,
    e2e.WithServiceLogger(logger))

processor := queue.NewProcessor(...,
    queue.WithE2EService(e2eSvc),
)
```

E2E 不创建 PR/Issue（M5.1），复用 `ServiceDeps.GiteaClient`（review token）。

## 6. Prompt 设计与输出 Schema

### 6.1 Prompt 结构（四段式）

通过 stdin 传入 Claude。

**第一段：上下文**

```
你正在为仓库 {repo_full_name} 执行 E2E 测试。

测试环境：
- 应用地址：{base_url}
- 数据库：通过 mysql --defaults-extra-file=/tmp/.my.cnf 访问
- 测试账号：
  - admin: 用户名 {admin_username}（密码通过环境变量 E2E_ACCOUNT_ADMIN_PASSWORD 读取）
  - normal: 用户名 {normal_username}（密码通过环境变量 E2E_ACCOUNT_NORMAL_PASSWORD 读取）

代码基线：{base_ref}
测试范围：{scope_description}
```

**第二段：执行指令**

```
执行步骤：

1. 进入 e2e/ 目录，根据测试范围扫描 case.yaml 文件
   - 模块指定时：扫描 e2e/{module}/cases/*/case.yaml
   - 用例指定时：读取 e2e/{module}/cases/{case}/case.yaml
   - 未指定时：扫描 e2e/*/cases/*/case.yaml

2. 对每个用例，按以下阶段顺序执行：

   a) Setup（数据准备）：
      - .sql 文件：mysql --defaults-extra-file=/tmp/.my.cnf < {script}
      - .js 文件：node {script}
      - .ts 文件（非 .spec.ts）：npx tsx {script}
      - 任一 setup 脚本失败 → 跳过该用例的 test 阶段，标记为 error，仍执行 teardown

   b) Test（测试执行）：
      - .spec.ts 文件：npx playwright test {script} --reporter=json
        --output=/workspace/artifacts/test-results/{case_name}/
      - 记录每个测试脚本的通过/失败/错误状态

   c) Teardown（清理）：
      - 执行方式同 setup
      - 无论 test 结果如何都必须执行
      - teardown 失败记录警告但不影响用例结果判定

3. 执行完所有用例后，汇总结果

重要约束：
- 每个用例独立执行，用例之间不共享状态
- setup 失败的用例标记为 error（非 fail），区别于测试断言失败
- 严格按 case.yaml 中声明的脚本顺序执行
- 不要修改任何仓库文件
- 不要访问测试环境以外的网络地址
```

**第三段：失败分析指令**

```
当测试失败时，分析失败原因并分类：

- "bug"：应用行为与 case.yaml expectations 不符，属于真实缺陷
- "script_outdated"：页面元素或流程已变更导致脚本无法正常执行
- "environment"：环境问题（服务不可达、数据库连接失败、超时等）

分析时参考 case.yaml 中的 expectations 段落理解测试意图。

对于每个失败的用例，记录：
- 失败阶段（setup / test / teardown）
- 错误信息
- 截图文件路径（如有）
- 你的失败分类和简要分析
```

**第四段：输出格式**

```
以 JSON 格式输出结果，schema 如下：
{E2EOutput JSON schema}

确保 JSON 合法且完整。不要在 JSON 前后输出其他内容。
```

### 6.2 E2EOutput JSON Schema

```go
// E2EOutput Claude 容器内 E2E 执行的结构化输出。
type E2EOutput struct {
    Success      bool         `json:"success"`
    TotalCases   int          `json:"total_cases"`
    PassedCases  int          `json:"passed_cases"`
    FailedCases  int          `json:"failed_cases"`
    ErrorCases   int          `json:"error_cases"`
    SkippedCases int          `json:"skipped_cases"`
    Cases        []CaseResult `json:"cases"`
    Warnings     []string     `json:"warnings,omitempty"`
}

// CaseResult 单个用例的执行结果。
type CaseResult struct {
    Name            string       `json:"name"`
    Module          string       `json:"module"`
    CasePath        string       `json:"case_path"`
    Status          string       `json:"status"`          // "passed"/"failed"/"error"/"skipped"
    DurationMs      int64        `json:"duration_ms"`
    SetupResult     *PhaseResult `json:"setup_result,omitempty"`
    TestResult      *PhaseResult `json:"test_result,omitempty"`
    TeardownResult  *PhaseResult `json:"teardown_result,omitempty"`
    FailureCategory string       `json:"failure_category,omitempty"` // "bug"/"script_outdated"/"environment"
    FailureAnalysis string       `json:"failure_analysis,omitempty"`
    Screenshots     []string     `json:"screenshots,omitempty"`
}

// PhaseResult 单阶段执行结果。
type PhaseResult struct {
    Status     string         `json:"status"`      // "passed"/"failed"/"skipped"
    DurationMs int64          `json:"duration_ms"`
    Scripts    []ScriptResult `json:"scripts"`
}

// ScriptResult 单个脚本执行结果。
type ScriptResult struct {
    Name     string `json:"name"`
    Status   string `json:"status"`             // "passed"/"failed"/"error"
    ExitCode int    `json:"exit_code"`
    Output   string `json:"output,omitempty"`   // stdout+stderr 截断（最后 2KB）
    ErrorMsg string `json:"error_msg,omitempty"`
}
```

### 6.3 校验规则

`validateE2EOutput` 不变量校验：

- `Success=true` 时 `FailedCases` 和 `ErrorCases` 必须为 0
- `TotalCases == PassedCases + FailedCases + ErrorCases + SkippedCases`
- `CaseResult.Status` 必须为四枚举之一
- `FailureCategory` 非空时必须为三枚举之一
- `CaseResult.Status="failed"` 时 `FailureCategory` 必须非空
- `ScriptResult.Output` 字段 2KB 截断

校验失败 → `ErrE2EParseFailure` → asynq 重试 → 重试耗尽后降级回写。

### 6.4 安全加固

- Prompt 通过 stdin 传入
- `--disallowedTools`：禁用文件编辑工具
- Prompt 首部 READ-ONLY 约束
- `sanitizeE2EOutput`：过滤自由文本字段防 prompt-injection
- `SanitizeErrorMessage`：ParseError 详情不进返回 error

### 6.5 解析与降级链

双层 JSON 解析（与 Phase 2-4 一致）：

1. 正常解析 → 结构化结果
2. E2EOutput 解析失败 → `ErrE2EParseFailure` → asynq 重试
3. 重试耗尽 → `WriteDegraded`（M5.1 仅记录日志）
