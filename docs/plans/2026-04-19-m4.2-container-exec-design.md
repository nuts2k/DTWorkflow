# M4.2 容器执行 + 测试验证 + PR 创建 + 通知 — 设计文档

> 日期：2026-04-19（v3）
> 状态：设计已完成二轮审核修订，所有阻断级矛盾已解除，等待实施
> 依赖：M4.1（`internal/test` 包骨架 + Processor 已装配 + entrypoint.sh `gen_tests` 分支已加固）
> 范围：Phase 4 第一轮第二个里程碑——把 M4.1 的"路由已通 + PR 占位"补齐为「容器真跑 + 进度可恢复 + PR 真建 + 通知齐全 + 结果持久化」

## 1. 概述

### 1.1 目标

承接 M4.1 已落地的 `test.Service` 骨架与 Processor 路由，本里程碑解决 **运行时能力 + 长任务韧性 + 与既有 review 流水线的协作**：

1. **容器内真实执行 `mvn test` / `vitest run`** —— prompt 已写好，本期保证它真的跑、且区分基础设施故障与测试质量问题
2. **`createTestPR` 实装** —— 复用 M3.5 fix.createFixPR 的幂等模板，签名按 M4.1 冻结
3. **「分批 push + 断点续传」** —— 长任务中途失败不丢已完成工作（用户最关心的痛点）
4. **review enqueue 拦截** —— 防止分批 push 引发 N 次 review 任务被反复 Cancel-and-Replace
5. **稳定分支命名 + Cancel-and-Replace 主动清理** —— 同 module 永远只有一个 auto-test 分支与一个 PR
6. **通知三件套**（开始 / 完成 / 失败）—— 对齐 review / fix
7. **`test_gen_results` 表** —— 结果持久化，便于审计与 M4.3 变更驱动复用
8. **完成后主动 enqueue review** —— 任务结束时由 test.Service 显式触发 1 次 review

### 1.2 非目标（延后到 M4.3 / 后续）

- PR-merged Webhook 触发变更驱动测试生成（M4.3）
- 持久卷跨任务复用 Maven/Gradle 缓存（待评估）
- 多模块并行 gen_tests 调度优化（用户当前可手动并发）
- 远程孤儿分支定时清理工具（本设计已确保不产生孤儿，无需）

### 1.3 与 M4.1 的边界（增量视图）

| 能力 | M4.1 | M4.2 |
|------|------|------|

## 2. 关键设计决策

> 本节决策来自 2026-04-19 brainstorming，按 D1..D8 编号。每条决策列「选定方案 + 否决方案 + 理由」。

### D1：长任务失败处理 — 「分批 push + 断点续传」

**选定**：每个测试文件 commit 后立即 push 到稳定分支 `auto-test/{module}`；失败时已 push 的工作保留在远程，重试时从中断点继续。

**否决**：
- A. 「全或无 + 最后一刻 push」（M4.1 默认） — 失败成本太高，大仓 30 分钟工作可能全废
- C. 服务端分片调度（拆多个子任务） — 架构改动太大，多 PR 用户体验差
- D. 时间窗 push（每 5 分钟攒一次） — 实现复杂，prompt 难表达

**理由**：直接解决用户提出的"长任务中途失败"痛点；实现最简单（prompt 改 1 句话）；失败爆炸半径最小。

---

### D2：Review 拦截 — 方案 N（分支前缀 + 服务端任务状态查询）

**选定**：review enqueue 在收到 PR webhook 时，若 head 匹配 `auto-test/*`，查 SQLite 是否有该 repo 的活跃 gen_tests 任务（status IN queued/running），有则跳过入队。

**否决**：
- α. 不拦截（现状） — 首次任务因 D4 延迟建 PR 天然无 review 入队；**但**断点续传（asynq retry）与失败保留 PR（D5）后的续跑会在 PR 已存在的前提下继续 push，触发 `pull_request.synchronize` → review 反复入队 + Cancel-and-Replace，浪费 Claude 预算与飞书噪音
- γ. PR draft 状态拦截 —— **Gitea 不支持真正的 draft PR**（仅靠标题前缀 `WIP:`，依赖实例 `app.ini` 配置 `WORK_IN_PROGRESS_PREFIXES`，且去掉前缀不会发 `ready_for_review` webhook，不可靠）
- δ. 拦截一切 auto-test/* PR —— auto-fix/* 也算"机器人产出"但用户期望被审，区分对待

**理由**：
- 不依赖 Gitea 任何 draft 概念，跨 Gitea 实例完全可移植
- 状态由 DTWorkflow 自己掌控，无需 PR 编辑动作
- 与 auto-fix（M3.5 现状）零干扰
- 实现成本低：~20 行白名单逻辑 + 1 个 store 查询方法

**关键 finding**（已查证）：
- Gitea `CreatePullRequestOption` **无 `draft` 字段**
- Gitea `PullRequest` 响应**无 `draft` 布尔字段**
- Gitea webhook **无 `ready_for_review` 动作**
- 唯一接近的机制是 `WORK_IN_PROGRESS_PREFIXES`（实例级配置，默认 `WIP:,[WIP]`），但去掉前缀仅触发 `pull_request.edited`

---

### D3：稳定分支命名 — `auto-test/{module}`（无 timestamp）

**选定**：分支命名固定为 `auto-test/{sanitize(module)}`（整仓时 module 回落 `all`，与 M4.1 现有 `branchSuffix` 口径一致，避免历史分支命名漂移），同 module 永远只有一个分支。

**否决**：
- M4.1 的 `auto-test/{module}-{timestamp}` —— 每次任务新分支，失败留孤儿，重试无法续传
- `auto-test/{module}-{task_id_short}` —— 同样产生孤儿

**理由**：
- 重试时容器内 `git fetch origin auto-test/{module}` → checkout → Claude 看到已有测试加入 ExistingTests → 跳过已生成的，只补缺失的 → **真正的断点续传**
- PR 同 head 分支幂等复用 → 远程**永远不堆积孤儿**
- Cancel-and-Replace 时由 test.Service 主动删除分支删除 PR（用户语义就是"重做"）

**`module` 路径合法化**：M4.1 已有的 `internal/test/prompt.go: sanitizeBranchRef` 把 `/` 等非法字符统一替换为 `-`，已满足 git ref 命名要求——本期**直接复用**，不另起名。

**统一口径**：M4.1 已在 `internal/test/prompt.go` 提供 `branchSuffix(module, ts)`（空 module → `"all"`）+ `sanitizeBranchRef`。M4.2 新增 `test.BuildAutoTestBranchName(module string) string`，返回 `"auto-test/" + moduleKey(module)`，其中 `moduleKey = sanitizeBranchRef(module)`，空或清洗后为空则回落 `"all"`——这是 `queue`（cleanup）、`worker`（env 注入）、`prompt`（header 文案）、`entrypoint.sh` 四处共用的唯一入口，避免定义漂移。

Worker 侧（`internal/worker/container.go`）在注入 `MODULE_SANITIZED` 环境变量前调用 `test.BuildAutoTestBranchName(module)` 并取"/"后半部分（即 moduleKey 部分）；或直接暴露 `test.ModuleKey(module) string` 返回清洗后 key。容器内 entrypoint.sh 直接读 `MODULE_SANITIZED`，不再自行 sanitize。

### D4：PR 创建时机 — 第一次 push 后立即创建（普通 ready 状态）

**选定**：容器内 Claude 完成第一个测试文件的 commit + push 后，由后续 push 触发 webhook → enqueue 已被 D2 拦截 → 不入队 review；test.Service 在 `Execute` 末尾调 `createTestPR` 检查并创建/复用 PR。

**否决**：
- 延迟到任务结束才创建 — 失败时无 PR，用户失去观察"Claude 跑了多少"的能力
- 创建时打 WIP 前缀 — Gitea draft 不可用，且污染 PR 标题

**理由**：用户能实时通过 PR 看到分批 push 的进度；失败时 PR 保留，便于接管；不依赖 Gitea draft。

**实施细节**：
- `createTestPR` 在 `Execute` 末尾调用（即使 `Success=false` 但 `len(CommittedFiles)>0` 也调，保证半成品也有 PR）
- 幂等：调 `findExistingTestPR`（仿 fix.findExistingFixPR），按 head 分支查 open PR；存在则复用 number/url

---

### D5：失败时 PR 处理 — 保留 + 默认不审

**选定**：任务 `failed` / `cancelled` 时 PR 保留，且**不主动 enqueue review**。

**否决**：
- 失败时删分支删 PR — 用户失去现场，无法接管
- 失败时也自动审 — 浪费预算，Claude 评审不完整工作意义有限

**理由**：用户失败现场可见、可接管；审与不审是用户决策。

**配置开关**：`test_gen.review_on_failure: false`（默认 false），置 true 则 `failed` 时也 enqueue review。

**配置 struct 落点**（与 M4.1 `config.TestGenConfig` 对齐）：
```go
type TestGenConfig struct {
    // ... M4.1 已有字段 ...
    ReviewOnFailure bool `mapstructure:"review_on_failure" yaml:"review_on_failure"`
}
```
- `config.WithDefaults`：`ReviewOnFailure=false`
- `config.validate`：无需特殊校验（bool 合法性自保）
- **per-repo override**：M4.1 的 `test_gen` 已支持 `repos.<name>.test_gen` 覆盖，本字段天然参与该机制，无需额外改 merge 逻辑
- `configs/dtworkflow.example.yaml`：在 `test_gen:` 段追加注释示例

---

### D6：完成后主动 enqueue review — `EnqueueManualReview`

**选定**：`test.Service.Execute` 在 `Success=true && PRNumber>0` 时调 `EnqueueHandler.EnqueueManualReview(repo, pr_number, triggered_by="gen_tests:<task_id>")`。

**否决**：
- 不主动入队 — 由用户手动触发，体验差
- 通过 Gitea API 触发空 commit / 重新 open PR 让 webhook 自然触发 — 绕路且不可靠

**理由**：
- review 触发时机精准在 1 次：任务真正结束后
- 触发源标记 `triggered_by="gen_tests:<task_id>"` → 便于审计与日志关联
- review 自身已有 Cancel-and-Replace 机制 → 即使有边缘并发也安全

---

### D7：基础设施故障 vs 测试质量问题区分

**选定**：在 `TestGenOutput` 增加 `failure_category` 字段，Claude 在 prompt 指令下分类填写，service 层根据该字段决定通知严重级与是否计入"测试质量问题"。

**`failure_category` 枚举**：
- `infrastructure` — 依赖下载失败、Maven/Gradle 工具异常、网络问题、容器资源不足
- `test_quality` — 编译失败 / 断言失败 / mock 配置错误（已耗尽 max_retry_rounds）
- `info_insufficient` — 信息不足，需用户补充
- `none` — 成功路径

**通知差异化**：
- `infrastructure` → severity=`Warning`，通知文案"基础设施故障，建议重试或检查环境"
- `test_quality` → severity=`Info`，通知文案"测试质量未达标，已生成的 N 个测试可参考"
- `info_insufficient` → severity=`Info`，通知文案附 `MissingInfo`

**否决**：
- 不区分，统一报"失败" — 运维无法定位问题源
- 在服务端用启发式（看 RawOutput 关键词）推断 — 不可靠，Claude 自报最准确

**理由**：基础设施问题应触发 oncall 重试，测试质量问题不应；分类由 Claude 在容器内现场判断最准。

---

### D8：通知策略 — 三事件 + 飞书 + Gitea PR 评论

**选定**：新增 3 个事件类型，`buildStartMessage` / `buildNotificationMessage` 增加 `gen_tests` case：

```go
EventGenTestsStarted EventType = "gen_tests.started"
EventGenTestsDone    EventType = "gen_tests.done"
EventGenTestsFailed  EventType = "gen_tests.failed"
```

**触发时机**：
- `Started` → 任务从 queued 进入 running 时（Processor 调 `sendStartNotification` 时）
- `Done` → `Success=true` 且 PR 创建成功
- `Failed` → 各类 sentinel 失败 / 解析失败 / 容器异常退出

**通知载体**：
- **飞书**（与 review/fix 一致）：开始 + 结束都发，metadata 含 `repo / module / pr_url / generated_count / failure_category`
- **Gitea PR 评论**：仅 `Done` 时在 PR 上评论一次，附 GapAnalysis 摘要 + 测试运行结果（用户在 PR 上能看到）

**否决**：
- 中间发进度通知（"已生成 5/30"） — 噪音，用户看 PR commits 即可
- 仅发飞书不发 PR 评论 — PR 评论对评审者更直接

**理由**：对齐 review/fix 模式；PR 评论补足上下文。

## 4. 组件细节

### 4.1 `TestGenOutput` schema 增量（result.go）

```go
type TestGenOutput struct {
    // ... M4.1 已有字段 ...

    // M4.2 新增：失败分类，由 Claude 在 prompt 指令下分类填写
    FailureCategory FailureCategory `json:"failure_category"` // 枚举见下
}

type FailureCategory string

const (
    FailureCategoryNone           FailureCategory = "none"
    FailureCategoryInfrastructure FailureCategory = "infrastructure"
    FailureCategoryTestQuality    FailureCategory = "test_quality"
    FailureCategoryInfoInsufficient FailureCategory = "info_insufficient"
)
```

**不变量补充**：
- `Success=true ⇒ FailureCategory == "none"`
- `Success=false ⇒ FailureCategory ∈ {infrastructure, test_quality, info_insufficient}`
- `InfoSufficient=false ⇒ FailureCategory == "info_insufficient"`

### 4.2 `Execute` 流程重构（service.go）

**为何要重构**：M4.1 现有 `Execute` 在 `!out.InfoSufficient` 或 `!out.Success` 时**直接 return error**（`service.go:231-236`），根本走不到 `createTestPR`。但 M4.2 §5.4 / D5 明确要求「`Success=false && len(CommittedFiles)>0` 仍然创建 PR，供用户接管半成品」——两条相互矛盾。M4.2 必须把失败分支与 PR 创建/持久化/review 入队的路径解耦。

**重构后的 Execute 流程**（取代 M4.1 `service.go:136-246` 的现状）：

```
1. 前置校验：Enabled / validateModule / resolveBaseRef / validateModuleExists
   → 任一失败 → 立即 return error（此阶段没有任何 CommittedFiles，无需建 PR）

2. resolveFramework / buildPrompt / 容器执行
   → execErr / ExitCode!=0 / result==nil → return error（容器侧未产出）

3. parseResult（外层 CLI 信封 + 内层 TestGenOutput）
   → ParseError → return ErrTestGenParseFailure（无 branch/commit，无 PR 可建）

4. 不变量校验 validateSuccessfulTestGenOutput
   → 仅对 Success=true 分支强校验；Success=false 分支放行
   → Success=true 但不变量违反 → return ErrTestGenInvariantViolation

5. 【M4.2 新增】若 len(CommittedFiles) > 0，即使 Success=false 也继续：
   5.1 createTestPR（幂等；CommittedFiles==0 返回 0/""，不建 PR）
   5.2 persistTestGenResult（无论成功失败，只要有 result 就存库；失败仅 warn）

6. 业务失败判定：
   - !InfoSufficient → return ErrInfoInsufficient（此时 result.PRNumber 可能 >0）
   - !Success        → return ErrTestGenFailed（同上）
   - Success=true 但 review_on_failure=true 路径不走此处（见 7）

7. 完成后入队 review（仅 Success=true && PRNumber>0 进入；配置 review_on_failure=true 时，
   Success=false && PRNumber>0 也进入——与 D5 配置项对齐）：
   reviewEnqueuer.EnqueueManualReview(reviewPayload, "gen_tests:"+taskID)
   → 失败仅 warn 不阻断

8. return result, nil
```

**关键语义**：
- Processor 通过 `result.PRNumber` / `result.PRURL` 构造通知文案，即使返回 sentinel error 也能拿到 PR 链接
- 重构使 createTestPR / persistTestGenResult 成为无差别"产出落地"步骤，和"整体任务是否 Success"正交
- `validateSuccessfulTestGenOutput` 改名语义更准——**仅对 Success=true 分支生效**；Success=false 分支只做基础字段校验（BranchName 非空、GeneratedFiles 与 CommittedFiles 结构合法），不强制 CommitSHA / TestResults 非空

**测试要求**：
- `Success=true` happy path（走完 8 步）
- `Success=false && CommittedFiles>0` → PR 建 / 存库 / review 不入队 / 返回 `ErrTestGenFailed`（且 result.PRNumber>0）
- `Success=false && CommittedFiles=0` → PR 不建 / 存库 / 返回 `ErrTestGenFailed`（且 result.PRNumber=0）
- `InfoSufficient=false && CommittedFiles=0` → PR 不建 / 存库 / 返回 `ErrInfoInsufficient`
- ParseError 路径：不 persist（无 result.Output）、不建 PR、返回 `ErrTestGenParseFailure`
- `review_on_failure=true` 下 `Success=false && PRNumber>0` → EnqueueManualReview 被调用

### 4.3 `createTestPR` 实装（service.go）

```go
// 幂等：先 ListRepoPullRequests 查询同 head 分支的 open PR，存在则复用。
// 即使 Success=false 但 len(CommittedFiles) > 0 也创建 PR（保留半成品供用户接管）。
func (s *Service) createTestPR(ctx context.Context, payload model.TaskPayload,
    result *TestGenResult) (prNumber int64, prURL string, err error) {

    if s.prClient == nil {
        return 0, "", fmt.Errorf("PRClient 未注入")
    }

        return 0, "", nil  // 容器未产出分支，无 PR 可建

    if num, url, ok := s.findExistingTestPR(ctx, payload.RepoOwner, payload.RepoName,
        result.Output.BranchName); ok {
        return num, url, nil
    }

    if moduleLabel == "" {
        moduleLabel = "整仓"
    }
    title := fmt.Sprintf("test: 补充 %s 测试用例", moduleLabel)
    if len(title) > 250 { title = title[:250] }
```

### 4.5 review enqueue 拦截（enqueue.go）

**关键修正**：拦截按 `repo + module` 双字段匹配，避免 module A 运行中误拦 module B 的 PR review。

从 `HeadRef = "auto-test/<moduleKey>"` 解析出 `moduleKey`，查库时匹配"`module` 经过 `test.ModuleKey` 清洗后 = `moduleKey`"的活跃任务。由于 `tasks.module` 存原始 module 字符串（未清洗），查询需在 SQL 侧做清洗或在应用层遍历；推荐在应用层加薄封装：先用 `SELECT id, module FROM tasks WHERE task_type='gen_tests' AND repo_full_name=? AND status IN (...)` 捞候选，再用 Go `test.ModuleKey(row.Module) == moduleKey` 判定。由于同 repo 活跃 gen_tests 数量天然很小（最多几个 module），该遍历开销可忽略。

`store.HasActiveGenTestsForModule` 实现（纳入 `retrying` 以覆盖 asynq backoff 窗口——此时 tasks 表 status 已改写为 retrying，容器尚未重新 checkout 分支，仍应视作"活跃"）：

```go
func (s *SQLiteStore) HasActiveGenTestsForModule(ctx context.Context, repo, moduleKey string) (bool, error) {
    rows, err := s.db.QueryContext(ctx, `
        SELECT module FROM tasks
        WHERE task_type = 'gen_tests'
          AND repo_full_name = ?
          AND status IN ('queued', 'running', 'retrying')
    `, repo)
    if err != nil { return false, err }
    defer rows.Close()
    for rows.Next() {
        var m string
        if err := rows.Scan(&m); err != nil { return false, err }
        if test.ModuleKey(m) == moduleKey {
            return true, nil
        }
    }
    return false, rows.Err()
}
```

### 4.6 Cancel-and-Replace 主动清理（enqueue.go）

`EnqueueManualGenTests` 在 `cancelTasks(activeTasks)` 之后追加：

```go
// M4.2：稳定分支策略下，用户手动重触发意味着"重做"，需主动清理远程旧分支与旧 PR
if len(activeTasks) > 0 && h.branchCleaner != nil {
    // 分支名走 internal/test 单一入口，避免与 prompt/worker/entrypoint 漂移
    branchName := test.BuildAutoTestBranchName(payload.Module)  // "auto-test/<moduleKey>"
    if err := h.branchCleaner.CleanupAutoTestBranch(ctx, payload.RepoOwner, payload.RepoName,
        branchName); err != nil {
        // 清理失败不阻断新任务（容器内 fetch 时若分支异常会自行重置）
        h.logger.WarnContext(ctx, "清理旧 auto-test 分支失败，继续入队",
            "branch", branchName, "error", err)
    }
}
```

**竞态与兜底（cancel → kill 容器 → cleanup → 旧容器仍在 push）**：
- `cancelTasks` 调用 asynq cancel 后，旧容器被 SIGKILL，但 kill 是异步的（容器可能仍在执行 `git push` 的 TCP 写入）
- 极端情况：cleanup 成功删除分支后，旧容器的 push 抵达 Gitea → Gitea 自动重建 auto-test/{module}（push 到不存在的分支 = 新建）→ 新任务 fetch 看到"非空"分支
- **三层兜底**：
  1. `branchCleaner` 顺序：**先关 PR 再删分支**（关 PR 能中断 Gitea webhook，削弱旧 PR 的语义存在）
  2. `entrypoint.sh` 在 fetch 后做「落后 base 重置」+「非 bot author 重置」双重守护——即使分支被旧容器重建，只要其内容与当前基线偏离或含有污染，即会被重置
  3. 新任务的容器独占 `auto-test/{module}` 的写入权——重建的"孤儿 commit"最终被新容器覆盖（force push 由 entrypoint 的 `git checkout -B ... origin/BASE_REF` 语义隐式达成）
- **接受的剩余风险**：极小概率下 cleanup 与旧容器 push 的时间窗重合，但不会造成用户可见的错误——最多是新任务容器内日志多一条"重置 auto-test/{module}"

`BranchCleaner` 接口（**定义与实现均在 `internal/queue`**，便于 EnqueueHandler 直接 new 并调用 gitea client；不放 `internal/test` 是为了避免增加 `queue → test` 之外的耦合，且清理动作在"入队"阶段触发，语义归属 queue）：

```go
// 定义在 internal/queue/branch_cleaner.go
type BranchCleaner interface {
    // CleanupAutoTestBranch 关闭分支上的 open PR + 删除分支。
    // 任一步失败仅记日志，不阻断（容器内 entrypoint 兜底重置）。
    CleanupAutoTestBranch(ctx context.Context, owner, repo, branch string) error
}
```

实现步骤：
1. `gitea.ListRepoPullRequests(state=open)` → 找到 head=branch 的 PR → `gitea.ClosePullRequest`
2. `gitea.DeleteBranch(owner, repo, branch)`

`EnqueueHandler` 通过构造函数或 Option 注入 `BranchCleaner`，wiring 在 cmd 层完成。

### 4.7 entrypoint.sh `gen_tests` 分支增量

M4.1 已有 5 项配置（credential / identity / cache / pre-push hook）。M4.2 追加：

```bash
gen_tests)
    # ... M4.1 已有的 5 项配置 ...

    # M4.2 新增：稳定分支断点续传
    AUTO_TEST_BRANCH="auto-test/${MODULE_SANITIZED:-all}"
    BOT_EMAIL="dtworkflow-bot@noreply.local"
    log "尝试 fetch 已存在的 ${AUTO_TEST_BRANCH} 分支以续传"
    if git fetch origin "${AUTO_TEST_BRANCH}" 2>/dev/null; then
        BASE_SHA=$(git rev-parse "origin/${BASE_REF}")
        BRANCH_SHA=$(git rev-parse "origin/${AUTO_TEST_BRANCH}")
        RESET_REASON=""
        # 1. 孤儿检测：分支不含 base SHA（被 squash/rebase merge）
        if ! git merge-base --is-ancestor "${BASE_SHA}" "${BRANCH_SHA}" 2>/dev/null; then
            RESET_REASON="落后 base"
        else
            # 2. 作者污染检测：分支上仅允许 bot commit（防止人工直推）
            NON_BOT=$(git log "${BASE_SHA}..${BRANCH_SHA}" --pretty=format:'%ae' 2>/dev/null \
                | grep -v "^${BOT_EMAIL}$" | head -n 1)
            if [ -n "${NON_BOT}" ]; then
                RESET_REASON="检测到非 bot 提交者 ${NON_BOT}"
            fi
        fi
        if [ -z "${RESET_REASON}" ]; then
            log "复用现有 ${AUTO_TEST_BRANCH}（断点续传）"
            git checkout -B "${AUTO_TEST_BRANCH}" "origin/${AUTO_TEST_BRANCH}"
        else
            log "${AUTO_TEST_BRANCH} 重置回 ${BASE_REF}（原因：${RESET_REASON}）"
            git checkout -B "${AUTO_TEST_BRANCH}" "${BASE_SHA}"
        fi
    else
        log "未发现已有 ${AUTO_TEST_BRANCH}，从 ${BASE_REF} 创建"
        git checkout -B "${AUTO_TEST_BRANCH}" "origin/${BASE_REF}"
    fi
    ;;
```

`MODULE_SANITIZED` 由 `worker/container.go` 在 env 注入前调用 **`test.ModuleKey(module)`**（复用 M4.1 `internal/test/prompt.go` 现有 `sanitizeBranchRef` + 空值回落 `"all"` 的口径，在 `internal/test` 包导出为单一入口）计算。容器内 entrypoint.sh 直接读取 env，不自行处理。

```sql
CREATE INDEX IF NOT EXISTS idx_test_gen_results_created ON test_gen_results(created_at);
```

**主键与幂等策略**：
- `id`：独立 UUID，由 `store.SaveTestGenResult` 内部生成（若 record.ID 为空）
- `task_id`：UNIQUE；同一 task 在 asynq retry 周期内每次 Execute 完成都会 UPSERT 同一行（`INSERT ... ON CONFLICT(task_id) DO UPDATE SET ... , updated_at = datetime('now')`），结果始终为最新一次 Execute 的产出
- 重试导致的多次 Execute 在 `tasks` 表里是同一 `task_id`；这里借 UNIQUE 约束自然去重——不需要判断"是否已存在"的应用层逻辑

**与 `tasks` 表字段的分工**：
- `triggered_by = "gen_tests:<taskID>"` 字符串由 `ReviewEnqueuer` 写入 `tasks.triggered_by`（新 review 任务的一列），**`test_gen_results` 不重复存储**；审计时通过 `test_gen_results.task_id → tasks.id` 反查出发源任务，再通过 `tasks.triggered_by` 找到上游 gen_tests 任务
- `review_enqueued` 仅表征 EnqueueManualReview 是否成功落队，与 review 任务自身状态无关（review 后续成功/失败看 tasks 表）

`store.SaveTestGenResult(ctx, *TestGenResultRecord) error` —— 由 `test.Service` 在 Execute 中间阶段调用（见 §4.2 流程第 5.2 步；无论成功失败，只要有 result.Output 就持久化）。

## 5. 数据流

### 5.1 长任务 happy path（断点续传 + 拦截）

```
T0  dtw gen-tests --repo o/r --module srv
    └─ EnqueueManualGenTests → task1 status=queued

T0+5s  asynq pull task1 → Processor → status=running
       ├─ sendStartNotification → 飞书"测试生成开始"
       └─ testService.Execute
            ├─ entrypoint.sh: fetch auto-test/srv → 不存在 → 从 main 创建
            └─ Claude 开始

T0+30s  Claude 生成第 1 个测试文件 → mvn test 通过 → commit + push
        ※ 此时 PR 还未创建（D4：Execute 末尾才调 createTestPR）
        ※ Gitea 对"尚未关联 PR 的分支 push"只发 push event，DTWorkflow 的
          HandlePullRequest 仅订阅 pull_request 事件 → 天然不会触发 review 入队
        ※ 用户能从 Gitea UI 看到 auto-test/srv 分支已有 commit（分支视图可见）

T1+25min  Claude 完成全部 50 个目标 → 最后 push → 输出 TestGenOutput JSON
          └─ 容器退出，pool.Run 返回

T1+25min  parseResult → validateSuccessfulTestGenOutput → 通过
          ├─ createTestPR
          │  ├─ findExistingTestPR(head=auto-test/srv) → 未找到
          │  └─ CreatePullRequest → PR #42 创建
          ├─ persistTestGenResult → test_gen_results 表 insert
          └─ EnqueueManualReview(o/r, #42, "gen_tests:<delivery_id>")
              └─ task2 (review_pr) created status=queued

T1+25min+1s  Processor → task1 status=succeeded
             └─ sendCompletionNotification → 飞书 "Done" + PR #42 评论 GapAnalysis

T1+25min+2s  asynq pull task2（review_pr）→ 正常审 PR #42
```

**关键性质**：
- 首次任务期间 PR 尚未创建，Gitea 不会对"无 PR 分支 push"发 `pull_request.synchronize` → 天然 0 次 review 入队，D2 拦截**本次不起作用**
- 任务结束后**精准触发** 1 次 review（由 test.Service 主动 `EnqueueManualReview`），时机可控
- 失败时（容器超时 / Claude error）：已 push 的工作保留在远程 → 重试时容器内 fetch 复用 → Claude 跳过已生成的 → 真断点续传
- **D2 拦截真正生效的场景**（在 §5.2 / §5.4 展开）：
  ① asynq 断点续传重跑——前一轮末尾已创建 PR，重跑期间继续 push 会触发 `synchronize`，此时必须拦截；
  ② 失败保留 PR（D5）后用户再次 dtw gen-tests——新任务会先 cleanup 再新建分支，短暂窗口若恰逢旧 PR 上有外部 push 事件进来，也靠拦截兜底

### 5.2 容器超时失败 → asynq 重试（断点续传）

```
T0      task1 启动，Claude 生成 30 / 50 个文件，已 push 到 auto-test/srv
T0+30m  worker.timeouts.gen_tests 触发，容器被 SIGKILL
T0+30m  testService.Execute 返回 ctx.Err()
        └─ Processor → shouldRetry=true → task1 status=retrying
            （tasks 表同一行 status 字段被就地更新，task_id 不变；
             asynq 自身的 retry 与 DTWorkflow tasks 行一一对应）
        └─ sendCompletionNotification → 飞书 "Retrying"
        ※ 此期间 ListActiveGenTestsModules 返回列表含本 module，status IN (queued/running/retrying)

T1      asynq 调度重试 task1（同 asynq_id 同 task_id）→ Processor → status=running
        ├─ entrypoint.sh: fetch auto-test/srv → 已存在 → checkout 复用
        ├─ git merge-base --is-ancestor: 仍可达 main → 不重置
        └─ Claude 启动
            ├─ 扫现有测试（含分支上已有的 30 个） → ExistingTests=30
            ├─ GapAnalysis 排除已生成 → target_queue=20（剩余 20 个）
            ├─ FOR 20 个 target: 生成 + 验证 + commit + push
            └─ 输出 TestGenOutput

T1+15m  Success → createTestPR → 已有 open PR（findExistingTestPR 命中）→ 复用
        └─ EnqueueManualReview → task3
        └─ status=succeeded
```

**收益**：第一次 30 分钟 + 重试 15 分钟 = **节省了第一次的 30 分钟**重做成本。

### 5.3 Cancel-and-Replace（用户主动重触发）

```
T0   task1 跑了 5 分钟，已 push 10 个文件
T0   dtw gen-tests --repo o/r --module srv （第二次手动触发）
     ├─ EnqueueManualGenTests
     │   ├─ listActiveGenTestsTasks → [task1]
     │   ├─ enqueueTask(task2) → status=queued
     │   ├─ cancelTasks([task1]) → asynq cancel → 容器被 kill
     │   │  └─ task1 status=cancelled
     │   └─ branchCleaner.CleanupAutoTestBranch(o, r, "auto-test/srv")  ← D3 + D5
     │      ├─ findOpenPRByHead(auto-test/srv) → PR 不存在（task1 还没到 createTestPR）
     │      ├─ DeleteBranch(o, r, auto-test/srv) → 远程分支被删
     │      └─ 完成
     └─ 返回 task2 ID

T1   asynq pull task2 → entrypoint fetch auto-test/srv → 不存在 → 从 main 重建
     └─ Claude 从干净状态开始（用户预期"重做"语义达成）
```

**关键性质**：用户重触发 = 干净重做；asynq 自动重试 = 断点续传。两种语义清晰区分。

### 5.4 失败保留 PR（D5）

```
T0      task1 跑了 20 分钟，Claude 生成 25 / 50，遇到 5 次 mvn 编译失败
        Claude 决定 Success=false, FailureCategory="test_quality"
        └─ committed_files=20 (其余 5 个 retry 耗尽 + skipped 25 因预算)

T0      parseResult → validateSuccessfulTestGenOutput → 失败（Success=false）
        ├─ createTestPR：len(CommittedFiles)=20 > 0 → 仍然创建 PR
        │  └─ PR #50 created（用户能看到 20 个测试文件）
        ├─ persistTestGenResult → 写表（含 failure_category）
        ├─ ※ 不调 EnqueueManualReview（D5：失败不主动审）
        └─ Execute 返回 ErrTestGenFailed

T0      Processor → SkipRetry → status=failed
        └─ sendCompletionNotification → 飞书 "Failed"
            ├─ severity=Info（test_quality 不严重）
            └─ body 含 PR #50 链接 + "已生成 20 个测试可参考"

T0+     用户在 Gitea 看到 PR #50（20 个测试 + 失败说明）
        ├─ 可手动接管修复剩余 5 个
        ├─ 可在 PR 上 /review（如有命令）触发评审
        └─ 可关闭 PR，重新 dtw gen-tests 触发新一轮
```

## 6. 错误处理增量

### 6.1 sentinel 增量

```go
// 沿用 M4.1 已有 sentinel；M4.2 不新增 sentinel，仅扩展 FailureCategory 取值
// 但增加：
var ErrTestGenInfraFailure = errors.New("基础设施故障，建议重试")  // 严重，触发 oncall
```

仅用于 Processor 在判定 `Output.FailureCategory == FailureCategoryInfrastructure` 时**额外**包装一层 error，便于通知层用 `errors.Is` 切换 severity；不替换既有 `ErrTestGenFailed`。

### 6.2 重试决策表（M4.2 增量）

| 触发点 | 情况 | 重试? | M4.2 行为 |
|--------|------|-------|----------|
| `failure_category=infrastructure` | Claude 自报基础设施故障 | ✅ 重试（asynq backoff） | 飞书 severity=Warning，body 含"建议人工检查环境" |
| `failure_category=test_quality` | Claude 自报测试质量问题 | ❌ SkipRetry | 飞书 severity=Info；PR 保留半成品 |
| `failure_category=info_insufficient` | Claude 自报信息不足 | ❌ SkipRetry | 飞书附 MissingInfo |
| 容器超时（worker.timeouts.gen_tests） | ctx.Err() | ✅ 重试（断点续传） | sendCompletionNotification → "Retrying" |
| createTestPR 失败 | Gitea 5xx / 网络 | ✅ 重试（findExistingTestPR 幂等） | 已 push 的 commit 保留 |
| persistTestGenResult 失败 | SQLite 错误 | ❌ 不阻断主流程 | 仅日志，主流程继续返回 |
| EnqueueManualReview 失败 | Redis / SQLite 错误 | ❌ 不阻断主流程 | 仅日志；用户可手动触发 review |

### 6.3 Processor 完成路径增量

`Processor.handleSuccess` 在调用 `sendCompletionNotification` 之后，对 `TaskTypeGenTests` 增加分支：
- 已由 `test.Service.Execute` 内部完成 `createTestPR` + `persistTestGenResult` + `EnqueueManualReview`
- Processor 仅负责把 `TestGenResult` 序列化到 `record.Result` 字段（与 review/fix 一致）

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| **稳定分支被人工提交污染** | entrypoint.sh 在 fetch 后检查 `git log --author` 是否仅含 `dtworkflow-bot`；非 bot commit 存在则重置回 base 并日志告警；容器退出前把"已重置"事实写到 `TestGenOutput.Warnings` 字段（M4.1 已有），Processor 通知层读取 Warnings 对应发飞书 severity=Warning 的补充消息，确保运维可见 |
| **断点续传时 Claude 误把已有测试当做"既有"误判风格** | prompt 明确"当前分支上的测试也是本服务历史产出，作为续传基线，不要把它们当成用户既有风格"；ExistingTests 字段加 `Source` 区分 |
| **review 拦截被绕过**（用户手动从 auto-test 创建 PR 时正好有 gen_tests 在跑） | 接受——极小概率；用户可在 PR 上手动评论触发；后续可加 `triggered_by` 维度区分 |
| **Cancel-and-Replace 清理失败导致状态不一致**（旧 PR 没关 + 新任务又跑） | branchCleaner 失败仅 warn；entrypoint 启动时若分支已存在但任务期望"干净"可由 prompt 决定（M4.2 简化为：信任 cleanup 已尽力，cleanup 失败时容器内会复用旧分支 → 退化为断点续传，无副作用） |
| **Cancel 后容器异步 push 与 cleanup 竞态**（cleanup 删分支后旧容器 push 触发 Gitea 重建分支） | 三层兜底：①cleanup 顺序先关 PR 再删分支；②entrypoint 的"落后 base 重置"+"非 bot author 重置"双重守护；③新任务独占 auto-test/{module} 写入，force checkout 语义覆盖孤儿 commit。详见 §4.6 |
| **失败时 PR 保留 + 不审导致仓库 PR 列表堆积** | 失败 PR 数量受限于 module 数量（每 module 最多 1 个）；用户可定期手动清理；后续可加 `dtw gc-test-prs` 命令 |
| **failure_category 由 Claude 自报，可能不准** | 服务端在持久化时做交叉校验（如 RawOutput 含 `mvn-resolver` 错误关键词但 Claude 报 test_quality → 日志告警，但仍信任 Claude 的分类） |
| **EnqueueManualReview 与已自然触发的 review 重复**（用户在 task 结束前已手动从 PR 触发 /review） | review 自身的 Cancel-and-Replace 兜底（同 PR 第二个 review 任务取消第一个） |
| **gen_tests succeeded 瞬间的竞态双触发**：Execute 已 createTestPR 但尚未 EnqueueManualReview 时，外部 push 进来导致 webhook 查 ListActiveGenTestsModules → 可能漏判 active（依任务 status 更新与 Execute 返回顺序而定），正常入队 review；随后 Execute 又 EnqueueManualReview → 两次 review | 接受——review 侧 Cancel-and-Replace 兜底，最多浪费一次任务创建/取消开销；不引入分布式锁 |
| **大仓首次 fetch + git checkout 慢** | 容器 timeout 配额已留余量；后续可考虑浅克隆 + 持久卷加速 |
| **prompt 中 max_retry_rounds 与 worker.timeouts.gen_tests 配合不当** | 在 config validate 增加交叉校验：`max_retry_rounds * 单文件预估 60s * 文件数估算` 若超过 timeout 则 warn |
