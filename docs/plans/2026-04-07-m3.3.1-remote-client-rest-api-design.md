# M3.3.1 远程瘦客户端与 REST API 层 — 设计文档

> 日期：2026-04-07
> 状态：已通过
> 依赖：Phase 1（M1.1-M1.8）、Phase 3 第一轮（M3.1-M3.3）

## 1. 概述

### 1.1 目标

构建两个独立组件：

1. **服务端 REST API 层** — 在现有 `dtworkflow serve` 基础上增加 `/api/v1/` 业务端点，支持任务管理、评审/修复触发、服务状态查询
2. **瘦客户端 `dtw`** — 独立二进制，通过 HTTP API 远程操作 DTWorkflow 服务端，类似 GitHub CLI（`gh`）的定位

### 1.2 动机

当前 DTWorkflow 仅有 Webhook 被动触发入口，无法主动远程操作。运维和开发人员需要：
- 在任意客户端/服务器上远程查看任务状态
- 手动触发 PR 评审和 Issue 修复（不依赖 Webhook 事件）
- 管理多个 DTWorkflow 实例（如生产/测试环境）

## 2. 关键设计决策

| # | 决策点 | 选定方案 | 理由 |
|---|--------|---------|------|
| D1 | CLI 职责边界 | **完全分离** — `dtworkflow` 专做服务端，`dtw` 专做远程客户端 | 职责清晰，避免二进制膨胀 |
| D2 | API 认证 | **静态 Token + 身份标识** — YAML 配置 | 内网场景简单可靠，审计可追溯 |
| D3 | 长耗时任务交互 | **异步 + 轮询语法糖** — 202 返回 task ID，`--wait` 自动轮询 | 底层简单，UX 友好 |
| D4 | 多服务器管理 | **多服务器命名 + `--server` 命令级覆盖** | 类 kubectl context，CI 友好 |
| D5 | API 端点范围 | **全量交付** — 6 类 API 一次实现 | 工作量可控，避免分批集成开销 |
| D6 | 空壳命令处理 | **直接删除** — review-pr / fix-issue / gen-tests | 从未实现，无兼容性负担 |
| D7 | task 命令归属 | **双入口** — `dtworkflow task` 直连 DB，`dtw task` 走 API | 本地排障 + 远程操作并存 |
| D8 | 手动触发校验 | **跳过标签/事件约束** — API 手动触发仅校验目标存在且 open | 用户已显式表达意图 |

## 3. 服务端 REST API 设计

### 3.1 路由结构

所有业务 API 在 `/api/v1/` 前缀下，与现有端点共存：

```
既有端点（不变）：
  GET  /healthz
  GET  /readyz
  POST /webhooks/gitea

新增 REST API（需认证）：
  GET  /api/v1/status                              # 服务状态与版本
  GET  /api/v1/tasks                               # 任务列表（支持过滤）
  GET  /api/v1/tasks/:id                           # 任务详情
  POST /api/v1/tasks/:id/retry                     # 重试失败任务
  POST /api/v1/repos/:owner/:repo/review-pr        # 触发 PR 评审
  POST /api/v1/repos/:owner/:repo/fix-issue        # 触发 Issue 修复
```

### 3.2 认证中间件

**请求格式**：

```
Authorization: Bearer dtw_a1b2c3d4e5f6
```

Token 以 `dtw_` 为固定前缀，便于在日志和代码仓库中排查意外泄露。

**服务端配置**（`dtworkflow.yaml`）：

```yaml
api:
  tokens:
    - token: "dtw_a1b2c3d4e5f6"
      identity: "admin"
    - token: "dtw_x9y8z7w6v5u4"
      identity: "ci-bot"
```

**中间件行为**：
- 无 `Authorization` 头 → 401 Unauthorized
- Token 不匹配 → 401 Unauthorized（`crypto/subtle.ConstantTimeCompare` 防时序攻击）
- 匹配成功 → 将 `identity` 注入 Gin Context，供 handler 和审计日志使用
- `/healthz`、`/readyz`、`/webhooks/gitea` 不经过认证中间件

**配置校验规则**：
- `api.tokens` 非空时注册 API 路由；为空时跳过注册（API 不可用，仅日志提示）
- Token 格式：必须以 `dtw_` 开头，最短 16 字符
- Identity：非空且列表内唯一

### 3.3 统一响应格式

```json
// 成功
{
  "data": { ... }
}

// 错误
{
  "error": {
    "code": "not_found",
    "message": "任务不存在: abc123"
  }
}
```

**错误码枚举**：`unauthorized`、`not_found`、`bad_request`、`conflict`、`bad_gateway`、`internal_error`

#### POST /api/v1/tasks/:id/retry

```json
// Response 200
{
  "data": {
    "task_id": "new-task-id",
    "message": "任务已重新入队"
  }
}
```

约束：仅 `failed` / `cancelled` 状态的任务可重试，否则返回 409 Conflict。

#### POST /api/v1/repos/:owner/:repo/review-pr

```json
// Request
{ "pr_number": 42 }

// Response 202 Accepted
{
  "data": {
    "task_id": "abc123",
    "message": "评审任务已创建"
  }
}
```

**服务端处理流程**：
1. 校验 `pr_number` 为正整数
2. 调用 Gitea API 校验 PR 存在且状态为 open
3. 调用 `EnqueueManualReview()` 创建任务并入队
4. 执行 Cancel-and-Replace（与 Webhook 路径一致）
5. 返回 202 + task_id

#### POST /api/v1/repos/:owner/:repo/fix-issue

```json
// Request
{ "issue_number": 15 }

// Response 202 Accepted
{
  "data": {
    "task_id": "def456",
    "message": "修复任务已创建"
  }
}
```

**与 Webhook 触发的差异**：不检查 `auto-fix` 标签，用户通过 API 手动触发即为显式授权。

### 3.5 手动触发 vs Webhook 触发对比

| 方面 | Webhook 触发 | API 手动触发 |
|------|-------------|-------------|
| 入口 | `HandlePullRequest()` / `HandleIssueLabel()` | `EnqueueManualReview()` / `EnqueueManualFix()` |
| 幂等性 | `delivery_id` 去重 | 合成 ID `manual-{timestamp}-{random}` |
| 前置校验 | Webhook 隐含目标存在 | 主动调用 Gitea API 校验存在且 open |
| 标签/事件约束 | PR opened/synchronized、`auto-fix` 标签 | 无约束，显式意图即授权 |
| Cancel-and-Replace | 自动取消旧任务 | 同样执行（复用现有逻辑） |
| 审计来源 | `triggered_by: "webhook"` | `triggered_by: "manual:{identity}"` |

### 3.6 错误处理

| 场景 | HTTP 状态码 | 错误码 |
|------|-----------|--------|
| 无/错误 Token | 401 | `unauthorized` |
| 路径参数无效 | 400 | `bad_request` |
| PR/Issue 不存在或已关闭 | 400 | `bad_request` |
| Task 不存在 | 404 | `not_found` |
| 重试非终态任务 | 409 | `conflict` |
| Gitea API 不可达 | 502 | `bad_gateway` |
| Redis/Store 异常 | 500 | `internal_error` |

安全原则：不暴露内部错误堆栈，401 响应不区分 "token 不存在" vs "token 错误"。

### 4.3 配置文件

**路径**：`~/.config/dtw/hosts.yml`

```yaml
active: prod
servers:
  prod:
    url: "https://dtworkflow.internal:8080"
    token: "dtw_a1b2c3d4e5f6"
  staging:
    url: "https://staging.internal:8080"
    token: "dtw_x9y8z7w6v5u4"
```

**安全**：文件创建时强制 `0600` 权限，每次读取校验权限并在不安全时警告。

**环境变量覆盖**（CI/CD 场景）：

| 变量 | 说明 |
|------|------|
| `DTW_SERVER_URL` | 覆盖活跃服务器 URL |
| `DTW_TOKEN` | 覆盖活跃服务器 Token |

**优先级链**：`--server flag` > 环境变量 > `hosts.yml active`

### 4.5 `--wait` 轮询策略

- 默认启用（`--no-wait` 关闭）
- 轮询间隔：前 30 秒每 2 秒，之后每 5 秒
- 超时：默认 30 分钟（`--timeout 30m` 可配置）
- 超时后打印当前状态并退出（退出码 2）
- 终端支持时显示 spinner 动画

### 4.6 退出码规范

| 退出码 | 含义 | 示例 |
|-------|------|------|
| 0 | 成功 | 任务完成、查询返回结果 |
| 1 | 失败 | 认证失败、网络错误、任务执行失败 |
| 2 | 超时/部分成功 | `--wait` 超时、评审结果为 request_changes |

### 5.3 服务端 API 依赖注入

```go
// internal/api/router.go

type Dependencies struct {
    Store       store.Store
    QueueClient *queue.Client
    GiteaClient *gitea.Client
    WorkerPool  *worker.Pool
    RedisAddr   string
    Tokens      []TokenConfig
    Version     string
    StartTime   time.Time
    Logger      *slog.Logger
}

func RegisterRoutes(r *gin.Engine, deps Dependencies) {
    if len(deps.Tokens) == 0 {
        deps.Logger.Info("api.tokens 未配置，跳过 REST API 注册")
        return
    }
    v1 := r.Group("/api/v1", TokenAuth(deps.Tokens))
    // ... 注册各 handler
}
```

API handler 只做**校验 + 入队**，不直接调用 `review.Service.Execute()` 或 `fix.Service.Execute()`。任务始终通过 asynq 队列分发，保证：
- 重试、超时、并发控制统一由 asynq 管理
- API 请求快速返回（202），不阻塞 HTTP 连接
- 与 Webhook 触发的任务走同一条执行路径，行为一致

// internal/queue/enqueue.go 新增

func (h *EnqueueHandler) EnqueueManualReview(

## 6. 安全设计

### 6.1 Token 安全

- 服务端配置文件建议 `0600` 权限（文档提示）
- 客户端 `hosts.yml` 创建时强制 `0600`
- `dtw_` 前缀便于 grep 排查泄露
- API 日志中 Token 脱敏显示（仅记录 `dtw_****f6` 末 2 位）

### 6.2 请求安全

- 认证中间件使用 `crypto/subtle.ConstantTimeCompare` 防时序攻击
- API handler 不暴露内部错误堆栈
- `owner` 和 `repo` 路径参数做合法性校验（禁止 `../`、长度限制）

### 6.3 审计日志

- 每个 API 请求记录结构化日志：`identity`、`method`、`path`、`status_code`、`latency`
- 手动触发的任务 `triggered_by` 字段可追溯操作来源

## 10. 验证标准

- [ ] `dtw auth login` 认证到 DTWorkflow 服务端 → `dtw status` 查看连接状态
- [ ] `dtw task list` 远程查询任务列表 → 结果与 `dtworkflow task list` 一致
- [ ] `dtw review-pr --repo owner/myrepo --pr 42` 远程触发评审 → 服务端创建评审任务
- [ ] `dtw fix-issue --repo owner/myrepo --issue 15` 远程触发修复 → 无需 auto-fix 标签
- [ ] `dtw review-pr --repo owner/myrepo --pr 42 --wait` 等待完成 → 显示评审结果
- [ ] 认证多个服务器 → `dtw auth switch` 切换 → 命令作用于切换后的服务器
- [ ] `--server` flag 临时覆盖活跃服务器
- [ ] `DTW_SERVER_URL` + `DTW_TOKEN` 环境变量覆盖（CI 场景）
- [ ] 错误 Token → 401 → 客户端友好提示
- [ ] `dtw task retry` + `--wait` → 重试并等待完成
