# M2.2 评审 Prompt 工程 - 设计文档

> Phase 2 第二个里程碑，在 M2.1 评审编排层基础上精细化 prompt 并改善传递机制
>
> **状态**：设计完成，待实施

## 1. 背景与范围

### 1.1 M2.1 已完成的基础

- 四段式 prompt 结构（上下文 -> 指令 -> JSON schema -> 大 PR 警示）
- `defaultReviewInstructions` 覆盖四维度（security/logic/architecture/style）+ 严重程度定义
- `Dimensions` 字段已预留但未在 prompt 中动态使用
- 评审指令可配置化（全局 YAML + 仓库级追加覆盖）
- Prompt 通过 `claude -p "..."` CLI 参数传入

### 1.2 M2.2 范围

**做**：
- Prompt 改用 stdin 传入（避免 `ps aux` 暴露完整 prompt，消除参数长度限制）
- Java 专项评审 prompt（Spring Boot + MyBatis + MySQL）
- Vue 专项评审 prompt（Vue 3 + Composition API）
- 技术栈自动检测（基于 PR 变更文件扩展名，可被配置覆盖）
- 项目编码规范文件集成（指导 Claude 在容器内自行读取）
- 通用评审指令统一为中文

**不做**：
- Dimensions 动态控制 prompt 段落开关（优先级低，字段保留待后续）
- Prompt 模板引擎化（当前仅 2 个技术栈，模块化拼接足够）
- 其他技术栈专项 prompt（Go、Python 等按需后续扩展）

## 2. Prompt 结构设计

### 2.1 扩展后的 Prompt 结构

四段式框架不变，第 2 段拆分为四个子段：

```
+-------------------------------------------+
| 1. 任务上下文（不变）                        |
|    PR #号、仓库、作者、base 分支、描述、文件列表 |
+-------------------------------------------+
| 2a. 通用评审指令                            |
|     角色设定、评审原则、严重程度定义、四维度要点   |
|     （现有 defaultReviewInstructions 改为中文）|
+-------------------------------------------+
| 2b. 专项评审指令（条件性）                    |
|     Java: Spring Boot / MyBatis 专项检查清单  |
|     Vue: Vue 3 / Composition API 专项检查清单 |
|     可同时触发（前后端混合 PR）                |
+-------------------------------------------+
| 2c. 仓库级追加指令（条件性）                   |
|     来自 YAML 配置的 repo.review.instructions |
|     （现有逻辑不变）                          |
+-------------------------------------------+
| 2d. 项目编码规范（条件性）                    |
|     指导 Claude 在容器内读取仓库中的规范文件     |
|     截断到合理长度，避免 prompt 膨胀            |
+-------------------------------------------+
| 3. 输出格式约束（不变）                       |
|    JSON schema + verdict 规则               |
+-------------------------------------------+
| 4. 大 PR 警示（条件性，不变）                  |
+-------------------------------------------+
```

### 2.2 关键规则

- 2b 根据 PR 变更文件自动检测技术栈，Java 和 Vue 可同时触发
- 2d 采用"指导 Claude 自行读取"策略，零 API 调用
- 整体 prompt 预估上限约 8-10KB（通用指令 ~2KB + 专项 ~2KB x 2 + 规范指引 ~1KB），stdin 传入无长度压力

## 3. stdin 改造

### 3.1 改造动机

| | CLI 参数（当前） | stdin（改造后） |
|---|---|---|
| **安全性** | `ps aux` / `docker top` 可看到完整 prompt | 进程列表只显示 `claude -p - --output-format json` |
| **长度限制** | Linux 单个参数上限 128KB | 无限制 |
| **调试** | 超长参数难以阅读 | 可单独记录 prompt 到日志文件 |

### 3.2 改造后流程

```
review.Service.buildCommand() -> []string{"claude", "-p", "-", "--output-format", "json"}
review.Service.buildPrompt()  -> promptText string

  -> Pool.RunWithCommandAndStdin(ctx, payload, cmd, stdinData)
    -> Docker ContainerCreate(AttachStdin: true, StdinOnce: true)
    -> Docker ContainerAttach(Stdin: true)
    -> 写入 stdinData -> 关闭 stdin
    -> 等待容器结束 -> 读取 stdout
```

### 3.3 接口变更

```go
// internal/worker/pool.go 新增方法

// RunWithCommandAndStdin 与 RunWithCommand 相同，但额外通过 stdin 传入数据。
// 用于评审场景：prompt 通过 stdin 传入，避免命令行参数暴露。
func (p *Pool) RunWithCommandAndStdin(
    ctx context.Context,
    payload model.TaskPayload,
    cmd []string,
    stdinData []byte,
) (*ExecutionResult, error)
```

`review.ReviewPoolRunner` 接口同步更新：

```go
type ReviewPoolRunner interface {
    RunWithCommand(ctx context.Context, payload model.TaskPayload, cmd []string) (*ExecutionResult, error)
    RunWithCommandAndStdin(ctx context.Context, payload model.TaskPayload, cmd []string, stdinData []byte) (*ExecutionResult, error)
}
```

### 3.4 Docker SDK 实现要点

```go
// 容器创建时开启 stdin
containerCfg := &container.Config{
    Cmd:       cmd,
    OpenStdin: true,
    StdinOnce: true,  // stdin 关闭后不再接受输入
    // ... 其他配置不变
}

// 创建后 Attach stdin
resp, err := cli.ContainerAttach(ctx, containerID, container.AttachOptions{
    Stream: true,
    Stdin:  true,
})

// 启动容器
cli.ContainerStart(ctx, containerID, ...)

// 写入 prompt 并关闭 stdin
resp.Conn.Write(stdinData)
resp.CloseWrite()
```

### 3.5 向后兼容

- `RunWithCommand` 保留不变，`fix_issue` / `gen_tests` 等任务继续走原路径
- `review.Service` 改用 `RunWithCommandAndStdin`
- `buildContainerCmd` 中的 `review_pr` 分支保留作为回退

## 4. 技术栈自动检测

### 4.1 检测逻辑

根据 PR 变更文件的扩展名判断涉及哪些技术栈：

```go
// internal/review/prompt.go

type TechStack int

const (
    TechJava TechStack = 1 << iota  // .java
    TechVue                          // .vue, .ts, .tsx, .js, .jsx（需辅助信号）
)

// detectTechStack 从 PR 变更文件列表中检测技术栈
func detectTechStack(files []*gitea.ChangedFile) TechStack {
    var stack TechStack
    for _, f := range files {
        ext := filepath.Ext(f.Filename)
        switch ext {
        case ".java":
            stack |= TechJava
        case ".vue":
            stack |= TechVue
        case ".ts", ".tsx", ".js", ".jsx":
            // 仅当存在 Vue 信号时才标记，避免纯 Node.js 后端误触发
            if hasVueSignal(files) {
                stack |= TechVue
            }
        }
    }
    return stack
}
```

### 4.2 Vue 信号检测

单独的 `.ts`/`.js` 文件不足以判定是 Vue 项目，需要辅助信号：

```go
// hasVueSignal 检查文件列表中是否有 Vue 项目的特征信号
func hasVueSignal(files []*gitea.ChangedFile) bool {
    for _, f := range files {
        if filepath.Ext(f.Filename) == ".vue" {
            return true
        }
        // 路径特征：src/views/, src/components/, src/composables/, src/stores/
        dir := filepath.Dir(f.Filename)
        for _, marker := range []string{"src/views", "src/components", "src/composables", "src/stores"} {
            if strings.Contains(dir, marker) {
                return true
            }
        }
    }
    return false
}
```

### 4.3 配置覆盖

仓库配置中可选的 `tech_stack` 字段，配置优先于自动检测：

```yaml
repos:
  - name: "changyu/backend"
    review:
      tech_stack: ["java"]        # 明确指定，跳过自动检测
  - name: "changyu/frontend"
    review:
      tech_stack: ["vue"]
```

```go
func (s *Service) resolveTechStack(files []*gitea.ChangedFile, cfg ReviewConfig) TechStack {
    // 配置显式指定时直接使用
    if len(cfg.TechStack) > 0 {
        var stack TechStack
        for _, t := range cfg.TechStack {
            switch strings.ToLower(t) {
            case "java":
                stack |= TechJava
            case "vue":
                stack |= TechVue
            }
        }
        return stack
    }
    // 否则自动检测
    return detectTechStack(files)
}
```

### 4.4 前后端混合 PR

当 Java 和 Vue 同时检测到时，两个专项 prompt 段都拼接，不互斥。适用于 monorepo 或前后端同仓场景。

## 5. Java 专项评审 Prompt

基于 Spring Boot + MyBatis + MySQL 常见问题的检查清单：

```go
const javaReviewInstructions = `

### Java 专项评审（Spring Boot + MyBatis）

在通用评审基础上，额外关注以下 Java 生态特有问题：

#### 事务与数据一致性
- @Transactional 使用是否正确：是否加在 public 方法上、传播级别是否合理、是否存在自调用导致事务失效
- 跨表操作是否遗漏事务注解
- 异常类型与 rollbackFor 是否匹配（默认仅回滚 RuntimeException）

#### MyBatis SQL 安全与质量
- ${} 与 #{} 使用是否正确：动态拼接 SQL 必须用 #{}，仅表名/列名等无法参数化的场景才允许 ${}
- 动态 SQL（<if>、<foreach>）是否存在注入风险
- 批量操作是否考虑数据量上限（避免 SQL 过长）
- 分页查询是否使用 LIMIT，避免全表扫描

#### Spring Boot 常见陷阱
- @Autowired 循环依赖：是否通过构造器注入避免
- Controller 层是否混入业务逻辑（应委托给 Service）
- 接口是否遵循 RESTful 规范：HTTP 方法语义、状态码使用、URI 命名
- 配置项是否硬编码（应使用 @Value 或 @ConfigurationProperties）

#### 并发与性能
- 共享可变状态是否有同步保护（Spring Bean 默认单例）
- 数据库连接是否及时释放（try-with-resources 或框架管理）
- N+1 查询问题：循环中是否存在逐条查询数据库
- 大数据量查询是否使用流式或分页处理
`
```

## 6. Vue 专项评审 Prompt

基于 Vue 3 + Composition API 常见问题的检查清单：

```go
const vueReviewInstructions = `

### Vue 专项评审（Vue 3 + Composition API）

在通用评审基础上，额外关注以下 Vue 生态特有问题：

#### 响应式使用
- ref 与 reactive 是否正确选择：原始类型用 ref，对象用 reactive
- 解构 reactive 对象是否使用 toRefs 保持响应式
- watchEffect / watch 是否存在不必要的依赖触发或遗漏清理
- computed 是否用于派生状态（避免 watch + 手动赋值的反模式）

#### 组件设计
- 组件是否遵循单一职责（超过 300 行需关注拆分）
- Props 是否定义类型和默认值（defineProps 使用是否规范）
- 事件是否通过 defineEmits 声明，避免隐式事件
- v-model 使用是否正确（组件双向绑定的 modelValue + update:modelValue 约定）

#### 安全
- v-html 使用是否存在 XSS 风险（是否对用户输入做过滤）
- 动态属性绑定（:href、:src）是否校验来源，防止 javascript: 协议注入
- 敏感信息是否泄露到前端代码或 localStorage

#### 性能
- v-for 是否提供唯一稳定的 key（避免使用 index）
- 大列表是否考虑虚拟滚动
- 组件是否合理使用 defineAsyncComponent 按需加载
- 不必要的深层 watch（deep: true）是否可用更精确的监听替代

#### 状态管理（Pinia）
- Store 是否职责清晰，避免单个 Store 过于庞大
- 异步操作是否在 actions 中处理（避免组件内直接修改 state）
- 是否存在跨 Store 循环依赖
`
```

## 7. 项目编码规范文件集成

### 7.1 策略选择

采用"指导 Claude 在容器内自行读取"策略，而非宿主机通过 API 预取。

理由：
1. Claude Code CLI 在容器内本来就能自主读文件，这是"完整仓库"模式的核心优势
2. 省去 Gitea API 调用和错误处理（文件不存在、权限、网络）
3. 规范文件通常不大，Claude 读取后可以完整理解上下文

### 7.2 Prompt 指令

在 2d 段添加规范文件扫描指引：

```go
const codeStandardsInstruction = `

### 项目编码规范

在评审前，请检查仓库中是否存在以下编码规范文件（按优先级顺序），如果存在请先阅读，并将其作为评审依据：

1. CLAUDE.md 中的"编码规范"或"开发规范"章节
2. .code-standards.md 或 .code-standards/
3. CONTRIBUTING.md
4. .editorconfig
5. docs/coding-standards.md 或 docs/code-style.md
6. README.md 中的"编码规范"或"开发规范"章节

如果找到规范文件，评审时应检查变更代码是否符合项目自身的编码规范。
违反项目规范的问题使用 WARNING 级别，category 标记为 style。
`
```

CLAUDE.md 放在第一优先级，因为在有 Claude Code 的项目中它是最权威的项目约束来源。

### 7.3 配置覆盖

允许仓库配置中指定自定义的规范文件路径，替换默认扫描列表：

```yaml
repos:
  - name: "changyu/backend"
    review:
      code_standards_paths:
        - "docs/java-standards.md"
        - ".checkstyle.xml"
```

当配置了自定义路径时，prompt 中的默认扫描列表替换为配置值：

```go
// prompt 中动态生成
"在评审前，请先阅读以下项目编码规范文件：\n1. docs/java-standards.md\n2. .checkstyle.xml\n..."
```

## 12. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 模块化拼接而非模板引擎 | 当前仅 Java + Vue 两个技术栈，无需 text/template 的灵活性；prompt 内容在代码中方便 code review |
| D2 | stdin 传入 prompt | 安全性（不暴露 prompt）+ 无长度限制 + 可扩展性 |
| D3 | 技术栈自动检测 + 配置覆盖 | 自动检测减少配置负担，配置覆盖应对误检场景 |
| D4 | Vue 信号检测（非仅扩展名） | 单独的 .ts/.js 文件不足以判定 Vue 项目，避免纯 Node.js 后端误触发 |
| D5 | 编码规范采用"指导 Claude 自行读取"策略 | 利用容器内完整仓库的优势，零 API 调用，零额外错误处理 |
| D6 | CLAUDE.md 作为规范文件第一优先级 | Claude Code 项目中最权威的约束来源 |
| D7 | 全中文 prompt（指令和输出） | 团队需求，中文评审意见可读性更高 |
| D8 | Dimensions 动态控制延后 | 当前四维度全开即可，优先级低，字段保留待后续 |
| D9 | 前后端混合 PR 同时触发双专项 prompt | 适用于 monorepo 或前后端同仓场景，不互斥 |
