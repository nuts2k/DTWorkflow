# M4.3 变更驱动测试生成 — 设计文档

> PR 合并后自动分析变更代码是否有对应测试覆盖，缺失时自动触发测试生成。

## 1. 设计决策

| # | 决策点 | 选定方案 | 理由 |
|---|--------|---------|------|
| D1 | 变更驱动粒度 | 混合模式：模块级入队 + prompt 注入变更文件列表 | 入队逻辑复用现有管线，prompt 精确引导 Claude 聚焦变更文件 |
| D2 | 预判断策略 | Host 侧预过滤：按扩展名/路径排除非源码变更 | 避免纯文档/配置 PR 启动容器浪费资源 |
| D3 | 模块映射 | 复用 ScanRepoModules 结果做路径前缀匹配 | 与手动触发行为一致，已处理根级/子模块/双框架边界 |
| D4 | 去重策略 | 复用现有 Cancel-and-Replace | 零新代码，保证最新变更上下文 |
| D5 | BaseRef | 使用 PR 的 Base 分支（合并目标分支） | 变更实际落地的分支，语义最准确 |
| D6 | Bot PR 过滤 | 按分支名前缀过滤 `auto-test/`、`auto-fix/` | 成本极低，杜绝循环风险 |
| D7 | 配置结构 | 最小集：`enabled` + `ignore_paths`，默认关闭 | 新行为应显式启用，避免升级后意外触发 |

## 2. 整体架构与数据流

```
PR Merged (Gitea Webhook)
    │
    ▼
parsePullRequest()  ── action="closed" + merged=true → 映射为 action="merged"
    │
    ▼
HandlePullRequest()
    │
    ├─ action ∈ {opened, synchronized, reopened} → 现有 review 入队逻辑（不变）
    │
    └─ action == "merged" → handleMergedPullRequest()
            │
            ▼
        ① Bot PR 过滤：HeadRef 以 auto-test/ 或 auto-fix/ 开头 → 跳过
            │
            ▼
        ② change_driven.enabled 检查（全局 + 仓库级）→ 未启用则跳过
            │
            ▼
        ③ ListPullRequestFiles() → 获取变更文件列表
            │
            ▼
        ④ filterSourceFiles() → 排除非源码文件（内置规则 + ignore_paths）
            │  过滤后为空 → 跳过，日志记录"无源码变更"
            ▼
        ⑤ ScanRepoModules() → 发现仓库模块结构
            │
            ▼
        ⑥ matchFilesToModules() → 将变更文件按路径前缀归组到模块
            │  无匹配模块 → 跳过
            ▼
        ⑦ 逐模块调用 enqueueSingleGenTests()
           - triggered_by = "webhook:pr_merged:<pr_number>"
           - payload.ChangedFiles = 该模块下的变更文件列表
           - payload.BaseRef = event.PullRequest.BaseRef
           - 复用 Cancel-and-Replace 逻辑
```

核心原则：新增的所有逻辑都在入队层（`internal/queue/enqueue.go`）完成，`test.Service.Execute` 零改动，仅 prompt 层新增变更文件上下文注入。

## 3. Webhook 扩展

### 3.1 parser.go 变更

当前 `parsePullRequest` 仅接受 `opened`/`synchronized`/`reopened`，需扩展支持 `closed`。Gitea 的 PR 合并事件 action 是 `"closed"`，通过 `pull_request.merged` 字段区分关闭和合并。

```go
// 当前条件（L35）：
if payload.Action != "opened" && payload.Action != "synchronized" && payload.Action != "reopened" {
    return nil, ErrUnsupportedAction
}

// 改为：
if payload.Action != "opened" && payload.Action != "synchronized" &&
   payload.Action != "reopened" && payload.Action != "closed" {
    return nil, ErrUnsupportedAction
}

// 在构造 PullRequestEvent 之前，将 closed+merged 映射为语义化的 "merged" action：
action := payload.Action
if payload.Action == "closed" && payload.PullRequest.Merged {
    action = "merged"
} else if payload.Action == "closed" {
    return nil, ErrUnsupportedAction  // 仅关闭未合并，不处理
}
```

### 3.2 event.go 变更

`PullRequestRef` 新增 `Merged` 字段：

```go
type PullRequestRef struct {
    // ... 现有字段 ...
    Merged bool   // PR 是否已合并
}
```

### 3.3 payload 结构补充

`giteaPullRequestEventPayload` 内部类型中 `PullRequest` 需确认包含 `Merged bool` 字段（Gitea webhook payload 原生提供）。

### 3.4 行为语义

| Gitea action | merged 字段 | 映射后 action | 处理 |
|---|---|---|---|
| `opened` | - | `opened` | review 入队（不变） |
| `synchronized` | - | `synchronized` | review 入队（不变） |
| `reopened` | - | `reopened` | review 入队（不变） |
| `closed` | `true` | `merged` | 变更驱动 gen_tests |
| `closed` | `false` | - | `ErrUnsupportedAction` |

## 4. Host 侧预过滤逻辑

### 4.1 内置过滤规则

新建 `internal/queue/change_filter.go`，定义源码文件过滤逻辑：

```go
// 内置忽略的扩展名
var defaultIgnoredExtensions = map[string]bool{
    ".md": true, ".txt": true, ".rst": true,
    ".yaml": true, ".yml": true,
    ".json": true,
    ".toml": true, ".ini": true, ".cfg": true,
    ".xml": true,
    ".lock": true,
    ".gitignore": true, ".editorconfig": true,
    ".dockerignore": true,
    ".env": true,
    ".svg": true, ".png": true, ".jpg": true, ".jpeg": true,
    ".gif": true, ".ico": true,
    ".csv": true,
    ".license": true,
}

// 内置忽略的路径前缀
var defaultIgnoredPrefixes = []string{
    "docs/", "doc/", ".github/", ".gitea/",
    "deploy/", ".vscode/", ".idea/",
}

// 测试文件模式（这些文件本身是测试，不需要为它们生成测试）
var testFilePatterns = []string{
    "*_test.go", "*Test.java", "*Tests.java",
    "*.test.ts", "*.test.js", "*.spec.ts", "*.spec.js",
    "*.test.tsx", "*.test.jsx", "*.spec.tsx", "*.spec.jsx",
}
```

### 4.2 过滤函数签名

```go
func filterSourceFiles(files []string, extraIgnorePaths []string) []string
```

- `files`：`ListPullRequestFiles` 返回的文件路径列表
- `extraIgnorePaths`：来自 `change_driven.ignore_paths` 配置，doublestar glob 匹配
- 返回：通过过滤的源码文件列表

### 4.3 过滤链

按顺序执行，命中任一规则即排除：

1. 扩展名匹配 `defaultIgnoredExtensions` → 排除
2. 路径前缀匹配 `defaultIgnoredPrefixes` → 排除
3. 文件名匹配 `testFilePatterns`（doublestar） → 排除
4. 路径匹配 `extraIgnorePaths`（doublestar） → 排除
5. 通过以上全部 → 保留

### 4.4 边界处理

- `package.json` / `pom.xml` 等构建文件：被 `.json` / `.xml` 内置规则排除。构建配置变更不需要补测试，如果用户希望触发可手动调用 `dtw gen-tests`。
- 过滤后空列表：日志记录 `"PR #N merged: no source code changes after filtering, skipping gen_tests"`，正常返回不报错。

## 5. 模块映射与入队逻辑

### 5.1 模块映射函数

新增 `matchFilesToModules`，将过滤后的源码文件按路径前缀归组到 `ScanRepoModules` 发现的模块：

```go
func matchFilesToModules(files []string, modules []test.DiscoveredModule) map[test.DiscoveredModule][]string
```

映射规则：
- 根级模块（`Path == ""`）：匹配所有不属于其他子模块的文件
- 子模块（如 `Path == "backend"`）：匹配以 `backend/` 为前缀的文件
- 同目录双框架（如 `backend` 既有 `pom.xml` 又有 `package.json`）：该目录下的文件同时归入两个模块，各自独立生成测试
- 无法匹配任何模块的文件（如位于 `scripts/` 下且无框架标记文件）：丢弃，日志记录

### 5.2 handleMergedPullRequest 主流程

```go
func (h *EnqueueHandler) handleMergedPullRequest(
    ctx context.Context, event webhook.PullRequestEvent,
) error {
    repo := event.Repository.FullName
    pr := event.PullRequest

    // 1. Bot PR 过滤
    if strings.HasPrefix(pr.HeadRef, "auto-test/") ||
       strings.HasPrefix(pr.HeadRef, "auto-fix/") {
        return nil
    }

    // 2. 配置检查
    cfg := h.configProvider.Get()
    changeDriven := resolveChangeDrivenConfig(cfg, repo)
    if !changeDriven.IsEnabled() {
        return nil
    }

    // 3. 获取变更文件
    changedFiles, err := h.prFilesLister.ListPullRequestFiles(...)
    if err != nil { return fmt.Errorf("list PR files: %w", err) }

    // 4. 预过滤
    filenames := extractFilenames(changedFiles)
    sourceFiles := filterSourceFiles(filenames, changeDriven.IgnorePaths)
    if len(sourceFiles) == 0 { return nil }

    // 5. 模块扫描
    modules, err := test.ScanRepoModules(ctx, h.moduleScanner,
        event.Repository.Owner, event.Repository.Name, pr.BaseRef)
    if errors.Is(err, test.ErrNoFrameworkDetected) { return nil }
    if err != nil { return fmt.Errorf("scan modules: %w", err) }

    // 6. 文件→模块归组
    moduleFiles := matchFilesToModules(sourceFiles, modules)
    if len(moduleFiles) == 0 { return nil }

    // 7. 逐模块入队
    triggeredBy := fmt.Sprintf("webhook:pr_merged:%d", pr.Number)
    for mod, files := range moduleFiles {
        payload := model.TaskPayload{
            TaskType:     model.TaskTypeGenTests,
            RepoOwner:    event.Repository.Owner,
            RepoName:     event.Repository.Name,
            RepoFullName: repo,
            CloneURL:     event.Repository.CloneURL,
            Module:       mod.Path,
            Framework:    string(mod.Framework),
            BaseRef:      pr.BaseRef,
            ChangedFiles: files,
        }
        _, _ = h.enqueueSingleGenTests(ctx, payload, triggeredBy)
    }
    return nil
}
```

### 5.3 TaskPayload 扩展

`model.TaskPayload` 新增一个字段：

```go
type TaskPayload struct {
    // ... 现有字段 ...
    ChangedFiles []string `json:"changed_files,omitempty"` // 变更驱动：触发变更的源码文件列表
}
```

该字段仅在变更驱动场景下非空，手动触发时为空。prompt 层据此决定是否注入变更文件上下文。

### 5.4 依赖注入

`handleMergedPullRequest` 需要的依赖：

| 依赖 | 现有状态 | 需要新增？ |
|------|---------|-----------|
| `moduleScanner` (RepoFileChecker) | M4.2.1 已通过 `WithModuleScanner` 注入 | 否 |
| `configProvider` | 已有 `WithConfigProvider` | 否 |
| `prFilesLister` (ListPullRequestFiles) | 需新增 | **是** |
| `branchCleaner` | 已有 `WithBranchCleaner` | 否 |

新增窄接口：

```go
type PRFilesLister interface {
    ListPullRequestFiles(ctx context.Context, owner, repo string,
        index int64, opts gitea.ListOptions) ([]*gitea.ChangedFile, error)
}

func WithPRFilesLister(lister PRFilesLister) EnqueueOption {
    return func(h *EnqueueHandler) { h.prFilesLister = lister }
}
```

`handleMergedPullRequest` 中如果 `prFilesLister == nil`，静默跳过变更驱动（fail-open）。

### 5.5 HandlePullRequest 路由扩展

```go
func (h *EnqueueHandler) HandlePullRequest(ctx context.Context, event webhook.PullRequestEvent) error {
    switch event.Action {
    case "opened", "synchronized", "reopened":
        return h.handleReviewPullRequest(ctx, event)  // 现有逻辑提取为方法
    case "merged":
        return h.handleMergedPullRequest(ctx, event)
    default:
        return nil
    }
}
```

现有 `HandlePullRequest` 的逻辑体提取为 `handleReviewPullRequest`（纯重命名，零行为变更）。

## 6. Prompt 变更上下文注入

### 6.1 PromptContext 扩展

```go
type PromptContext struct {
    // ... 现有字段 ...
    ChangedFiles []string // 变更驱动场景下的变更文件列表，空切片 = 手动触发
}
```

`Execute` 中从 `payload.ChangedFiles` 透传到 `PromptContext.ChangedFiles`。

### 6.2 注入位置

在现有 prompt 的"上下文段"和"指令段"之间插入变更聚焦段（仅 `ChangedFiles` 非空时）：

```
[现有] 上下文段（仓库、模块、分支信息）
                │
[新增] 变更聚焦段（仅 ChangedFiles 非空时）
                │
[现有] 指令段（缺口分析 + 生成 + 验证）
```

### 6.3 变更聚焦段内容

```text
## 变更驱动上下文

本次测试生成由 PR 合并触发，以下是本次合并涉及的源码文件：

- backend/src/main/java/com/example/UserService.java
- backend/src/main/java/com/example/UserController.java

请优先为上述变更文件生成或补充测试。如果这些文件已有充分的测试覆盖，
你仍然可以为同模块内其他缺少测试的代码生成测试，但变更文件优先级最高。
如果所有变更文件均已有充分测试覆盖且你认为无需额外补充，请在 analysis
字段中说明，并将 success 设为 true、generated_files 设为空数组。
```

### 6.4 行为差异

| 场景 | ChangedFiles | Prompt 行为 |
|------|-------------|------------|
| 手动触发 | 空 | 现有行为不变：全模块缺口分析 |
| 变更驱动 | 非空 | 优先变更文件，兼顾模块缺口 |

### 6.5 文件列表截断

变更文件过多时，prompt 中的文件列表上限 **50 个文件**，超过时截断并注明 `...及其他 N 个文件`。

## 7. 配置结构与校验

### 7.1 新增类型定义

```go
type ChangeDrivenConfig struct {
    Enabled     *bool    `mapstructure:"enabled"`      // 指针语义，nil=false（默认关闭）
    IgnorePaths []string `mapstructure:"ignore_paths"`  // doublestar glob
}

func (c ChangeDrivenConfig) IsEnabled() bool {
    return c.Enabled != nil && *c.Enabled
}
```

注意：`Enabled` 的 `nil` 语义与 `TestGenOverride.Enabled` **相反**——test_gen 默认启用（nil=true），change_driven 默认关闭（nil=false）。

### 7.2 结构挂载

```go
type TestGenOverride struct {
    // ... 现有字段（Enabled, ModuleScope, MaxRetryRounds, TestFramework, ReviewOnFailure）
    ChangeDriven *ChangeDrivenConfig `mapstructure:"change_driven"`
}
```

仓库级通过 `RepoConfig.TestGen.ChangeDriven` 覆盖。

### 7.3 配置解析函数

```go
func resolveChangeDrivenConfig(cfg *Config, repoFullName string) ChangeDrivenConfig {
    // 1. 全局默认：enabled=false, ignore_paths=nil
    // 2. 全局配置覆盖
    // 3. 仓库级覆盖（ignore_paths 替换而非追加）
}
```

### 7.4 配置校验

- `change_driven.ignore_paths` 每个条目做 `doublestar.ValidatePattern` 语法校验
- `change_driven.enabled=true` 时要求 `test_gen.enabled` 不为 `false`

### 7.5 YAML 配置示例

```yaml
test_gen:
  enabled: true
  max_retry_rounds: 3
  change_driven:
    enabled: true
    ignore_paths:
      - "scripts/**"
      - "deploy/**"

repos:
  - name: "myorg/backend-monorepo"
    test_gen:
      change_driven:
        enabled: true
  - name: "myorg/docs-only-repo"
    test_gen:
      change_driven:
        enabled: false
```

### 7.6 Clone 深拷贝

`Config.Clone()` 补充 `ChangeDrivenConfig` 的深拷贝（指针字段 `Enabled` + 切片字段 `IgnorePaths`）。

## 8. 通知、错误处理与可观测性

### 8.1 通知复用

变更驱动触发的 gen_tests 任务与手动触发在 Processor 层完全一致，复用现有三事件：

- `EventGenTestsStarted` — 飞书蓝色卡片
- `EventGenTestsDone` — 飞书绿色卡片 + PR 评论
- `EventGenTestsFailed` — 飞书红色/橙色卡片

触发来源通过 `triggered_by` 列区分：

| 触发来源 | triggered_by 格式 |
|---------|------------------|
| API 手动触发 | `api:<identity>` |
| CLI 手动触发 | `cli` |
| 变更驱动 | `webhook:pr_merged:<pr_number>` |

### 8.2 错误处理策略

| 步骤 | 失败行为 | 理由 |
|------|---------|------|
| Bot PR 过滤 | 不可能失败 | 纯字符串判断 |
| 配置检查 | 不可能失败 | 配置已加载 |
| ListPullRequestFiles | 返回 error，Processor 重试 | 网络/API 临时故障 |
| filterSourceFiles | 不可能失败 | 纯内存操作 |
| ScanRepoModules | `ErrNoFrameworkDetected` → 静默跳过；其他 error → 返回重试 | 无框架仓库是正常情况 |
| matchFilesToModules | 空结果 → 静默跳过 | 变更不在可测试模块内 |
| enqueueSingleGenTests | 单模块失败 warn 日志，不阻断其他模块 | best-effort 策略 |

### 8.3 日志可观测性

关键日志点（结构化 slog）：

```go
// 跳过场景 — Debug 级别
slog.Debug("change-driven: skipping bot PR", "repo", repo, "head_ref", pr.HeadRef)
slog.Debug("change-driven: disabled for repo", "repo", repo)
slog.Debug("change-driven: no source files after filtering",
    "repo", repo, "pr", pr.Number, "total_files", len(filenames))
slog.Debug("change-driven: no framework detected", "repo", repo)
slog.Debug("change-driven: no modules matched",
    "repo", repo, "source_files", len(sourceFiles))

// 正常触发 — Info 级别
slog.Info("change-driven: enqueueing gen_tests",
    "repo", repo, "pr", pr.Number,
    "source_files", len(sourceFiles),
    "modules", len(moduleFiles),
    "triggered_by", triggeredBy)

// 失败 — Warn 级别
slog.Warn("change-driven: failed to enqueue module",
    "repo", repo, "module", mod.Path, "error", err)
```

### 8.4 不新增数据库表/列

变更驱动触发的任务使用现有 `tasks` 表和 `test_gen_results` 表，通过 `triggered_by` 列区分触发来源。无需数据库迁移。
