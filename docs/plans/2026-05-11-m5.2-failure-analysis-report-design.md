# M5.2 失败分析与报告 — 设计文档

> 日期：2026-05-11
> 状态：设计确认
> 依赖：M5.1（E2E 基础设施与执行引擎）已完成
> 前序设计：`docs/plans/2026-05-09-m5.1-e2e-infrastructure-design.md`

## 1. 概述

### 1.1 目标

在 M5.1 E2E 执行引擎基础上，新增失败分析与报告能力：AI 分类失败原因、自动创建 Gitea Issue（附截图和上下文）、脚本过时类失败联动 Phase 3 自动修复、增强飞书通知卡片。

### 1.2 M5.1 已有基础

- `FailureCategory`（bug / script_outdated / environment）和 `FailureAnalysis` 已在 `CaseResult` 输出 schema 中
- `Screenshots` 字段已在 `CaseResult` 中（但 artifact volume 未实装，截图在 tmpfs 中随容器销毁）
- E2E 三事件通知已在 Processor 中接线（EventE2EStarted / EventE2EDone / EventE2EFailed）
- 飞书卡片尚未对 E2E 做专项渲染（走 default 分支）

### 1.3 不做什么

- 模块拆分并行执行（M5.3）
- 回归自动化 / Webhook PR merged 触发（M5.4）
- E2E 趋势分析与每日统计报告（M5.4）
- Issue 修复后自动重跑 E2E 验证
- Artifact 定期清理的 periodic task（启动时清理即可，periodic 留后续）

## 2. 关键设计决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | Issue 创建粒度 | 混合策略：bug / script_outdated 逐用例创建，environment 不创建 Issue | bug / script_outdated 需要精准修复，逐用例与 Phase 3 单 Issue → 单 PR 模型对齐；environment 通常是统一原因，创建 Issue 增加噪音 |
| D2 | 截图附到 Issue | Gitea Attachment API 上传，截图作为 Issue 附件展示 | 截图是 E2E 失败分析的核心证据，直接附在 Issue 中价值最高 |
| D3 | auto-fix 联动标签 | `fix-to-pr`（非 `auto-fix`） | `auto-fix` 触发 `analyze_issue`（只读分析），`fix-to-pr` 触发 `fix_issue`（修复 + PR 创建）。E2E 失败报告本身已是分析结果，直接走修复链路 |
| D4 | 测试报告载体 | 飞书卡片即报告 | E2E 主要消费者通过飞书群接收通知，卡片能承载用例统计 + 失败摘要；失败详情已写入独立 Issue |
| D5 | Issue 创建 Token | 复用 fix token（`GiteaFixClient`） | review token 保持纯粹的评审场景；fix token 已有写权限；E2E 不创建 PR，无自评审冲突 |
| D6 | 幂等保护 | DB guard（e2e_results 表 `created_issues` 字段）+ Issue 标题带标记 | 本地 DB 幂等 guard 最可靠，与 test_gen_results 两阶段写入模式一致；多实例迁移时统一切换共享 DB 即可 |
| D7 | environment 失败 | 仅飞书通知，不创建 Issue | 环境问题处理方式是"修环境 + 重跑"，不需要 Issue 跟踪 |
| D8 | Issue 创建逻辑位置 | e2e.Service 内部处理（方案 A） | 与 fix.Service 模式一致；Service 天然拥有完整上下文（case.yaml expectations、截图路径） |

```go
type CreateIssueOption struct {

    Body   string   `json:"body"`

// CreateIssue 创建 Issue。
// POST /api/v1/repos/{owner}/{repo}/issues
func (c *Client) CreateIssue(ctx context.Context, owner, repo string,

```

### 3.2 ListRepoLabels

```go
// Label 标签。
type Label struct {
    ID   int64  `json:"id"`
    Name string `json:"name"`
}

// ListRepoLabels 列出仓库所有标签。
// GET /api/v1/repos/{owner}/{repo}/labels
func (c *Client) ListRepoLabels(ctx context.Context, owner, repo string) ([]Label, *Response, error)
```

用于按名称查找 `fix-to-pr` 标签的 ID（Gitea 创建 Issue 的 labels 字段接受 ID 数组）。

### 3.3 CreateIssueAttachment

```go

type Attachment struct {
    ID          int64  `json:"id"`
    Name        string `json:"name"`
    DownloadURL string `json:"browser_download_url"`
}

// CreateIssueAttachment 上传 Issue 附件。
// POST /api/v1/repos/{owner}/{repo}/issues/{index}/assets
// Content-Type: multipart/form-data
func (c *Client) CreateIssueAttachment(ctx context.Context, owner, repo string,
```

## 4. Artifact Volume 实装

### 4.1 Worker 层改动

`runContainer` 中为 `TaskTypeRunE2E` 追加 bind mount：

```
宿主机: {data_dir}/e2e-artifacts/{task_id}/
容器内: /workspace/artifacts/
```

`PoolConfig` 新增 `DataDir` 字段，从 `config.DataDir` 透传。容器创建前 `os.MkdirAll` 确保宿主机目录存在。

### 4.2 目录结构

```
{data_dir}/e2e-artifacts/
└── {task_id}/
    ├── test-results/           # Playwright 截图 + trace
    │   ├── create-order/
    │   │   └── screenshot-1.png
    │   └── login/
    │       └── screenshot-1.png
    └── reports/                # Playwright JSON 报告
        └── results.json
```

### 4.3 清理机制

`e2e.artifact_retention_days` 配置（默认 7 天，范围 [1, 90]）。`serve` 启动时扫描 `e2e-artifacts/` 目录，删除超过保留天数的子目录。后续可升级为 asynq periodic task。

### 4.4 截图路径转换

`processFailures` 中将容器内路径转为宿主机路径：

```
容器路径: /workspace/artifacts/test-results/login/screenshot-1.png
宿主路径: {data_dir}/e2e-artifacts/{task_id}/test-results/login/screenshot-1.png
```

转换规则：将 `/workspace/artifacts/` 前缀替换为 `{data_dir}/e2e-artifacts/{task_id}/`。

## 5. e2e.Service 扩展

### 5.1 新增窄接口

```go
// IssueClient Issue 创建 + 标签查询 + 附件上传。
type IssueClient interface {
    CreateIssue(ctx context.Context, owner, repo string,
        opts gitea.CreateIssueOption) (*gitea.Issue, *gitea.Response, error)
    ListRepoLabels(ctx context.Context, owner, repo string) ([]gitea.Label, *gitea.Response, error)
    CreateIssueAttachment(ctx context.Context, owner, repo string,
        issueIndex int64, filename string, reader io.Reader) (*gitea.Attachment, *gitea.Response, error)
}

// E2EStore 持久化接口。
type E2EStore interface {
    SaveE2EResult(ctx context.Context, result *E2EResultRecord) error
    GetE2EResultByTaskID(ctx context.Context, taskID string) (*E2EResultRecord, error)
    UpdateE2ECreatedIssues(ctx context.Context, id string, issues map[string]int64) error
}
```

### 5.2 Service 结构体扩展

```go
type Service struct {
    pool        E2EPoolRunner
    cfgProv     E2EConfigProvider
    issueClient IssueClient   // M5.2 注入，nil 时跳过 Issue 创建
    store       E2EStore      // M5.2 注入，nil 时降级为仅日志记录
    artifactDir string        // 宿主机 artifact 根目录
    logger      *slog.Logger
}

func WithIssueClient(c IssueClient) ServiceOption { ... }
func WithStore(store E2EStore) ServiceOption { ... }
func WithArtifactDir(dir string) ServiceOption { ... }
```

### 5.3 Execute 八步流程

```
1. 前置校验（Enabled / resolveEnvironment / baseURL 覆盖）     — M5.1 不变
2. 构建 Prompt + 容器执行                                      — prompt 增强（§7）
3. 解析结果（双层 JSON → E2EOutput → validate → sanitize）     — M5.1 不变
4. 持久化阶段 1（SaveE2EResult — 聚合统计，created_issues={}） — M5.2 新增
5. processFailures（遍历失败 → 创建 Issue → 上传截图）         — M5.2 新增
6. 持久化阶段 2（UpdateE2ECreatedIssues — partial UPDATE）     — M5.2 新增
7. 返回 E2EResult（含 CreatedIssues）
→ Processor：通知（飞书增强卡片）
```

解析失败（步骤 3）→ `ErrE2EParseFailure` → asynq 重试（步骤 4-6 不执行）。

### 5.4 processFailures 逻辑

```go
func (s *Service) processFailures(ctx context.Context, payload model.TaskPayload,
    result *E2EResult, savedIssueNumbers map[string]int64) error
```

遍历 `result.Output.Cases`，对每个失败用例：

1. **跳过条件**：`FailureCategory == "environment"`（D7）；`savedIssueNumbers[casePath]` 已有值（幂等）
2. **构建 Issue body**：调用 `formatBugIssueBody` 或 `formatScriptOutdatedIssueBody`
3. **查找标签 ID**：`script_outdated` 时查找 `fix-to-pr` 标签 ID（缓存到本次执行，只查一次）
4. **创建 Issue**：标题 `[E2E] {module}/{caseName} — {failureCategory}`
5. **上传截图**：遍历 `case.Screenshots`，路径转换后打开文件，调用 `CreateIssueAttachment`（单张 > 5MB 跳过）
6. **记录 Issue number** 到 `CreatedIssues` map

### 5.5 失败处理保证

- Issue 创建失败：warn 日志，不阻断主流程
- 截图上传失败：warn 日志，Issue 已创建不受影响
- 标签查找失败：warn 日志，Issue 仍创建但不带标签（降级为人工标记）
- processFailures 部分成功：已创建的 Issue 写入阶段 2，重试时断点续传

## 6. Issue Body 格式化

### 6.1 新建 `internal/e2e/formatter.go`

```go
func formatBugIssueBody(c CaseResult, payload model.TaskPayload, envName string) string
func formatScriptOutdatedIssueBody(c CaseResult, payload model.TaskPayload, envName string) string
```

### 6.2 bug 类 Issue Body

```markdown
## E2E 测试失败 — 疑似应用缺陷

**用例**：{case_name}
**模块**：{module}
**用例路径**：`e2e/{module}/cases/{case_name}/`
**测试环境**：{env_name} ({base_url})
**代码基线**：{base_ref}
**失败阶段**：{failed_phase}

### 失败分析

{failure_analysis}

### 错误信息

｀｀｀
{script_error_msg}
｀｀｀

### 业务意图（case.yaml expectations）

| 步骤 | 预期结果 |
|------|---------|
| {step_1} | {expect_1} |
| ... | ... |

### 失败脚本

- `{script_name}` — 退出码 {exit_code}

### 截图

见下方附件。

---
<!-- dtworkflow:e2e:{task_id}:{case_path} -->
此 Issue 由 DTWorkflow E2E 测试引擎自动创建。
```

### 6.3 script_outdated 类 Issue Body

与 bug 类结构相同，额外追加修复指引段：

```markdown
### 修复指引

此用例因页面元素或流程变更导致脚本无法正常执行。
请修复 `{script_path}` 以匹配当前页面结构，保持上述 expectations 中描述的业务意图不变。

> 此 Issue 已标记 `fix-to-pr`，DTWorkflow 将尝试自动修复。
```

### 6.4 Expectations 来源

`CaseResult` 扩展 `Expectations` 字段（复用 `case.go` 中已定义的 `Expectation` 类型）：

```go
type CaseResult struct {
    // ... 现有字段不变
    Expectations []Expectation `json:"expectations,omitempty"`
}
```

通过 prompt 指引 Claude 将 case.yaml 中的 expectations 原样输出到结果中。容器销毁后无需再访问仓库文件。

### 6.5 安全加固

- 所有外部内容做 Markdown 转义（复用现有 `escapeMarkdown`）
- `failure_analysis` 和 `error_msg` UTF-8 安全截断（上限 4KB）
- expectations 的 step/expect 做转义（防 Markdown 注入）
- Issue body 总长度上限 60000 字节
- 底部锚点 `<!-- dtworkflow:e2e:{task_id}:{case_path} -->` 不转义（供审计用，幂等依赖 DB guard）

## 7. E2E 结果持久化

### 7.1 迁移脚本

```sql
CREATE TABLE e2e_results (
    id              TEXT PRIMARY KEY,
    task_id         TEXT UNIQUE REFERENCES tasks(id) ON DELETE SET NULL,
    repo            TEXT NOT NULL,
    environment     TEXT NOT NULL DEFAULT '',
    module          TEXT NOT NULL DEFAULT '',
    total_cases     INTEGER NOT NULL DEFAULT 0,
    passed_cases    INTEGER NOT NULL DEFAULT 0,
    failed_cases    INTEGER NOT NULL DEFAULT 0,
    error_cases     INTEGER NOT NULL DEFAULT 0,
    skipped_cases   INTEGER NOT NULL DEFAULT 0,
    success         INTEGER NOT NULL DEFAULT 0,
    duration_ms     INTEGER NOT NULL DEFAULT 0,
    created_issues  TEXT NOT NULL DEFAULT '{}',
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_e2e_results_task_id ON e2e_results(task_id);
CREATE INDEX idx_e2e_results_repo ON e2e_results(repo);
CREATE INDEX idx_e2e_results_created_at ON e2e_results(created_at);
```

### 7.2 关键设计点

- **`task_id` 可空 + ON DELETE SET NULL**：与 `test_gen_results` 一致
- **`created_issues` JSON 字段**：`{"e2e/order/cases/create-order": 42, ...}`，用于幂等 guard
- **不存储用例级明细**：单用例结果在 `E2EOutput` JSON 中（TaskRecord.result），e2e_results 只存聚合统计 + Issue 映射

### 7.3 Store 接口

```go
SaveE2EResult(ctx context.Context, result *E2EResultRecord) error
GetE2EResultByTaskID(ctx context.Context, taskID string) (*E2EResultRecord, error)
UpdateE2ECreatedIssues(ctx context.Context, id string, issues map[string]int64) error
```

### 7.4 两阶段写入

- **阶段 1**：Execute 解析结果后立即 UPSERT 主记录（`created_issues = '{}'`）
- **阶段 2**：processFailures 完成后 `UpdateE2ECreatedIssues` partial UPDATE（只更新 `created_issues` + `updated_at`）

```

```

**部分失败卡片（橙色）**：

```
E2E 测试部分失败 -- {repo}
```

- 失败用例列表最多展示 5 个，超过时显示"...及其他 N 个失败用例"

- 已创建的 Issue 号作为链接展示（`{gitea_base_url}/{repo}/issues/{number}`）
- environment 失败用 ⚠ 前缀（区别于 ✗），暗示非代码问题

## 9. Prompt 增强

### 9.1 失败分析指令扩充

在 `prompt.go` 第三段（失败分析指令）中替换为更详细的分析步骤：

```
当测试失败时，执行以下分析步骤：

1. 确定失败阶段：setup / test / teardown
2. 对于 test 阶段失败，对比 case.yaml 中的 expectations 和实际行为：
   - 应用行为与 expectations 描述的业务意图不符 → "bug"
   - 页面元素找不到、选择器失效、DOM 结构变更 → "script_outdated"
   - 服务不可达、数据库连接超时、网络错误 → "environment"
3. 对于 setup/teardown 阶段失败：
   - SQL 连接错误 → "environment"
   - SQL 语法错误或脚本 bug → "script_outdated"

failure_analysis 字段请详细描述：
- 具体哪个操作/断言失败了
- 实际观察到的行为是什么
- 与 expectations 的哪条预期不符
- 如果是 script_outdated，指出哪个选择器或交互方式需要更新
```

### 9.2 Expectations 回填要求

在第四段（输出格式）中追加：

```
每个用例的结果中，请将 case.yaml 中的 expectations 原样输出到 "expectations" 字段：
"expectations": [{"step": "...", "expect": "..."}]
如果 case.yaml 中没有 expectations 段，该字段输出为空数组。
```

同步更新 `e2eOutputSchema` 常量，追加 `expectations` 字段示例。
