## 1. 背景

今天中午 12 点之后，代码库已连续落地以下能力：

- M1.5-6：共享数据模型、SQLite 持久化、任务队列、Docker Worker、`serve` 集成、`task status/list/retry`

当前问题不在于“缺少设计”，而在于**文档状态落后于代码状态**：

- `docs/ROADMAP.md` 仍将 M1.5 / M1.6 / M1.7 主要标记为未完成

本次文档同步要准确回答三个问题：

1. 今天中午后到底已经实现了什么

## 4. 方案比较

### 方案 A：只改路线图

优点：改动最小。

缺点：无法保留“为什么是 M1.5-6 合并交付”“M1.7 为什么先最小接线再等 M1.8 配置化”等实施上下文。

### 方案 B：路线图 + 实施记录同步（采用）

优点：

- 主文档和实施文档能相互印证
- 既反映状态，又保留设计与边界
- 最适合本次“根据今天中午后代码同步文档”的需求

缺点：改动略多于方案 A，但仍可控。

### 方案 C：再新增阶段总结文档

优点：汇报视角更集中。

缺点：容易与 `ROADMAP` 和 `plans` 下文档重复，维护成本高。

## 5. 文档表达策略

### 5.1 ROADMAP 表达策略

采用折中表达：

- 保留 `M1.5`、`M1.6`、`M1.7` 原编号
- 在标题或说明中明确：**本轮按 M1.5-6 合并方案实施与交付**
- 不改动 Phase 结构，避免破坏既有路线图可读性

### 5.2 状态判定原则

只有当代码中已存在稳定实现、并且主流程已接线时，才在路线图中标记为已完成。

据当前代码，以下内容可视为已落地：

  - 任务类型定义与生命周期管理

  - `dtworkflow task status/list/retry`
- M1.6
  - Docker SDK 集成
  - Worker Pool 与容器生命周期管理
  - `serve` 对 queue/store/pool/asynq 的组装

  - `notify.Notifier` / `notify.Router` / `notify.GiteaNotifier`
