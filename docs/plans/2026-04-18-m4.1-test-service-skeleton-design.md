# M4.1 test.Service 骨架 + AI 原生测试缺口分析 + Prompt 工程 — 设计文档

> 日期：2026-04-18
> 状态：已通过
> 依赖：Phase 1（M1.1-M1.8）、Phase 2（review 全链路）、Phase 3（M3.1-M3.5：fix.Service 模式、PR 创建、worker-full 镜像）
> 范围：Phase 4 第一轮第一个里程碑；产出 `internal/test` 包骨架与 Prompt，M4.2 激活容器执行 + Processor 路由。

## 1. 概述

### 1.1 目标

为 Phase 4「集中化测试（测试补全）」搭建骨架：

1. **新建 `internal/test` 包**，架构仿照 `internal/review` 与 `internal/fix` 的 Service/Execute 模式
2. **定义 `TestExecutor` 窄接口**（M4.2 激活路由时使用）
3. **AI 原生测试缺口分析**的 Prompt 工程（Java / Vue 两套独立模板 + 公共段）
4. **`TestGenOutput` JSON schema** + 成功不变量校验
5. **手动触发入口**（REST API / 服务端 CLI / 瘦客户端）
6. **`test_gen` 配置段** 及仓库级覆盖

### 1.2 非目标（延后到 M4.2 / M4.3）

- 容器内真实执行 `mvn test` / `npm test`（M4.2）
- `ProcessorOption.WithTestService` 实际装配与路由（M4.2）
- `buildStartMessage` / `buildNotificationMessage` 新增 `gen_tests` case（M4.2）
- PR 创建的真实 Gitea API 调用（M4.2，`createTestPR` 在 M4.1 仅占位签名）
- 结果持久化 `test_gen_results` 表（M4.2）
- PR merged Webhook 触发变更驱动测试（M4.3）

### 1.3 M4.1 与 M3.1（fix 骨架期）的类比

| 阶段 | 已落地 | 未激活 |
|------|--------|--------|
| M3.1（fix 骨架） | `internal/fix` 包、`FixExecutor` 接口、sentinel、prompt | Processor 未注入 `fixService`，fix_issue 走 `pool.Run` default |
| M4.1（test 骨架） | `internal/test` 包、`TestExecutor` 接口、sentinel、prompt、**手动入口四套** | Processor 未注入 `testService`，gen_tests 走 `pool.Run` default |

刻意对称：M4.2 只需加 `WithTestService(svc)` 一行装配 + Processor 错误分发代码，即可激活新路径，不触碰 M4.1 产出的 Service 骨架。

## 2. 关键设计决策

> 以下 7 决策在 2026-04-18 brainstorming 中确认，另追加 2 项补充。

| # | 决策点 | 选定方案 | 理由 |
|---|--------|---------|------|
| D1 | Service 骨架深度 | **完整 Service，Processor 暂不注入**（M3.1 模式） | `TestExecutor` 接口签名完整；Service 方法签名齐全；`createTestPR` 占位；M4.2 加一行装配即激活 |
| D2 | 容器调用结构 | **单次 `pool.Run`，Claude 内部多步**（read/edit/bash） | 与 `FixOutput` schema 对称；`max_retry_rounds` 指容器内 Claude 自主修正次数；避免跨任务上下文传递 |
| D3 | 手动入口参数 | **`--owner --repo` 必填；`--module --ref --framework` 可选**；复用 `BaseRef` 传递 ref | 与 review-pr 风格对齐；`test_gen.module_scope` / `test_gen.test_framework` 配置已是护栏；小仓库整仓生成合理 |
| D4 | `TestGenOutput` 细节 | **独立 `TestResults` 含 Framework**；保留 `InfoSufficient/MissingInfo`；保留 `GeneratedFile.TargetFiles` | 混合仓 Java/Vue 区分；对称 FixOutput 的不足判定；PR 描述与 M4.3 变更驱动可直接用 |
| D5 | Cancel-and-Replace | **按 `repo + module` 粒度替换** | 同 review / fix 惯例；防止重复 PR + Claude 预算浪费；不同 module 并发放行 |
| D6 | `WriteDegraded` | **接口保留，M4.1 no-op 实现** | 保持接口形状，M4.2 不需破坏 `TestExecutor` 签名；M4.1 无通知集成，no-op 符合 milestone 边界 |
| D7 | Prompt 模板 | **Java/Vue 独立模板 + `resolveFramework` + 公共段常量化** | 可独立单测；指令密度高，Claude 不在混合仓错位；公共段抽 `promptCommonHeader` 等常量避免双份维护 |
| D8 | **既有测试处理**（补充） | **M4.1 Prompt 工程 + Schema 补丁** | ROADMAP 本身已纳入 M4.1 范围；落到 prompt 指令 + `ExistingTests` / `Operation` 字段 |
| D9 | **工作丢失防护**（补充） | **M4.1 埋 Prompt 指令（增量 commit + 最后一刻 push），M4.2 落实执行与恢复** | 不做半成品 push；容器被 kill 时分支未创建无副作用；M4.2 可选择持久卷抢救策略 |

### 4.1 `TestExecutor` 接口（`internal/queue/processor.go`）

```go
type TestExecutor interface {
    Execute(ctx context.Context, payload model.TaskPayload) (*test.TestGenResult, error)
    WriteDegraded(ctx context.Context, payload model.TaskPayload, result *test.TestGenResult) error
}

func WithTestService(svc TestExecutor) ProcessorOption { // M4.2 激活
    return func(p *Processor) { p.testService = svc }
}
```

### 4.2 `test.Service` 结构

**窄接口**（仅所需 Gitea API / 执行器 / 配置）：

```go
// 仓库信息/ref 校验
type RepoClient interface {
    GetRepo(ctx context.Context, owner, repo string) (*gitea.Repository, *gitea.Response, error)
    GetBranch(ctx context.Context, owner, repo, branch string) (*gitea.Branch, *gitea.Response, error)
}

// 复用 M3.5 PR 创建能力；*gitea.Client 已满足
type PRClient interface {
    CreatePullRequest(ctx context.Context, owner, repo string,
        opts gitea.CreatePullRequestOption) (*gitea.PullRequest, *gitea.Response, error)
    ListRepoPullRequests(ctx context.Context, owner, repo string,
        opts gitea.ListPullRequestsOptions) ([]*gitea.PullRequest, *gitea.Response, error)
}

type TestPoolRunner interface {
    RunWithCommandAndStdin(ctx context.Context, payload model.TaskPayload,
        cmd []string, stdinData []byte) (*worker.ExecutionResult, error)
}

type TestConfigProvider interface {
    ResolveTestGenConfig(repoFullName string) config.TestGenOverride
    GetClaudeModel() string
    GetClaudeEffort() string
}

type Service struct {
    gitea    RepoClient
    prClient PRClient        // M3.5 复用；M4.1 签名占位、M4.2 使用
    pool     TestPoolRunner
    cfgProv  TestConfigProvider
    logger   *slog.Logger
}

// ServiceOption: WithServiceLogger / WithPRClient
// NewService(gitea, pool, cfgProv, opts...) 对 gitea/pool/cfgProv panic 校验（同 review/fix 惯例）
```

**`Execute` 骨架**：

```
1. 前置校验（validateModule）
2. resolveBaseRef(payload)：若 payload.BaseRef 空 → Gitea GetRepo → default_branch
3. resolveFramework(payload, cfg)：探测 module/pom.xml 或 package.json → Framework 枚举
4. buildPrompt(framework, ctx)：Java 或 Vue + 公共段
5. pool.RunWithCommandAndStdin
6. parseResult(output)：外层 CLI 信封 → TestGenOutput → validateSuccessfulTestGenOutput
7. if !InfoSufficient / !Success → 对应 sentinel
8. createTestPR（M4.1 占位 return nil,"",nil；M4.2 实装）
```

**`WriteDegraded`（M4.1 实现）**：

```go
func (s *Service) WriteDegraded(ctx context.Context, payload model.TaskPayload,
    result *TestGenResult) error {
    if result == nil { return nil }
    s.logger.ErrorContext(ctx, "gen_tests 降级回写（M4.1 no-op，仅记录原始输出）",
        "repo", payload.RepoFullName,
        "module", payload.Module,
        "raw_output_len", len(result.RawOutput),
        "has_parse_error", result.ParseError != nil,
    )
    return nil
}
```

### 4.3 `TestGenOutput` 完整 schema

```go
type TestGenOutput struct {
    Success            bool            `json:"success"`
    InfoSufficient     bool            `json:"info_sufficient"`
    MissingInfo        []string        `json:"missing_info,omitempty"`
    Analysis           *GapAnalysis    `json:"analysis,omitempty"`
    TestStrategy       string          `json:"test_strategy,omitempty"`
    GeneratedFiles     []GeneratedFile `json:"generated_files,omitempty"`
    CommittedFiles     []string        `json:"committed_files,omitempty"` // 已 commit + 验证通过
    SkippedTargets     []SkippedTarget `json:"skipped_targets,omitempty"`
    TestResults        *TestResults    `json:"test_results,omitempty"`
    VerificationPassed bool            `json:"verification_passed"`
    BranchName         string          `json:"branch_name,omitempty"`
    CommitSHA          string          `json:"commit_sha,omitempty"` // 分支 head（最后 push 的）
    FailureReason      string          `json:"failure_reason,omitempty"`
    RetryRounds        int             `json:"retry_rounds,omitempty"`
}

type GapAnalysis struct {
    UntestedModules []UntestedModule      `json:"untested_modules"`
    ExistingTests   []ExistingTestSummary `json:"existing_tests,omitempty"`
    ExistingStyle   string                `json:"existing_style"`
    PriorityNotes   string                `json:"priority_notes"`
}

type UntestedModule struct {
    Path     string `json:"path"`
    Kind     string `json:"kind"`     // service/controller/util/component/composable/store
    Priority string `json:"priority"` // high/medium/low
    Reason   string `json:"reason"`
}

type ExistingTestSummary struct {
    TestFile    string   `json:"test_file"`
    TargetFiles []string `json:"target_files"`
    Framework   string   `json:"framework"`
}

type GeneratedFile struct {
    Path        string   `json:"path"`
    Operation   string   `json:"operation"`   // "create" | "append"
    Description string   `json:"description"`
    Framework   string   `json:"framework"`   // junit5 / vitest
    TargetFiles []string `json:"target_files"`
    TestCount   int      `json:"test_count"`
}

type SkippedTarget struct {
    Path   string `json:"path"`
    Reason string `json:"reason"` // time_budget_exhausted / token_budget_exhausted / environment_issue / ...
}

type TestResults struct {
    Framework  string `json:"framework,omitempty"`
    Passed     int    `json:"passed"`
    Failed     int    `json:"failed"`
    Skipped    int    `json:"skipped"`
    AllPassed  bool   `json:"all_passed"`
    DurationMs int64  `json:"duration_ms,omitempty"`
}
```

**`validateSuccessfulTestGenOutput` 不变量**（`Success=true` 必须全部成立）：

- `InfoSufficient=true`
- `VerificationPassed=true` ∧ `TestResults.AllPassed=true` ∧ `TestResults.Failed=0`
- `BranchName != ""` ∧ `CommitSHA != ""`
- `len(CommittedFiles) > 0`，且 `CommittedFiles ⊆ GeneratedFiles.Path`
- `GeneratedFile.Operation ∈ {"create", "append"}`；`Operation="append" ⇒ len(TargetFiles) > 0`

`Success=false` 且 `len(CommittedFiles) > 0` → 视为"部分交付"，`FailureReason` 必填。

### 4.4 Prompt 结构

**公共段常量**（顺序即 prompt 顺序）：

| 常量 | 内容要点 |
|------|----------|
| `writePermissionNotice` | gen_tests 需要写权限；`allowedTools` 含 bash/edit/write |
| `testGenPreamble` | "你是资深测试工程师，目标是补全测试缺口" |
| `existingTestsInstruction` | **扫描 `**/*Test.java`、`**/*.{spec,test}.{ts,js}` 清单 → 吸收风格 → 列出 `existing_tests`** |
| `gapAnalysisInstruction` | 识别无测试覆盖的模块，按"公共 API > 复杂业务逻辑 > 工具类"排序 |
| `noOverwriteInstruction` | **`CRITICAL: NEVER overwrite an existing test file. Either skip that target, or append (operation="append") preserving all existing content. Each GeneratedFile must have operation="create" (new file) or "append" (added to existing).`** |
| `incrementalCommitInstruction` | **每完成一个测试文件的生成 + 本地验证，立即 `git add <file> && git commit -m 'test: <target>'`。单次 commit 粒度 = 单个测试文件。所有目标处理完、全部通过后，才 `git push origin HEAD`。容器被 kill 时未 push 的工作将丢失。** |
| `budgetAwareInstruction` | **token / 时间预算接近上限时，主动在 `skipped_targets` 中记录剩余目标并提前退出，保证已完成的 commit 能 push。** |
| `verificationInstruction` | 每 commit 前在本地运行对应测试；失败时最多修正 `{max_retry_rounds}` 轮；仍不过则回滚该文件并加入 `skipped_targets` |
| `outputJSONSchemaInstruction` | 完整 JSON schema 硬编码描述（同 `fix.analysisJSONSchemaInstruction` 风格） |

**Java 特有段**（`buildJavaPrompt`）：

- `javaTestingInstruction`：JUnit 5 + Mockito 约定；Service / Controller / 工具类的 Mock 策略；断言风格
- `javaVerificationCmd`：`mvn -pl <module> test -Dtest=<ClassName>` 单文件验证命令

**Vue 特有段**（`buildVuePrompt`）：

- `vueTestingInstruction`：Vitest + Vue Test Utils；组件 / Composable / Store / 工具函数差异
- `vueVerificationCmd`：`npx vitest run <file>` 或 `pnpm vitest run <file>` 单文件验证命令

**`resolveFramework(payload, cfg)` 规则**：

1. `cfg.TestFramework` 显式设定 → 直接用
2. 扫 `module/pom.xml` 存在 → Java
3. 扫 `module/package.json` 存在且含 `vitest` / `vue` 依赖 → Vue
4. 两者都在 `module` 根下 → 返回 `ErrAmbiguousFramework`，要求用户缩窄 `--module` 或设 `test_framework`
5. 都不在 → 回退仓库根目录同样探测；仍无 → `ErrNoFrameworkDetected`

**b) `EnqueueManualGenTests`**（`internal/queue/enqueue.go`）：

```go
func (h *EnqueueHandler) EnqueueManualGenTests(ctx context.Context,
    payload model.TaskPayload, triggeredBy string) (string, error) {
    payload.TaskType = model.TaskTypeGenTests
    payload.DeliveryID = generateManualDeliveryID()

    activeTasks := h.listActiveGenTestsTasks(ctx, payload.RepoFullName, payload.Module)

    record := &model.TaskRecord{
        TaskType:     model.TaskTypeGenTests,
        Priority:     model.PriorityLow, // 对齐已有 PriorityLow=测试生成
        RepoFullName: payload.RepoFullName,
        DeliveryID:   payload.DeliveryID,
        TriggeredBy:  triggeredBy,
    }
    if err := h.enqueueTask(ctx, payload, record); err != nil { return "", err }
    h.cancelTasks(ctx, activeTasks)
    return record.ID, nil
}
```

```go
FindActiveGenTestsTasks(ctx context.Context, repoFullName, module string) ([]*model.TaskRecord, error)
```

### 5.1 手动触发 happy path（M4.2 激活后全链路）

```
[Client]                [Server API]            [Queue]                [asynq Worker]          [Container]
dtw gen-tests
  --repo o/r             │                       │                       │                       │
  --module srv          ─┼→ POST /api/v1/        │                       │                       │

                         │                       │   resolveFramework                            │

                         │                       │    └─ pool.RunWithCmdStdin ─────────────────→ [launched]
                         │                       │                                                │ 容器内 Claude：
                         │                       │                                                │   1. 扫既有测试
                         │                       │                                                │   2. GapAnalysis
                         │                       │                                                │   3. FOR target:
                         │                       │                                                │      - 生成测试
                         │                       │                                                │      - 本地 mvn/vitest
                         │                       │                                                │      - 通过 → git commit
                         │                       │                                                │   4. 全部完成
```

**M4.1 实际行为**：上图中 "TestService.Execute" 被跳过；Processor 因 `testService=nil` 走 default → `pool.Run(payload)`，容器里跑 `worker/container.go` 中既存的基础 gen_tests prompt（无 PR、无 TestGenOutput 解析）。用户看到 `status=succeeded`，`record.Result` 为 Claude 自由文本。这是**刻意的过渡行为**，M4.2 加 `WithTestService(svc)` 一行即切换到新路径。

### 5.2 Processor 路由决策表（M4.1 实际落地）

| payload.TaskType | reviewService 已注入 | fixService 已注入 | testService 已注入（M4.2） | 实际分支 |
|------------------|---------------------|-------------------|---------------------------|----------|
| review_pr | ✅ | — | — | `reviewService.Execute` |
| analyze_issue | — | ✅ | — | `fixService.Execute` |
| fix_issue | — | ✅ | — | `fixService.Execute` |
| **gen_tests（M4.1）** | — | — | ❌ | **`pool.Run` default（死路径）** |
| gen_tests（M4.2+） | — | — | ✅ | `testService.Execute` |
| gen_daily_report | — | — | — | `pool.Run` default |

### 5.3 Cancel-and-Replace 时序

```
T0  dtw gen-tests ... --module srv    → task1 created, status=queued
T1  asynq pulls task1 → Processor     → task1 status=running
T2  容器 launched，Claude 开始扫既有测试
T3  dtw gen-tests ... --module srv（第二次!）
     ↓
    EnqueueManualGenTests
    ├─ listActiveGenTestsTasks(repo, "srv") → [task1: running]
    ├─ enqueueTask(task2) → status=queued
    └─ cancelTasks([task1])
         ├─ canceller.CancelProcessing(task1.AsynqID)
         │    └─ asynq 触发 ctx cancel → pool 停止容器
         │       容器被 kill ← prompt "最后一刻才 push" 保证分支未创建
         │       task1 无副作用
         └─ SQLite UpdateTask(task1, cancelled, "被同一 gen_tests 的新任务取代")
T4  task2 被 asynq pull，正常执行
```

**关键性质**：即使 task1 在 T2-T3 期间已在容器里 commit 了多个文件，被 kill 时这些 commit 仍在本地临时分支里——容器销毁后**全部丢弃，无远程副作用**。task2 启动时是干净状态。

### 5.4 容器内 Claude 内部循环（由 M4.1 prompt 塑形，M4.2 真实运行）

```
[Container init] entrypoint.sh：clone repo + checkout <ref> + 创建工作分支 auto-test/<module>-<ts>

[Claude Agent loop]
  Step 1: 扫描既有测试
    ├─ find **/*Test.java **/*.{spec,test}.{ts,js}
    ├─ 读取各文件首部 → 识别 framework / 断言风格 / Mock 约定
    └─ 填充 GapAnalysis.ExistingTests + ExistingStyle

  Step 2: 缺口分析
    ├─ 列源代码 → 减去 ExistingTests.TargetFiles → untested set
    ├─ 按优先级（公共 API > 复杂业务 > 工具类）排序
    └─ 填充 GapAnalysis.UntestedModules → target_queue

  Step 3: 按目标循环生成
    FOR target IN target_queue:
      ├─ 判断 Operation：已有测试文件 & 可扩展 → "append"；否则 → "create"
      ├─ 生成测试代码（write / edit 文件）
      ├─ 本地单文件验证循环（最多 max_retry_rounds 轮）：
      │    ├─ 执行：mvn -pl <mod> test -Dtest=<cls> | npx vitest run <file>
      │    ├─ 通过 → git add <file> && git commit → append 到 committed_files，break
      │    └─ 失败 → 读错误 → 修正测试 → continue
      ├─ retry 耗尽 → git checkout -- <file>（回滚）→ 加入 skipped_targets
      └─ 预算检测（time / token 接近上限）→ 剩余 target 全部入 skipped_targets，跳出

  Step 4: 最后一刻 push
    IF len(committed_files) > 0:
      git push origin HEAD  ← 唯一的网络写操作
      VerificationPassed = true
    ELSE:
      Success = false, FailureReason = "无可交付测试"

  Step 5: 输出 TestGenOutput JSON 到 stdout
```

**"最后一刻 push" 的正确性**：

1. 容器被 kill（超时 / Cancel / 网络中断）→ 分支未创建 → 无残留 PR、无半成品
2. `len(committed_files) > 0` 但 push 失败（网络瞬断）→ 整任务失败进入 asynq 重试 → 下次容器启动从头开始
3. M4.2 若要加"抢救未 push 工作"能力，只需给容器挂载持久卷；prompt 无需改动

## 6. 错误处理

### 6.1 Sentinel 错误（`internal/test/errors.go`）

```go
var (
    // 参数层：用户可修正 → 上层 SkipRetry
    ErrInvalidModule       = errors.New("module 路径不合法")
    ErrModuleNotFound      = errors.New("module 子路径不存在于仓库")
    ErrInvalidRef          = errors.New("--ref 指定但 Gitea 无法解析")
    ErrAmbiguousFramework  = errors.New("module 下同时存在 pom.xml 与 package.json，请缩窄 --module 或指定 --framework")
    ErrNoFrameworkDetected = errors.New("未检测到 JUnit / Vitest 配置，请设置 test_gen.test_framework")

    // 业务层：Claude 判定或结果失败 → SkipRetry
    ErrInfoInsufficient    = errors.New("Claude 判定信息不足，无法生成测试")
    ErrTestGenFailed       = errors.New("测试生成执行失败（Success=false）")

    // 结果解析层：结构 / 不变量失败 → 允许重试，耗尽后降级
    ErrTestGenParseFailure = errors.New("TestGenOutput 解析失败")
)
```

### 6.2 `parseResult` + 不变量校验

外层 CLI 信封 → 内层 `TestGenOutput` → `validateSuccessfulTestGenOutput`，与 fix 包完全对称。

**不变量校验完整清单**（`Success=true` 必须全部成立）：

| 校验项 | 失败消息 |
|--------|----------|
| `InfoSufficient=true` | `"Success=true 但 InfoSufficient=false"` |
| `VerificationPassed=true` | `"Success=true 但 VerificationPassed=false"` |
| `TestResults != nil && AllPassed && Failed == 0` | `"测试未全部通过 all_passed=%v failed=%d"` |
| `BranchName != ""` | `"branch_name 为空"` |
| `CommitSHA != ""` | `"commit_sha 为空"` |
| `len(CommittedFiles) > 0` | `"committed_files 为空"` |
| `CommittedFiles ⊆ GeneratedFiles.Path` | `"committed_files 包含未在 generated_files 声明的路径 %q"` |
| `GeneratedFile.Operation ∈ {"create","append"}` | `"GeneratedFile.Operation 非法: %q"` |
| `Operation="append" ⇒ len(TargetFiles) > 0` | `"append 操作必须声明 target_files"` |

### 6.3 防 prompt-injection

**约束**（照搬 M3.5 fix 的决定）：`ParseError` 详情**仅走日志**，不进返回 `error` 的消息，避免被后续写入 `record.Error`，再进入通知 / PR 描述。

### 6.4 重试决策表

| 触发点 | sentinel / 情况 | 重试? | M4.1 降级路径 | M4.2 预期路径 |
|--------|----------------|-------|--------------|---------------|
| 参数非法 | `ErrInvalidModule` / `ErrModuleNotFound` / `ErrInvalidRef` / `ErrAmbiguousFramework` / `ErrNoFrameworkDetected` | SkipRetry | n/a | n/a — 直接 failed |
| 容器启动 / 基础设施 | 非 sentinel 的普通 error | ✅ 继续重试 | n/a | n/a |
| 容器非零退出码 | `execResult.ExitCode != 0` | `shouldRetry(ctx)` | n/a | n/a |
| Claude 判信息不足 | `ErrInfoInsufficient` | SkipRetry | **no-op**（仅日志） | 飞书通知 + CLI job 状态附 `MissingInfo` |
| Claude 返回 Success=false | `ErrTestGenFailed` | SkipRetry | **no-op** | 同上，附 `FailureReason` + `SkippedTargets` |
| JSON / 不变量解析失败 | `ErrTestGenParseFailure` | ✅ 重试，耗尽后降级 | `WriteDegraded` **no-op** | 飞书通知 metadata 含 `RawOutput[:2048]` snippet |
| PR 创建失败（push 已成功） | 普通 error | ✅ 重试（`findExistingTestPR` 幂等保护） | n/a | n/a |

**M4.1 与重试表的交互**：M4.1 的 Processor 仍走 default，**此表在 M4.1 是"单测可覆盖的规格"而非"运行时行为"**——`service_test.go` 验证 "Execute 返回正确 sentinel"，M4.2 加 Processor 分支后表才生效。与 M3.1 期间 fix.Service 的 sentinel 定义状态一致。

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| prompt 公共段过长导致 Claude token 消耗大 | 公共段常量化，按需拼接；Java/Vue 特有段分离，不相互污染 |
| Cancel-and-Replace 在容器运行中触发，`git push` 已开始 → 半成品分支 | prompt 约束"最后一刻 push"；push 是单次原子操作，时间窗极小；若仍发生，M4.2 的幂等 `findExistingTestPR` 兜底 |
| 既有测试风格混乱（多种 Mock 框架） | prompt 指令"吸收主流风格"；模棱两可时在 `ExistingStyle` 字段透明报告 |
| 框架探测误判（混合仓中 `module` 边界不清） | `ErrAmbiguousFramework` 明确要求用户缩窄 `--module` 或设 `test_framework`；不静默猜测 |
| M4.1 的过渡期内 gen_tests 走旧 prompt 产出无用结果 | 文档明示过渡行为；用户手动触发只在 M4.2 后才有意义；M4.1 的手动入口主要用于验证入队链路 |
| `test_gen_results` 表在 M4.1 未建，任务结果仅留 `tasks.result` 字段 | 接受；M4.2 建表时一次性迁移既有 `tasks.result` 中 gen_tests 数据 |
