# M2.3 评审结果解析与回写 — 设计文档

> 日期：2026-03-26
> 状态：待实施
> 依赖：M2.1（PR 评审上下文提取）、M2.2（评审 Prompt 工程）

## 1. 目标

将 `review.Service.Execute()` 返回的结构化评审结果（`ReviewOutput`）转化为 Gitea PR Review API 调用，实现：

- 逐行 comment：将 issues 映射到 diff 行号，通过 Gitea PR Review API 添加行级评论
- 整体 summary：生成格式化的总结评论并作为 Review body 发布
- 自动 Request Changes：当存在 CRITICAL 或 ERROR 时自动触发
- 评审结果持久化：新建 `review_results` 表存储结构化评审记录，便于后续统计分析

## 5. Diff 解析与行号映射

### 5.1 DiffMap 数据结构

```go
// DiffMap 存储 PR diff 中每个文件的行号映射信息
type DiffMap struct {
    files map[string]*FileDiff
}

// FileDiff 单个文件的 diff 信息
type FileDiff struct {
    NewName  string      // diff 后的文件名（rename 场景）
    IsDelete bool        // 是否为删除文件（+++ /dev/null）
    Hunks    []HunkRange // 所有 hunk 的行号范围
}

// HunkRange 一个 hunk 块的信息
type HunkRange struct {
    NewStart   int            // 新文件起始行号（@@ 中的 +c）
    NewCount   int            // 新文件行数（@@ 中的 d）
    DiffOffset int            // 该 hunk 在文件 diff 块中的起始行偏移（从 diff --git 行算起，0-based）
    Lines      []DiffLine     // hunk 内逐行解析结果
}

// DiffLine hunk 内单行的解析结果
type DiffLine struct {
    NewLineNum int  // 新文件中的行号（- 行为 0）
    DiffPos    int  // 该行在文件 diff 块中的位置（从 1 开始）
    Op         byte // '+', '-', ' '（context）
}
```

### 5.2 解析算法

输入为 `GET /pulls/{index}.diff` 返回的标准 unified diff 格式：

```
diff --git a/path/to/file.go b/path/to/file.go    <-- 文件 header 行 1
--- a/path/to/file.go                               <-- 文件 header 行 2
+++ b/path/to/file.go                               <-- 文件 header 行 3
@@ -10,5 +12,7 @@ func something() {                <-- hunk header
 context line                                        <-- context (空格开头)
+added line                                          <-- addition
-removed line                                        <-- deletion
```

解析步骤：

1. 按 `diff --git` 分割为文件块
2. 从 `+++ b/path` 提取新文件名（`+++ /dev/null` 标记为 `IsDelete=true`）
3. 解析每个 `@@ -a,b +c,d @@` hunk header，提取 `NewStart=c, NewCount=d`
4. 记录 hunk header 在文件 diff 块中的行偏移（`DiffOffset`，从 `diff --git` 行算起）
5. 在 hunk 内逐行追踪，构建 `DiffLine` 列表：
   - `+` 行和 ` ` 行：递增 newLineNum 计数器，记录 `DiffPos`
   - `-` 行：不递增 newLineNum，但递增 `DiffPos`

### 5.3 MapLine 行号映射

```go
// MapLine 将源文件绝对行号映射为 Gitea API 所需的位置值
// 返回 (position, true) 表示映射成功
// 返回 (0, false) 表示该行不在 diff 范围内
func (dm *DiffMap) MapLine(file string, line int) (position int, ok bool)
```

**`new_position` 语义说明：** Gitea API 的 `new_position` 字段有两种可能的语义（见 5.5 API 验证清单）。实现时需根据验证结果选择对应算法：

- **语义 A（新文件行号）**：`new_position` 直接等于新文件中的绝对行号。此时 MapLine 只需检查 `line` 是否落在某个 hunk 的 `[NewStart, NewStart+NewCount)` 范围内，命中则直接返回 `line`。
- **语义 B（diff 物理行偏移）**：`new_position` 是从文件 diff 块开头算起的行偏移（类似 GitHub API 的 position）。此时 MapLine 需在 `HunkRange.Lines` 中查找 `NewLineNum == line` 的 `DiffLine`，返回其 `DiffPos`。

两种语义的 MapLine 实现均已在数据结构中预留支持（`DiffLine.NewLineNum` 和 `DiffLine.DiffPos`）。

### 5.4 API 验证清单（实施前必须完成）

在实施 `MapLine` 前，需对目标 Gitea 实例做以下验证：

1. 创建一个测试 PR，手动调用 `POST /api/v1/repos/{owner}/{repo}/pulls/{index}/reviews`
2. 分别传入 `new_position` = 新文件行号 和 `new_position` = diff 物理行偏移
3. 观察 Gitea UI 中行级评论的定位是否正确
4. 记录目标 Gitea 版本号和验证结果
5. 将验证结果更新到本文档

> **注意**：不同 Gitea 版本可能行为不同。GitHub API 使用 diff 物理行偏移（语义 B），Gitea 可能与 GitHub 一致也可能使用语义 A。

### 6.3 Verdict 映射

```go
// mapVerdict 将 Claude 返回的 verdict 映射为 Gitea ReviewStateType。
// 安全网：即使 Claude 返回 approve/comment，只要 issues 中存在 CRITICAL 或 ERROR，
// 强制覆盖为 REQUEST_CHANGES（对应 ROADMAP 要求"当存在 CRITICAL 或 ERROR 时触发"）。
func mapVerdict(verdict VerdictType, issues []ReviewIssue, hasParseError bool) gitea.ReviewStateType {
    if hasParseError {
        return gitea.ReviewStateComment  // 降级场景固定为 COMMENT
    }
    // 安全网：检查是否存在 CRITICAL 或 ERROR 级别 issue
    for _, issue := range issues {
        sev := strings.ToUpper(issue.Severity)
        if sev == "CRITICAL" || sev == "ERROR" {
            return gitea.ReviewStateRequestChanges
        }
    }
    switch verdict {
    case VerdictApprove:
        return gitea.ReviewStateApproved
    case VerdictRequestChanges:
        return gitea.ReviewStateRequestChanges
    case VerdictComment:
        return gitea.ReviewStateComment
    default:
        return gitea.ReviewStateComment  // 未知 verdict 安全降级
    }
}
```

### 6.4 格式化约束

- Body 总长度上限 60000 字符（留安全余量，Gitea 上限约 64KB）
- RawOutput 降级时如超长则截断并附加 `...(已截断，完整输出请查看任务日志)`
- 统计表格中数量为 0 的 severity 行不显示
- UnmappedIssues 按 severity 降序排列（CRITICAL 在最前）
- 行级评论 body 格式：`**{severity}** | {category}\n\n{message}\n\n> 建议：{suggestion}`
- 单条行级评论 body 上限 8000 字符（安全截断，避免超长 suggestion）

### 6.5 Markdown 特殊字符处理

Claude 输出的 `message` 和 `suggestion` 字段可能包含 Markdown 特殊字符（`|`、`#`、`` ` ``、`*` 等），需要在以下场景做处理：

- **统计表格中**：对嵌入表格单元格的文本，转义 `|` 为 `\|`（`formatter.go` 中提供 `escapeTableCell(s string) string` 辅助函数）
- **行级评论 body 中**：代码片段用 code fence 包裹，其余内容不转义（保留 Claude 的 Markdown 格式化意图）
- **UnmappedIssues 列表中**：message 截断到 200 字符并转义 `|`，避免破坏列表格式

## 8. 错误处理与降级

### 8.1 回写阶段错误分级

| 错误场景 | 处理策略 | 任务状态 |
|---------|---------|---------|
| GetPullRequestDiff 失败 | 跳过行级评论，所有 issues 降级到 body，仍执行 CreatePullReview | succeeded + WritebackError |
| diff 解析异常 | 同上 | succeeded + WritebackError |
| CreatePullReview API 失败 | 回写失败，WritebackError 记录完整错误 | succeeded + WritebackError |
| SaveReviewResult 失败 | 仅日志告警，不影响回写结果 | succeeded + 日志 WARN |
| PR 在执行期间被关闭/合并 | CreatePullReview 返回 4xx，WritebackError 记录 | succeeded + WritebackError |
| Gitea 返回 429 (Rate Limit) | WritebackError 记录，不重试（当前 scope 不含回退重试逻辑） | succeeded + WritebackError |
| RawOutput 超 60KB | 截断 + 附加提示 | succeeded |

### 8.2 核心原则

回写失败不应导致任务失败。评审执行（Claude CLI 调用）是昂贵操作（约 30s + API 费用），回写只是结果投递。

### 8.3 降级链路

```
正常路径: ReviewOutput -> 行级评论 + summary -> APPROVED/REQUEST_CHANGES
    |
    +- diff 获取失败 -> 全部 issues 归入 summary body -> 仍按 verdict 设置 state
    |
    +- ParseError -> RawOutput 作为 body + 解析失败提示 -> COMMENT
    |
    +- 回写 API 失败 -> 降级到现有通知路径（sendCompletionNotification）
```

### 8.4 Processor 集成变更

**adaptReviewResult 修改后的伪代码：**

```go
func adaptReviewResult(r *review.ReviewResult) *worker.ExecutionResult {
    if r == nil {
        return nil
    }
    result := &worker.ExecutionResult{
        ExitCode: 0,
        Output:   r.RawOutput,
    }
    // 1. CLI 执行错误（已有）
    if r.CLIMeta != nil {
        result.Duration = r.CLIMeta.DurationMs
        if r.CLIMeta.IsError {
            result.ExitCode = 1
            result.Error = "Claude CLI 报告错误"
        }
    }
    // 2. JSON 解析错误（已有）
    if r.ParseError != nil && result.Error == "" {
        result.Error = r.ParseError.Error()
    }
    // 3. 回写错误（新增）——不改变 ExitCode，回写失败不等于任务失败
    if r.WritebackError != nil && result.Error == "" {
        result.Error = fmt.Sprintf("评审回写失败: %s", r.WritebackError)
    }
    return result
}
```

**通知策略变更：**

- 回写成功时：`sendCompletionNotification` 仍然发送，但内容简化为"PR #N 评审已发布"（而非跳过通知，确保向后兼容）
- 回写失败时：`sendCompletionNotification` 发送完整错误信息作为兜底
- 判断依据：`buildNotificationMessage` 检查 `record.Result` 中是否包含 `GiteaReviewID`（非 0 表示回写成功）

这样无需修改 `ProcessTask` 的调用逻辑和 `worker.ExecutionResult` 结构，仅调整 `buildNotificationMessage` 的消息内容。

- `Writer` 无状态，所有数据通过参数传入，天然并发安全
- `DiffMap` 在 `Writer.Write()` 内部构建，生命周期限定在单次调用中

| 单文件多 hunk | 多个 hunk 的 position 累加计算 |
| 多文件 diff | 文件名提取、按文件查询 |
| 新增文件（/dev/null -> b/path） | 无 old file，全部为 new_position |
| 删除文件（a/path -> /dev/null） | 标记为已删除，MapLine 返回 false |
| 文件重命名 | 新旧文件名都能查到 |
| 行号在 hunk 范围内 | MapLine 返回 (position, true) |
| 行号不在 hunk 范围 | MapLine 返回 (0, false) |
| 空 diff 文本 | 返回空 DiffMap，不 panic |
| 二进制文件 diff | 跳过，不影响其他文件 |

### 9.2 writeback.go（Writer）

| 用例 | 验证点 |
|------|--------|
| 正常路径：issues 全部映射成功 | CreatePullReview 调用参数正确，Comments 数量匹配 |
| 部分映射失败 | 映射成功的进 Comments，失败的进 Body |
| 全部映射失败 | Comments 为空，所有 issues 在 Body 中 |
| ParseError 降级（方案 D） | Body 包含 RawOutput + 降级提示，State=COMMENT |
| Verdict 映射（无 CRITICAL/ERROR） | approve->APPROVED, comment->COMMENT |
| Verdict 安全网 | Claude 返回 approve 但 issues 含 CRITICAL -> 强制 REQUEST_CHANGES |
| diff 获取失败降级 | 所有 issues 归入 Body，仍调用 CreatePullReview |
| CreatePullReview API 失败 | WritebackError 非 nil，不 panic |
| SaveReviewResult 失败 | WritebackError 为 nil（回写已成功），仅日志 |
| Writer 为 nil | Execute 跳过回写，行为与 M2.2 一致 |
