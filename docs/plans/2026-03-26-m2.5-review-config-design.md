## 1. 目标

激活 M2.1/M2.2 预留但未实现的评审配置字段，让团队能够：

1. **按仓库控制评审开关**：某些仓库不需要自动评审
2. **按维度裁剪评审范围**：关闭不关心的评审维度（如工具类仓库关闭 architecture）
3. **按严重程度过滤噪音**：低级别 issue 不回写到 Gitea，减少评论噪音
4. **按文件模式忽略评审**：生成代码、文档、vendor 等文件不纳入评审

### 不在本轮范围

- **增量评审**：仅评审新变更文件（延后至后续迭代，当前仍为全量评审）
- **每维度独立严重程度阈值**：当前使用全局统一阈值，未来可按需扩展
- **评审者身份代码实现**：Bot 账户配置仅提供文档指导（见 Section 10）

## 2. 配置结构

### 2.1 现有字段激活

以下字段已在 `ReviewOverride` 中定义，本轮激活其执行逻辑：

| 字段 | 类型 | 当前状态 | 本轮改动 |
|------|------|----------|----------|
| `Enabled` | `*bool` | 定义未使用 | Processor 层跳过禁用仓库的评审任务 |
| `Severity` | `string` | 定义未使用 | 回写层按阈值过滤 issue |
| `Dimensions` | `[]string` | 定义未使用 | Prompt 层动态组装评审维度指令 |
| `IgnorePatterns` | `[]string` | 定义未使用 | Prompt 层 + 回写层双重文件过滤 |

### 2.2 配置示例

```yaml
# 全局评审配置
review:
  enabled: true
  severity: "warning"              # 只回写 WARNING 及以上的 issue
  dimensions:                      # 启用的评审维度（默认全部四个）
    - security
    - logic
    - architecture
    # style 被关闭
  ignore_patterns:                 # 忽略的文件模式（doublestar glob）
    - "**/*.md"
    - "docs/**"
    - "**/vendor/**"
    - "*.pb.go"
    - "*.generated.go"

# 仓库级覆盖
repos:
  - name: "acme/utils"
    review:
      dimensions:                  # 工具库不需要架构评审
        - security
        - logic
        - style
      ignore_patterns:
        - "testdata/**"

  - name: "acme/legacy"
    review:
      enabled: false               # 遗留项目禁用自动评审
```

### 2.3 合并规则

沿用 M2.1 既有的 `ResolveReviewConfig` 合并策略，无需修改：

- `Enabled`：`*bool` 指针语义，nil = 未设置（继承全局），false/true = 覆盖
- `Severity`：非空字符串覆盖全局值
- `Dimensions`：nil = 未覆盖（用全局），non-nil 非空 = 覆盖全局。**空切片 `[]` 语义**：`resolveConfig()` 中 `len(cfg.Dimensions) == 0` 统一视为"使用默认四维度"，即显式写 `dimensions: []` 等价于未设置，不会产生"零维度"的评审
- `IgnorePatterns`：nil = 未覆盖（用全局），non-nil（含空切片 `[]`）= 覆盖全局（可用于显式清空全局 patterns）

## 3. 配置校验增强

在 `internal/config/config.go` 的 `Validate()` 方法中新增以下校验规则：

### 3.1 维度白名单

```go
var validDimensions = map[string]bool{
    "security": true, "logic": true, "architecture": true, "style": true,
}
```

- 全局 `review.dimensions` 和仓库级 `repos[].review.dimensions` 中的每个元素必须在白名单中
- 空列表合法（表示使用默认四维度）
- 非法值返回 `ValidationError`，消息示例：`review.dimensions 包含无效维度 "perf"，有效值: security, logic, architecture, style`

### 3.2 Severity 白名单

```go
var validSeverities = map[string]bool{
    "critical": true, "error": true, "warning": true, "info": true,
}
```

- 非空时必须在白名单中（大小写不敏感，存储统一为小写）
- 空字符串合法（等价于 `info`，即不过滤）
- 非法值返回 `ValidationError`

### 3.3 IgnorePatterns 语法校验

```go
for _, pattern := range review.IgnorePatterns {
    if !doublestar.ValidatePattern(pattern) {
        // 返回 ValidationError
    }
}
```

- 启动时校验每个 pattern 是否为合法 doublestar 语法
- 非法模式（如未闭合的 `[`）在启动阶段拦截，避免运行时报错

## 4. 过滤逻辑核心（新建 `internal/review/filter.go`）

集中管理所有过滤逻辑，纯函数设计，不依赖外部接口。

### 4.1 Severity 类型

```go
// SeverityLevel 表示严重程度的有序类型
type SeverityLevel int

const (
    SeverityInfo     SeverityLevel = iota // 最低
    SeverityWarning
    SeverityError
    SeverityCritical                      // 最高
)

// ParseSeverity 将字符串转为 SeverityLevel
// 无效值降级为 SeverityInfo（不过滤），fail-open
func ParseSeverity(s string) SeverityLevel

// AtLeast 判断当前级别是否 >= 阈值
func (l SeverityLevel) AtLeast(threshold SeverityLevel) bool
```

### 4.2 文件 Glob 匹配

```go
// MatchesIgnorePattern 检查文件路径是否匹配任一 ignore pattern
// 使用 doublestar.Match，支持 ** 递归匹配
// 路径使用 / 分隔（Gitea API 返回的 PR 文件路径格式）
func MatchesIgnorePattern(filePath string, patterns []string) bool
```

- 空 patterns 列表 → 不匹配任何文件（不过滤）
- `filePath` 为空字符串（项目级 issue，无关联文件）→ 不匹配任何 pattern（不被文件过滤）
- `doublestar.Match` 的 err 返回值在此处忽略（pattern 已在 Validate 阶段校验；热加载路径同样经过 Validate）

### 4.3 统一过滤入口

```go
// FilterResult 封装过滤结果
type FilterResult struct {
    Visible    []ReviewIssue // 过滤后可见的 issue
    Filtered   int           // 被过滤掉的总数
    BySeverity int           // 因 severity 阈值过滤的数量
    ByFile     int           // 因文件 glob 匹配过滤的数量
}

// FilterIssues 对 issue 列表应用 severity + 文件双重过滤
// 任一条件命中即过滤
func FilterIssues(issues []ReviewIssue, severityThreshold string, ignorePatterns []string) FilterResult
```

过滤优先级：先检查文件 glob（命中计入 ByFile），再检查 severity（命中计入 BySeverity）。
同时命中两个条件时计入 ByFile（文件忽略优先）。

## 5. 数据流：ReviewConfig 结构体扩展

当前 `ReviewConfig`（`prompt.go`）缺少 `Severity` 和 `IgnorePatterns` 字段，导致配置无法从 `resolveConfig` 传递到回写层。

### 5.1 ReviewConfig 新增字段

```go
type ReviewConfig struct {
    // ... 现有字段 ...
    Instructions       string
    RepoInstructions   string
    Dimensions         []string
    LargePRThreshold   int
    TechStack          []string
    CodeStandardsPaths []string

    // M2.5 新增
    Severity       string   // severity 过滤阈值（小写：critical/error/warning/info）
    IgnorePatterns []string // 文件忽略 glob 模式列表
}
```

### 5.2 resolveConfig 更新

在 `service.go` 的 `resolveConfig()` 中补充字段复制：

```go
cfg := ReviewConfig{
    // ... 现有字段复制 ...
    Severity:       override.Severity,
    IgnorePatterns: override.IgnorePatterns,
}
```

`Severity` 默认值为空字符串（等价于 `info`，不过滤）。
`IgnorePatterns` 默认值为 nil（不过滤任何文件）。

### 5.3 完整数据流

```
config.ReviewOverride          →  resolveConfig()
    .Severity                       →  ReviewConfig.Severity
    .IgnorePatterns                 →  ReviewConfig.IgnorePatterns
                                        ↓
                               service.Execute()
                                        ↓ (prompt 层使用 IgnorePatterns 过滤文件列表)
                                        ↓ (构造 WritebackInput 时填入两个字段)
                                        ↓
                               WritebackInput
                                   .SeverityThreshold  ← ReviewConfig.Severity
                                   .IgnorePatterns     ← ReviewConfig.IgnorePatterns
                                        ↓
                               writer.Write()
                                        ↓ (FilterIssues 使用两个字段过滤)
```

## 6. Prompt 层改动（`internal/review/prompt.go`）

### 5.1 文件过滤

修改 `buildPrompt()` 中的文件摘要构建逻辑：

1. 在调用 `formatFilesSummary()` 前，用 `MatchesIgnorePattern()` 过滤文件列表
2. 被过滤的文件不出现在 prompt 的"变更文件列表"中
3. 如果有被过滤的文件，在文件列表末尾追加：`（另有 N 个文件被配置忽略，不纳入评审范围）`
4. 变更行数统计排除被忽略文件（影响大 PR 检测阈值判断）

### 5.2 维度动态控制

将 `defaultReviewInstructions` 拆分为两部分：

1. **评审原则前言**（severity 定义、评审态度等）→ 保留为 `reviewPreamble` 常量，始终包含
2. **四个维度指令段** → 拆分为独立常量：

```go
// reviewPreamble 包含：严重程度定义（CRITICAL/ERROR/WARNING/INFO）、评审原则（理解意图、聚焦重点等）
const reviewPreamble = `...`

var dimensionInstructions = map[string]string{
    "security":     securityInstructions,     // 安全维度：注入、认证、敏感数据等
    "logic":        logicInstructions,         // 逻辑维度：边界、并发、错误处理等
    "architecture": architectureInstructions,  // 架构维度：职责划分、耦合、扩展性等
    "style":        styleInstructions,         // 风格维度：命名、一致性、可读性等
}
```

拆分切点：`defaultReviewInstructions` 当前结构为"评审原则 + 四维度指令"，按自然段落边界切分即可。

`buildPrompt()` 根据 `cfg.Dimensions` 动态组装：

1. 始终包含 `reviewPreamble`（severity 定义和评审原则）
2. 只拼接启用维度的指令文本，每段前标注维度名称
3. `jsonSchemaInstruction` 中 `category` 的可选值动态对应启用的维度

### 5.3 不受影响的部分

- 技术栈专项 prompt（Java / Vue）：独立于四维度体系，不受维度控制
- 代码规范文件集成（CodeStandardsPaths）：不受影响
- 大 PR 引导：不受影响，但行数统计基于过滤后的文件

## 6. 回写层改动（`internal/review/writeback.go`）

### 6.1 过滤插入点

在 `Write` 方法中，issue 到 diff 行号映射（`mapIssuesToComments`）完成后、生成 review body 前插入过滤：

```
现有流程：
  parseResult → getDiff → parseDiff → mapIssuesToComments → formatReviewBody → createReview

改为：
  parseResult → getDiff → parseDiff → mapIssuesToComments
    → 【FilterIssues】→ 【recalcVerdict】
    → formatReviewBody → createReview
```

### 6.1.1 过滤操作对象

`mapIssuesToComments` 返回两组数据：
- `mapped`：成功映射到 diff 行号的 `[]MapResult`（每个 MapResult 包含 `ReviewIssue` + diff 位置信息）
- `unmapped`：映射失败的 `[]ReviewIssue`（降级到 body 展示）

过滤需要分别作用于两组数据：

```go
// 1. 从 mapped 中提取 ReviewIssue，与 unmapped 合并为完整列表
allIssues := extractIssues(mapped, unmapped)

// 2. 统一过滤
filterResult := FilterIssues(allIssues, input.SeverityThreshold, input.IgnorePatterns)

// 3. 用 filterResult.Visible 重建两组数据
filteredMapped := rebuildMapped(mapped, filterResult.Visible)     // 保留 diff 位置信息
filteredUnmapped := rebuildUnmapped(unmapped, filterResult.Visible)
```

重建逻辑：遍历原始 mapped/unmapped，检查其 `ReviewIssue` 是否在 `Visible` 集合中（按 File+Line+Message 匹配）。这保证了 MapResult 的 diff 位置信息不丢失。

**关键约束**：`result.Review.Issues`（原始 ReviewOutput）**不可修改**，持久化层 `buildReviewRecord` 仍读取完整的原始 issue 列表存入数据库。

### 6.2 配置传递

`WritebackInput` 新增两个字段：

```go
type WritebackInput struct {
    // ... 现有字段 ...
    SeverityThreshold string   // resolved config 中的 severity 阈值
    IgnorePatterns    []string // resolved config 中的 ignore patterns
}
```

由 `service.go` 的 `Execute` 方法在构造 `WritebackInput` 时从 resolved config 填入。

### 6.3 Verdict 重算

```go
func recalcVerdict(visible []ReviewIssue, originalVerdict VerdictType) VerdictType
```

**与现有 `mapVerdict` 的关系**：`recalcVerdict` **替代** `mapVerdict` 中的安全网逻辑。调用链调整为：

```
现有：mapVerdict(review, parseErr) → 安全网 + 原始 verdict 映射
改为：mapVerdict(review, parseErr) → 原始 verdict 映射（仅做 VerdictType → Gitea state 转换）
      recalcVerdict(visible, mappedVerdict) → 基于过滤后 issues 的安全网重算
```

即 `mapVerdict` 不再做 CRITICAL/ERROR 安全网判断（因为它看到的是全量 issues），安全网逻辑移到 `recalcVerdict` 中基于 `visible` 执行。

重算逻辑：

1. **parse 失败的降级场景**：不受影响，固定 `COMMENT`（在过滤逻辑之前已返回）
2. **visible 中有 CRITICAL 或 ERROR** → `REQUEST_CHANGES`（安全网不变）
3. **visible 中无 CRITICAL/ERROR** → 使用 Claude 原始 verdict，但若原始 verdict 为 `REQUEST_CHANGES` 则降级为 `COMMENT`（没有高严重度 issue 不应打回）
4. **visible 为空** → `COMMENT`

## 7. Formatter 改动（`internal/review/formatter.go`）

### 7.1 FormatOptions 扩展

```go
type FormatOptions struct {
    // ... 现有字段 ...
    VisibleIssues      []ReviewIssue // M2.5: 过滤后的可见 issue 列表
    FilteredBySeverity int           // 因 severity 过滤的 issue 数
    FilteredByFile     int           // 因文件 glob 过滤的 issue 数
}
```

**关键设计**：`formatReviewBody` 使用 `VisibleIssues` 计算统计表格和 unmapped 列表，而**不再**从 `Review.Issues` 读取。`Review` 字段仍传入完整的 `ReviewOutput`（用于读取 `Summary` 和 `Verdict`），但 issue 统计来源切换到 `VisibleIssues`。

调用方（`writeback.go`）负责在调用 `formatReviewBody` 前将 `filterResult.Visible` 填入 `VisibleIssues`。未设置 `VisibleIssues`（nil）时降级为使用 `Review.Issues`（向后兼容，但正常路径不应触发）。

### 7.2 过滤提示文案

在 `formatReviewBody` 生成的 summary body 中，统计表格之后、元信息之前，当 `FilteredBySeverity + FilteredByFile > 0` 时插入一行：

```markdown
> 另有 3 个低于阈值(warning)的问题和 2 个被忽略文件的问题已按配置过滤
```

- 只按实际过滤原因拼接（可能只有其中一种）
- 仅显示一行，不展开详情

### 7.3 统计表格

统计表格只计算过滤后的 visible issues，与团队实际看到的行级评论数保持一致。

## 8. Processor 层改动（`internal/queue/processor.go`）

### 8.1 新增 ConfigProvider 依赖

当前 `Processor` 结构体没有配置查询能力，需要新增窄接口：

```go
// ReviewEnabledChecker 是 Processor 层的窄接口（ISP）
// 仅暴露 Enabled 检查所需的最小能力，避免 Processor 依赖完整的 review.ConfigProvider
type ReviewEnabledChecker interface {
    IsReviewEnabled(repoFullName string) bool
}
```

`Processor` 新增可选字段：

```go
type Processor struct {
    // ... 现有字段 ...
    reviewEnabledChecker ReviewEnabledChecker // 可选，nil 时默认启用（向后兼容）
}
```

注入方式：新增 `ProcessorOption`（沿用现有 `WithReviewExecutor` 模式）：

```go
func WithReviewEnabledChecker(c ReviewEnabledChecker) ProcessorOption {
    return func(p *Processor) { p.reviewEnabledChecker = c }
}
```

### 8.2 serve.go 适配

在 `internal/cmd/serve.go` 中，现有 `configAdapter` 已实现 `review.ConfigProvider`。
新增一个方法使其同时满足 `ReviewEnabledChecker` 接口：

```go
func (a *configAdapter) IsReviewEnabled(repoFullName string) bool {
    override := a.ResolveReviewConfig(repoFullName)
    return override.Enabled == nil || *override.Enabled
}
```

构造 Processor 时注入：

```go
proc := queue.NewProcessor(pool, db, notifier,
    queue.WithReviewExecutor(reviewSvc),
    queue.WithReviewEnabledChecker(cfgAdapter), // M2.5 新增
)
```

### 8.3 Enabled 检查逻辑

在 `ProcessTask` 方法中，`review_pr` 任务分发到 `review.Service.Execute` 之前：

```go
if s.reviewEnabledChecker != nil && !s.reviewEnabledChecker.IsReviewEnabled(repoFullName) {
    logger.InfoContext(ctx, "评审已禁用，跳过任务", "repo", repoFullName)
    return nil // 任务标记为 succeeded，不是 failed
}
```

设计要点：

- `reviewEnabledChecker == nil` → 默认启用（向后兼容，不影响现有仓库和测试）
- `IsReviewEnabled` 返回 `false` → 跳过，任务标记为成功，日志 INFO 级别记录原因
- 不使用 `asynq.SkipRetry`（禁用不是错误，不应标记失败或触发重试）
- 使用窄接口而非直接依赖 `review.ConfigProvider`，避免 `queue` 包对 `review` 包的循环依赖

## 10. Bot 账户配置指南（文档）

Gitea PR Review API 使用认证用户的身份发表评审。评审者的显示名和头像取决于 API Token 对应的 Gitea 账户，**API 不支持自定义评审者身份**。

建议配置步骤：

1. 在 Gitea 上创建专用 bot 账户（如 `dtworkflow-bot`）
2. 为该账户设置有意义的显示名（如 "DTWorkflow 自动评审"）和头像
3. 在该账户下生成 API Token，配置到 `dtworkflow.yaml` 的 `gitea.token` 字段
4. 为需要评审的仓库添加该 bot 账户为协作者（至少 Read 权限 + PR Review 权限）

此指南在示例配置文件的注释中说明，不在代码层面增加配置项。

## 11. 数据库影响

**无 schema 变更**。

- 过滤是内存操作，发生在回写之前
- `review_results` 表仍存储 Claude 返回的全量评审数据（含被过滤的 issue）
- severity 各级别计数字段（`critical_count` / `error_count` / `warning_count` / `info_count`）仍为全量计数
- 未来如需统计过滤前后的差异，可通过原始数据重算

## 14. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| doublestar 库维护性 | 低 | 低 | 成熟库，5k+ stars，纯 Go 零依赖；最坏情况可用自实现替换 |
| 维度拆分破坏 prompt 质量 | 中 | 中 | 拆分时保持指令完整性和上下文连贯，单测验证拼接结果 |
| 过滤后 verdict 与用户预期不一致 | 低 | 中 | 过滤提示明确告知被过滤数量，用户可调整阈值 |
| 大量 ignore patterns 性能 | 低 | 低 | 典型配置 5-10 条 pattern，PR 文件 <100，O(N*M) 可忽略 |
