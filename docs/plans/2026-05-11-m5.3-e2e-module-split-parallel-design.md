# M5.3 模块拆分与并行执行 — 设计文档

> 日期：2026-05-11
> 状态：设计确认
> 依赖：M5.1（E2E 基础设施）+ M5.2（失败分析与报告）已完成
> 前序设计：`docs/plans/2026-05-09-m5.1-e2e-infrastructure-design.md`、`docs/plans/2026-05-11-m5.2-failure-analysis-report-design.md`

## 1. 概述

### 1.1 目标

在 M5.1/M5.2 E2E 执行引擎基础上，新增全量执行时的模块拆分与并行调度能力：宿主侧扫描仓库 `e2e/` 目录发现模块，按模块拆分为独立任务，通过 Worker 池并行执行，各模块独立通知和报告。

### 1.2 M5.1/M5.2 已有基础

- 单模块/单用例 E2E 执行引擎已完整实装（`e2e.Service.Execute`）
- 失败分析 + Issue 自动创建 + 截图上传已实装
- 飞书卡片三事件通知已实装
- `e2e/{module}/cases/` 目录约定已固定
- `e2e_results` 表按任务独立记录
- `EnqueueManualE2E` 已有单任务入队能力

### 1.3 不做什么

- 聚合通知/汇总报告（各模块独立通知，与 gen_tests 一致）
- `suite.yaml`（无明确用户场景驱动，YAGNI）
- 显式多模块参数（如 `--module order,auth`）
- 回归自动化 / Webhook PR merged 触发（M5.4）
- 趋势分析与每日统计（M5.4）

## 2. 关键设计决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | 拆分触发条件 | 仅全量模式（`--module` 为空）走宿主侧拆分 | 指定模块时仍走单容器，与 M5.1 行为一致；与 M4.2.1 gen_tests 模式对齐 |
| D2 | 模块发现机制 | 新建 `e2e.ScanE2EModules`，`ListDir("e2e")` + 逐子目录校验 `cases/` | E2E 与 gen_tests 发现逻辑差异大（扫描路径、校验条件、返回类型全不同），强行抽象是过度设计 |
| D3 | 结果聚合 | 不做聚合通知，各模块独立通知 | 与 gen_tests 多模块拆分行为一致；聚合需要"等待所有子任务完成"机制，实现复杂度高，当前无明确需求 |
| D4 | `suite.yaml` | 不纳入 M5.3 | 无明确用户场景驱动；标签过滤等需求属 M5.4 回归自动化范畴 |
| D5 | Cancel-and-Replace 粒度 | 按 `(repo, module)` 粒度取消 | 用户可能在全量执行进行中单独重跑某个失败模块，不应影响其他模块 |
| D6 | 扫描函数归属 | `internal/e2e/` 包内独立实现 | 与 gen_tests 的 `test.ScanRepoModules` 完全解耦，各自演进 |
| D7 | Store 层查询 | 泛化 `FindActiveTasksByModule(repo, module, taskType)` | gen_tests 和 E2E 的查询逻辑完全相同（按 repo + module + 活跃状态过滤），唯一区别是 task_type |
| D8 | 全量拆分前清理 | 取消同仓库所有活跃 `run_e2e` 任务 | 全量执行意味着用户想重新测所有模块，旧任务应让路；比 gen_tests 简单（无分支/PR 清理） |

## 3. E2E 模块发现

### 3.1 新建 `internal/e2e/scan.go`

```go
// E2EModuleScanner 扫描仓库 E2E 模块的窄接口。
type E2EModuleScanner interface {
    ListDir(ctx context.Context, owner, repo, ref, dir string) ([]string, error)
}

const maxE2EScanDirs = 30

// ScanE2EModules 扫描仓库 e2e/ 目录，发现包含 cases/ 子目录的模块。
// 返回模块名列表（如 ["order", "auth"]）。
func ScanE2EModules(ctx context.Context, scanner E2EModuleScanner,
    owner, repo, ref string) ([]string, error)
```

发现流程：

1. 调用 `scanner.ListDir(ctx, owner, repo, ref, "e2e")` 获取 `e2e/` 下一级子目录列表
2. 子目录数量超过 `maxE2EScanDirs`（30）时截断并 warn 日志
3. 对每个子目录调用 `scanner.ListDir(ctx, owner, repo, ref, "e2e/{dir}")` 获取其内容
4. 检查返回列表中是否包含 `"cases"` 字符串，是则视为有效模块
5. 单个子目录 `ListDir` 失败时跳过继续（warn 日志），不阻断其他模块发现
6. 最终无有效模块 → 返回 `ErrNoE2EModulesFound`

### 3.2 Sentinel 错误

```go
var ErrNoE2EModulesFound = errors.New("e2e: 未发现包含 cases/ 的模块")
```

`ErrNoE2EModulesFound` 发生在入队层（`EnqueueManualE2E` 调用 `ScanE2EModules`），不经过 Processor。API handler / CLI handler 直接向用户返回错误（见 SS6.5）。

### 3.3 与 gen_tests `ScanRepoModules` 的差异

| 维度 | gen_tests | E2E |
|------|-----------|-----|
| 扫描路径 | 仓库根 + depth=1 子目录 | 固定 `e2e/` 下一级子目录 |
| 校验条件 | `pom.xml` / `package.json` 文件存在 | `cases/` 子目录存在 |
| 返回类型 | `[]DiscoveredModule{Path, Framework}` | `[]string`（模块名） |
| Framework 维度 | 有（junit5 / vitest） | 无 |
| 根级命中 | 根目录有标记文件时 early return | 不适用（`e2e/` 本身不是模块） |

### 3.4 接口复用

`E2EModuleScanner` 接口只需 `ListDir` 一个方法。已有的 `giteaRepoFileChecker`（M4.2.1 引入）天然满足此接口，装配层无需新建 adapter。

## 4. Store 层泛化

### 4.1 接口变更

将 gen_tests 专用查询泛化为按 task_type 参数化的通用查询：

```go
// 替换 FindActiveGenTestsTasks：
FindActiveTasksByModule(ctx context.Context, repoFullName, module string, taskType model.TaskType) ([]*model.TaskRecord, error)

// 替换 ListActiveGenTestsModules：
ListActiveModules(ctx context.Context, repoFullName string, taskType model.TaskType) ([]string, error)
```

### 4.2 SQL 变更

原有 SQL 中硬编码的 `task_type='gen_tests'` 改为参数化 `task_type=?`，其余查询条件不变（按 repo + module + 活跃状态集过滤）。

### 4.3 调用方迁移

| 原调用 | 新调用 |
|--------|--------|
| `store.FindActiveGenTestsTasks(ctx, repo, module)` | `store.FindActiveTasksByModule(ctx, repo, module, model.TaskTypeGenTests)` |
| `store.ListActiveGenTestsModules(ctx, repo)` | `store.ListActiveModules(ctx, repo, model.TaskTypeGenTests)` |
| E2E 新增 | `store.FindActiveTasksByModule(ctx, repo, module, model.TaskTypeRunE2E)` |
| E2E 新增 | `store.ListActiveModules(ctx, repo, model.TaskTypeRunE2E)` |

`enqueue.go` 中 `listActiveGenTestsTasks`、`listActiveGenTestsTasksByBranchKey`、`shouldSkipAutoTestReview`、`cleanupAllAutoTestBranches` 四处调用同步替换。

## 5. 入队层改造

### 5.1 返回类型变更

```go
// 原签名：
// EnqueueManualE2E(ctx, payload, triggeredBy) (string, error)

// 新签名：
EnqueueManualE2E(ctx context.Context, payload model.TaskPayload, triggeredBy string) ([]EnqueuedTask, error)
```

复用已有的 `EnqueuedTask` 类型（gen_tests 定义，`Framework` 字段 E2E 场景留空），避免类型膨胀：

```go
// 已有类型，E2E 复用（Framework 留空）：
type EnqueuedTask struct {
    TaskID    string
    Module    string
    Framework string
}
```

### 5.2 依赖注入

`EnqueueHandler` 新增字段和 option：

```go
type EnqueueHandler struct {
    // ... 现有字段
    e2eModuleScanner E2EModuleScanner // M5.3: 可选，nil 时全量模式退化为单任务
}

func WithE2EModuleScanner(s E2EModuleScanner) EnqueueOption {
    return func(h *EnqueueHandler) { h.e2eModuleScanner = s }
}
```

`enqueue.go` 直接使用 `e2e.E2EModuleScanner` 类型（`queue` 包已导入 `e2e` 包，无循环引用风险；与 `test.RepoFileChecker` 在 `enqueue.go` 中的使用方式一致）。

### 5.3 拆分主流程

`EnqueueManualE2E` 改造逻辑（仿 `EnqueueManualGenTests`）：

```
if module 为空 && e2eModuleScanner 已注入 && baseRef 非空:
    modules, err := e2e.ScanE2EModules(...)
    if err == ErrNoE2EModulesFound:
        return err
    if err != nil:
        warn 日志，回退单任务
    if len(modules) >= 2:
        cleanupAllActiveE2ETasks(repo)  // 全量清理
    for each module:
        enqueueSingleE2E(subPayload)    // 逐模块入队
    if 全部失败:
        return error
    return results
else:
    enqueueSingleE2E(payload)           // 单任务入队
```

### 5.4 `enqueueSingleE2E`

封装单个 E2E 任务的入队 + Cancel-and-Replace：

1. 设置 `payload.TaskType = TaskTypeRunE2E` + 生成 `DeliveryID`
2. `listActiveE2ETasks(repo, module)` 查找同模块旧任务
3. `enqueueTask` 入队新任务
4. `cancelTasks` 取消旧任务（best-effort）
5. 返回 `EnqueuedTask{TaskID, Module}`

### 5.5 `cleanupAllActiveE2ETasks`

全量拆分前清理（比 gen_tests 简单，无分支/PR 清理）：

1. `store.ListActiveModules(repo, TaskTypeRunE2E)` 获取活跃模块列表
2. 对每个模块 `store.FindActiveTasksByModule(repo, mod, TaskTypeRunE2E)` 查找活跃任务
3. 逐个 `cancelTasks`（best-effort，任一失败仅 warn）

### 5.6 `listActiveE2ETasks`

简单包装，与 `listActiveGenTestsTasks` 风格一致：按 `(repo, module, TaskTypeRunE2E)` 查询，查询失败返回 nil（fail-open）。

## 6. API / CLI / dtw 适配

### 6.1 REST API

`POST /api/v1/repos/{owner}/{repo}/e2e` 响应适配多任务返回，复用 gen_tests 的 `split/tasks` 结构：

单模块（指定 `--module`）：

```json
{
    "task_id": "xxx",
    "repo": "changyu/backend",
    "module": "order",
    "env": "staging",
    "status": "pending"
}
```

全量拆分（不指定 `--module`）：

```json
{
    "split": true,
    "tasks": [
        {"task_id": "aaa", "module": "order"},
        {"task_id": "bbb", "module": "auth"}
    ]
}
```

API handler 根据 `len(results) > 1` 判断是否返回 split 结构。单任务时保持现有响应格式向后兼容。

### 6.2 服务端 CLI

`dtworkflow e2e run` 多任务返回时输出拆分汇总表格：

```
E2E 测试已拆分为 3 个模块任务：

  模块          任务 ID
  order         aaa-bbb-ccc
  auth          ddd-eee-fff
  payment       ggg-hhh-iii
```

### 6.3 瘦客户端 dtw

`dtw e2e run` 同理：
- `--no-wait` 模式：输出汇总表格后退出
- 等待模式：逐个轮询所有子任务（与 gen_tests 的 dtw 行为一致）
- `--json` 模式：直接输出 API 的 JSON 响应

### 6.5 错误处理

`EnqueueManualE2E` 返回 `ErrNoE2EModulesFound` 时，调用方应向用户返回明确的业务错误而非 500：

- **API handler**：`errors.Is(err, e2e.ErrNoE2EModulesFound)` → HTTP 422，消息"仓库中未发现 E2E 测试模块（需 `e2e/{module}/cases/` 目录结构）"。与 gen_tests 处理 `test.ErrNoFrameworkDetected` 返回 422 的模式一致
- **服务端 CLI**：同理，输出错误提示后以退出码 1 退出
- **dtw 瘦客户端**：API 返回 422 时展示错误消息

### 6.6 调用方签名适配

`EnqueueManualE2E` 返回值从 `(string, error)` 变为 `([]EnqueuedTask, error)`，所有调用方（API handler、服务端 CLI、dtw CLI）需同步适配。单任务场景取 `results[0].TaskID`。

## 7. Processor 适配

Processor 层**无需改动**：

- `ErrNoE2EModulesFound` 发生在入队层，不经过 Processor（见 SS3.2）
- 路由逻辑不变（`TaskTypeRunE2E → e2eService.Execute`）
- 通知逻辑不变（各子任务独立发送飞书通知）
- `adaptE2EResult` 不变
- Issue 创建 / 截图上传 / 持久化均由 `e2e.Service` 内部处理，不受拆分影响

## 11. 测试策略

### 11.1 `e2e/scan_test.go`（新建）

- 正常：`e2e/` 下 3 个子目录，2 个含 `cases/`，1 个不含 → 返回 2 个模块
- 空目录：`e2e/` 下无子目录 → `ErrNoE2EModulesFound`
- 无合格模块：子目录都不含 `cases/` → `ErrNoE2EModulesFound`
- `ListDir("e2e")` 失败 → 返回 error
- 单个子目录 `ListDir` 失败 → 跳过该模块，继续扫描
- 截断：超过 `maxE2EScanDirs` 个子目录 → 截断处理

### 11.2 `store/sqlite_test.go`

- `FindActiveTasksByModule` 按 `TaskTypeGenTests` 查询 → 结果与原 `FindActiveGenTestsTasks` 一致（回归）
- `FindActiveTasksByModule` 按 `TaskTypeRunE2E` 查询 → 正确过滤
- `ListActiveModules` 同理

### 11.3 `queue/enqueue_test.go`

- 全量拆分路径：2 个模块 → 先清理旧任务 → 2 个子任务入队
- 全量拆分 scanner 未注入 → 单任务入队（fail-open）
- 扫描失败 → 回退单任务
- `ErrNoE2EModulesFound` → 返回 error
- 单模块 Cancel-and-Replace：同模块旧任务被取消
- 单模块入队不影响其他模块正在执行的任务
- gen_tests 回归：泛化后行为不变

### 11.4 回归验证

全量 `go test ./... -count=1` 确保 M5.1/M5.2 执行路径不受影响（`e2eModuleScanner=nil` 时退化为现有单任务逻辑）。

## 12. 与后续里程碑的衔接

| 后续里程碑 | M5.3 的预留 |
|-----------|------------|
| M5.4 回归自动化 | 模块发现 + 拆分入队已就绪；Webhook PR merged 事件复用 M4.3 模式，调用 `ScanE2EModules` + `EnqueueManualE2E` 即可 |
| M5.4 趋势分析 | `e2e_results` 各模块独立记录，按 `module` + `created_at` 聚合即可 |
| `suite.yaml` | 模块发现由宿主侧完成，`suite.yaml` 仅影响容器内 Claude 的用例排序/过滤，可在 prompt 层独立引入，不影响拆分架构 |
