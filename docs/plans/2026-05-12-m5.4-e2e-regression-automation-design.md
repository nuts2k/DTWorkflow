# M5.4 E2E 回归自动化 — 设计文档

> 日期：2026-05-12
> 状态：设计确认
> 依赖：M5.1（E2E 基础设施）+ M5.2（失败分析与报告）+ M5.3（模块拆分与并行执行）已完成
> 前序设计：`docs/plans/2026-05-11-m5.3-e2e-module-split-parallel-design.md`、`docs/plans/2026-05-03-m4.3-change-driven-test-gen-design.md`

## 1. 概述

### 1.1 目标

PR 合并后自动触发 E2E 回归测试。通过两阶段流水线实现智能模块选择：阶段 1 由 Claude 分析代码变更与 E2E 用例的关联性，输出应执行的模块列表；阶段 2 按模块拆分入队，复用 M5.1/M5.2/M5.3 全部能力并行执行。

### 1.2 核心设计决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | 变更 → 模块映射方式 | Claude 容器内分析（两阶段流水线） | 代码路径与 E2E 模块是业务语义关联，非路径前缀匹配；Claude 擅长理解这种语义关联；模块 >10 个时筛选比全量回归更经济 |
| D2 | 阶段实现方式 | 新增 `TaskTypeTriageE2E` + Processor 内链式入队 | 与 gen_tests 完成后入队 review（D6）模式一致；分析任务享有独立超时/重试/通知 |
| D3 | 分析容器模式 | 轻量镜像 + 只读 + 容器内自读仓库 | 只需读 git diff + `e2e/` 目录，无需浏览器；Claude 在容器内自行探索 case.yaml 比宿主侧 API 逐个读取更自然 |
| D4 | 分析失败兜底 | 重试耗尽后跳过 + 通知（fail-closed） | 全量回归资源耗费大，用户可随时手动补；与 fail-open 相比避免意外资源消耗 |
| D5 | 空模块结果处理 | 信任 Claude 判断，跳过 + 通知 | 选择 Claude 做分析本身就是信任其判断，空结果是合理判断 |
| D6 | 执行环境 | 使用 `e2e.default_env` | 自动触发无人指定环境；配置校验要求 `regression.enabled=true` 时 `default_env` 非空 |
| D7 | 手动触发 | 不新增 triage 手动入口 | 用户想手动回归直接用 `dtw e2e run`（全量或指定模块） |
| D8 | M5.4 范围 | 仅回归自动化核心；趋势分析 + 每日报告拆到 M5.4.1 | 两阶段流水线本身复杂度足够，避免交付风险 |

### 1.3 不做什么

- 趋势分析（M5.4.1）
- 每日 E2E 统计报告（M5.4.1）
- 手动触发 triage（用现有 `dtw e2e run`）
- 全量降级（分析失败时不自动全量回归）
- 新的结果持久化表（triage 产出通过 `tasks` 表 + `triggered_by` 反查）

## 2. 数据流总览

```
PR Merged (Gitea Webhook)
  │
  ▼
HandlePullRequest → action="merged" → handleMergedPullRequest
  │
  ├─ [现有] gen_tests 变更驱动逻辑（M4.3，不变）
  │
  └─ [新增] handleMergedE2ERegression
        │
        ├─ ① Bot PR 过滤（auto-test/* / auto-fix/* 跳过）
        ├─ ② 配置检查（e2e.regression.enabled）
        ├─ ③ ListPullRequestFiles → extractFilenames → filterSourceFiles
        │     过滤后为空 → 跳过（纯文档 PR）
        ├─ ④ 入队 triage_e2e 任务
        │     payload: 变更文件列表 + PR 信息 + baseRef + default_env
        │
        ▼
Processor → triage_e2e → 轻量容器（只读）
  │
  ├─ Claude 读取 git diff + e2e/**/case.yaml
  ├─ 输出 TriageE2EOutput JSON
  │
  ▼
Processor.handleTriageE2EResult
  │
  ├─ modules 为空 → 通知"无需回归" → succeeded
  ├─ modules 非空 → 逐模块调 EnqueueManualE2E
  │     ├─ run_e2e: order（env=staging）
  │     ├─ run_e2e: auth（env=staging）
  │     └─ ...
  │
  ▼
各模块独立执行（复用 M5.1/M5.2/M5.3 全部能力）
  → 失败分析 → Issue 创建 → 飞书通知
```

## 3. TaskType 设计

### 3.1 新增常量

```go
TaskTypeTriageE2E = "triage_e2e"
```

### 3.2 任务属性

| 属性 | 值 | 说明 |
|------|-----|------|
| 镜像 | 轻量镜像（`worker.image`） | 与 `review_pr` / `analyze_issue` 一致 |
| 超时 | 10 分钟 | 分析任务不需要太长 |
| 重试 | 3 次（默认） | 与其他任务一致 |
| 容器模式 | 只读（ReadonlyRootfs + `--disallowedTools` + prompt READ-ONLY） | 禁止修改文件 |
| Token | review 账号 | 只读克隆，无需 push 权限 |

### 3.3 超时配置

`TaskTimeoutsConfig` 新增 `TriageE2E` 字段（默认 10 分钟）。

## 4. 配置结构

### 4.1 YAML 格式

```yaml
e2e:
  enabled: true
  default_env: staging
  regression:
    enabled: false          # *bool，nil=false 默认关闭
    ignore_paths:           # 宿主侧初筛，doublestar glob
      - "docs/**"
      - "scripts/**"

repos:
  - name: "myorg/backend"
    e2e:
      regression:
        enabled: true       # 仓库级覆盖
        ignore_paths:
          - "deploy/**"
```

### 4.2 类型定义

```go
type RegressionConfig struct {
    Enabled     *bool    `mapstructure:"enabled"`      // nil=false，默认关闭
    IgnorePaths []string `mapstructure:"ignore_paths"`  // doublestar glob
}

func (c RegressionConfig) IsEnabled() bool {
    return c.Enabled != nil && *c.Enabled
}
```

挂载在 `E2EOverride.Regression *RegressionConfig`。

### 4.3 配置解析

`ResolveE2EConfig` 合并逻辑扩展：仓库级 `regression` 整体替换（与 M4.3 `change_driven` 一致，不做字段级合并）。

### 4.4 配置校验

在 `validateE2E` 中追加：

1. `regression.ignore_paths` 每条做 `doublestar.ValidatePattern` 语法校验
2. `regression.enabled=true` 时要求 `e2e.enabled` 不为 `false`（矛盾检测）
3. `regression.enabled=true` 时要求 `e2e.default_env` 非空（链式入队 E2E 时需要环境）

### 4.5 Clone 深拷贝

`Config.Clone()` 补充 `RegressionConfig` 的深拷贝（`Enabled` 指针 + `IgnorePaths` 切片），全局和仓库级各一处。

## 5. 宿主侧初筛与入队

### 5.1 handleMergedPullRequest 重构

现有 `handleMergedPullRequest` 函数体即 gen_tests 变更驱动逻辑，内含 12+ 个 early-return 分支（configProvider 为 nil、change_driven 未启用、过滤后无源码等）。直接在末尾追加 E2E 回归逻辑会导致 E2E 路径在 gen_tests early-return 时成为死代码。

**必须重构**：将现有 gen_tests 逻辑提取为 `handleMergedGenTests`，`handleMergedPullRequest` 改为调度层，共享 Bot PR 过滤后独立调用两条路径：

```go
func (h *EnqueueHandler) handleMergedPullRequest(ctx context.Context, event webhook.PullRequestEvent) error {
    pr := event.PullRequest

    // 共享：Bot PR 过滤（两条路径都需要）
    if strings.HasPrefix(pr.HeadRef, "auto-test/") || strings.HasPrefix(pr.HeadRef, "auto-fix/") {
        return nil
    }

    // gen_tests 变更驱动（M4.3，提取为独立方法）
    genErr := h.handleMergedGenTests(ctx, event)

    // E2E 回归（M5.4，独立方法，失败不阻断 gen_tests 返回值）
    h.handleMergedE2ERegression(ctx, event)

    return genErr
}
```

`handleMergedGenTests` 为现有逻辑的纯提取（去掉 Bot PR 过滤，已在调度层完成），零行为变更。两条路径各自维护 early-return 和错误处理，互不干扰。

### 5.2 handleMergedE2ERegression

```
func (h *EnqueueHandler) handleMergedE2ERegression(ctx, event):
    // 注：Bot PR 过滤已在 handleMergedPullRequest 调度层完成（见 SS5.1）
    ① 配置检查（e2eConfigProvider.ResolveE2EConfig → regression.IsEnabled）
    ② 宿主侧初筛：复用 filterSourceFiles（regression.IgnorePaths）
       过滤后为空 → Debug 日志 + return
    ③ 入队 triage_e2e 任务
       payload 携带：变更文件列表、PR 信息、baseRef、default_env
       triggered_by = "webhook:pr_merged:<pr_number>"
    // 注：本方法内部 log 所有错误（调度层丢弃返回值）
```

同一 PR 合并可同时触发 gen_tests 变更驱动和 E2E 回归——gen_tests 检查 `change_driven.enabled`，E2E 检查 `regression.enabled`，两者独立判断。

### 5.3 依赖注入

新增配置读取窄接口：

```go
type E2ERegressionConfigProvider interface {
    ResolveE2EConfig(repoFullName string) config.E2EOverride
}

func WithE2ERegressionConfigProvider(p E2ERegressionConfigProvider) EnqueueOption {
    return func(h *EnqueueHandler) { h.e2eConfigProvider = p }
}
```

装配层 `serve_deps.go` 注入 `cfgAdapter`（已实现 `ResolveE2EConfig`）。

复用现有依赖（无需新增）：
- `prFilesLister`（M4.3 已注入）：review token client，只读查询 PR 变更文件
- `filterSourceFiles`（M4.3 `change_filter.go`）：宿主侧初筛
- `extractFilenames`（M4.3）：提取文件名 + 排除 deleted

### 5.4 幂等保护

`buildE2ERegressionDeliveryID` 构建 `{webhookDeliveryID}:triage_e2e` 复合 ID（不需要模块维度——triage 是单任务），同一 webhook 重发不重复入队。

### 5.5 Cancel-and-Replace

同一仓库如果已有活跃的 `triage_e2e` 任务（前一个 PR 合并触发的分析还在跑），新入队后取消旧的。按 `(repo, TaskTypeTriageE2E)` 粒度查询和取消。

## 6. 分析容器执行

### 6.1 entrypoint.sh

新增 `triage_e2e)` case，与 `analyze_issue` 行为一致：

- 只读克隆 + checkout `BASE_REF`
- 无 git identity / 无 credential helper / 无 pre-push hook
- 环境变量照常清除（`GITEA_TOKEN` / `AUTH_URL` / `GITEA_URL` / `REPO_CLONE_URL`）

### 6.2 worker/container.go

- `resolveImage`：`triage_e2e` → 轻量镜像
- `buildContainerCmd`：stdin 传入分析 prompt + `--output-format json` + `--disallowedTools`（禁止写操作）
- `buildContainerEnv`：注入 `BASE_REF`
- `selectGiteaToken`：`triage_e2e` → review 账号

### 6.3 分析 Prompt

四段式结构：

```text
[上下文段]
你正在分析一个 PR 合并后的代码变更，判断哪些 E2E 测试模块需要运行回归测试。
仓库：{repo}，合并分支：{baseRef}
本次 PR 变更了以下文件：
- path/to/file1.java
- path/to/file2.vue
...（最多 50 个，超出截断并注明"及其他 N 个文件"）

[分析指令段]
1. 读取 e2e/ 目录下的所有模块，理解每个模块的 case.yaml 定义的业务场景
2. 分析上述变更文件的代码逻辑和业务影响
3. 判断哪些 E2E 模块的测试场景可能被本次变更影响
4. 考虑间接影响（如修改了被多个模块依赖的公共服务）

[约束段]
- READ-ONLY：禁止修改、创建、删除任何文件
- 禁止执行任何测试或脚本
- 禁止访问外部网络

[输出格式段]
以 JSON 格式输出，schema 如下：
{
  "modules": [{"name": "模块名", "reason": "选择理由"}],
  "skipped_modules": [{"name": "模块名", "reason": "跳过理由"}],
  "analysis": "整体分析说明"
}
modules 为必填字段（可为空数组），表示需要执行回归测试的 E2E 模块。
skipped_modules 为可选字段，列出明确跳过的模块及理由。
analysis 为可选字段，提供整体分析说明。
```

Prompt 安全约束与 `review_pr` / `analyze_issue` 一致：READ-ONLY 文本 + `--disallowedTools` 双层防御。额外追加"禁止修改任何测试相关文件"的显式约束。

## 7. 输出解析

### 7.1 TriageE2EOutput 类型

```go
// internal/e2e/triage.go

type TriageE2EOutput struct {
    Modules        []TriageModule `json:"modules"`
    SkippedModules []TriageModule `json:"skipped_modules"`
    Analysis       string         `json:"analysis"`
}

type TriageModule struct {
    Name   string `json:"name"`
    Reason string `json:"reason"`
}
```

### 7.2 解析流程

双层 JSON 解析（CLI 信封 → TriageE2EOutput），与项目所有其他任务类型一致。

校验规则：
- `modules` 字段必须存在（可为空数组，不可缺失）
- 每个 module 的 `name` 非空
- `sanitizeTriageOutput`：对 `reason` 和 `analysis` 做自由文本脱敏（防 prompt injection），截断 2KB

### 7.3 错误处理

解析失败返回 error（非 SkipRetry），允许 asynq 重试。重试耗尽后 Processor 走 fail-closed：标记失败 + 发通知，不触发全量回归。

## 8. Processor 集成

### 8.1 路由

`processTask` switch 新增 `TaskTypeTriageE2E` case：

```go
case model.TaskTypeTriageE2E:
    containerResult := pool.RunWithCommandAndStdin(...)
    triageOutput, err := e2e.ParseTriageResult(containerResult)
    // 解析失败 → 返回 error（允许重试）
    // 解析成功 → handleTriageE2EResult
```

triage 不需要独立 Service——逻辑足够简单（解析 JSON + 链式入队），直接在 Processor 内处理。

### 8.2 链式入队

```go
func (p *Processor) handleTriageE2EResult(ctx context.Context, record *model.TaskRecord, output *e2e.TriageE2EOutput) {
    if len(output.Modules) == 0 {
        // 通知"无需回归" + 标记 succeeded
        return
    }

    env := record.Payload.Environment  // 从 triage payload 透传的 default_env
    triggeredBy := fmt.Sprintf("triage_e2e:%s", record.TaskID)

    for _, mod := range output.Modules {
        payload := model.TaskPayload{
            TaskType:     model.TaskTypeRunE2E,
            RepoOwner:    record.Payload.RepoOwner,
            RepoName:     record.Payload.RepoName,
            RepoFullName: record.Payload.RepoFullName,
            CloneURL:     record.Payload.CloneURL,
            Module:       mod.Name,
            BaseRef:      record.Payload.BaseRef,
            Environment:  env,
        }
        _, err := p.enqueueHandler.EnqueueManualE2E(ctx, payload, triggeredBy)
        if err != nil {
            slog.Warn("triage_e2e: 链式入队失败", "module", mod.Name, "error", err)
        }
    }
}
```

链式入队的 `run_e2e` 通过 `triggered_by = "triage_e2e:{taskID}"` 标记来源，可反查到触发的 triage 任务。

### 8.3 依赖注入

```go
func WithEnqueueHandler(h *EnqueueHandler) ProcessorOption {
    return func(p *Processor) { p.enqueueHandler = h }
}
```

与 `WithTestService`、`WithE2EService` 注入模式一致。

## 9. 通知

### 9.1 事件定义

新增 3 个 EventType：

```go
EventE2ETriageStarted = "e2e_triage_started"
EventE2ETriageDone    = "e2e_triage_done"
EventE2ETriageFailed  = "e2e_triage_failed"
```

### 9.2 飞书卡片

| 事件 | 色调 | 内容 |
|------|------|------|
| Started | 蓝 | "E2E 回归分析开始"，显示触发 PR 信息 |
| Done + modules 非空 | 绿 | 选中模块列表 + 跳过模块列表 + 分析摘要 |
| Done + modules 为空 | 绿 | "本次变更无需 E2E 回归" |
| Failed | 红 | 失败原因，提示可手动触发全量回归 |

### 9.3 通知作用域

triage 三事件属 repo 级通知，用 `repoScopedGiteaNotifier` 包装（与 gen_tests / run_e2e 一致）。

链式入队的 `run_e2e` 任务复用现有 M5.1 三事件（`EventE2EStarted/Done/Failed`），不做额外处理。

### 9.4 用户视角时间线

```
[蓝] E2E 回归分析开始（PR #42 合并触发）
[绿] E2E 回归分析完成：选中 order、auth，跳过 payment
[蓝] E2E 测试开始：order 模块
[蓝] E2E 测试开始：auth 模块
[绿] E2E 测试完成：order 模块 — 3/3 通过
[红] E2E 测试失败：auth 模块 — 1/2 失败 → Issue #58
```

## 10. 持久化

**不新增结果表**。理由：

- Triage 的核心产出是"触发了哪些 E2E 任务"，通过 `tasks` 表的 `triggered_by = "triage_e2e:{taskID}"` 可反查
- `tasks` 表本身记录 triage 任务的状态、耗时、error
- `TaskRecord.Result` 字段保存 Claude 原始 JSON 输出，可回溯选中/跳过模块明细
- 趋势分析（M5.4.1）的数据基础是 `e2e_results` 表，不依赖 triage 结果表

**数据库迁移**：需要新增迁移 v23，重建 `tasks` 表追加 `triage_e2e` 到 `task_type` CHECK 约束（与 v21 追加 `run_e2e` 的模式一致）。当前 CHECK 枚举为 `('review_pr', 'analyze_issue', 'fix_issue', 'gen_tests', 'gen_daily_report', 'run_e2e')`，需追加 `'triage_e2e'`。
