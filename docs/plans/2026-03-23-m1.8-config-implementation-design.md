# M1.8 配置管理实施设计

> 日期：2026-03-23
> 范围：结合 `docs/ROADMAP.md` 中 M1.8 配置管理要求，以及 2026-03-23 已落地的 M1.5-M1.7 改动，形成一份以 M1.8 实施步骤为重点的承接设计文档。

## 1. 背景

根据 `docs/ROADMAP.md`，M1.8 的目标包括：

- 集成 Viper，建立全局 YAML 配置管理
- 支持仓库级配置
- 支持配置热加载

截至今天，代码库已经完成了 M1.5-M1.7 的关键基础能力：

- `internal/queue/`：任务入队、消费、重试、恢复
- `internal/worker/`：Docker Worker 池、容器生命周期、GC
- `internal/notify/`：通知接口、Gitea 通知器、通知路由
- `internal/cmd/serve.go`：`ServiceDeps` 依赖装配、`serve -> queue -> worker -> notify` 主链路接通
- `internal/cmd/task.go`：任务查询与重试 CLI

这意味着当前系统的“运行时能力”已经具备，M1.8 的主要任务不再是新增业务能力，而是把这些能力依赖的配置来源从零散的 CLI flags、环境变量、硬编码默认值和最小硬编码路由，统一收拢为可管理、可校验、可扩展的配置体系。

## 2. 当前基线与衔接关系

### 2.1 `root` 入口现状

`internal/cmd/root.go` 当前仅注册了：

- `--json`
- `--config`
- `--verbose`

但尚未真正使用 `--config` 加载配置文件，也没有统一的配置初始化入口。

### 2.2 `serve` 命令现状

`internal/cmd/serve.go` 当前仍采用“包级变量 + `serveConfig` 运行时快照”的方式管理配置，存在以下特点：

- `init()` 中为 `serve` 注册了较多 flags
- `runServe()` 从包级变量组装 `serveConfig`
- `runServeWithConfig()` 中仍会对敏感配置做环境变量回填
- `BuildServiceDeps()` 负责真实依赖装配
- `buildNotifier()` 仍使用最小硬编码路由：`* -> gitea`

这说明当前 `serve` 虽然已经具备良好的依赖装配结构，但配置入口仍未统一。

### 2.3 `task` 命令现状

`internal/cmd/task.go` 当前在自己的 `PersistentPreRunE` 中直接根据 `taskDBPath` 和 `taskRedisAddr` 初始化：

- `store.NewSQLiteStore`
- `queue.NewClient`

这意味着 `task` 命令仍在自主管理一套配置来源，与 M1.8 的统一配置目标不一致。

### 2.4 `notify` 路由现状

M1.7 已经把通知能力抽象为：

- `notify.GiteaNotifier`
- `notify.Router`
- `queue.Processor` 中的可选通知发送

但当前通知路由仍是最小闭环接线，路由规则与 fallback 由 `internal/cmd/serve.go` 中的 `buildNotifier()` 硬编码提供。M1.8 的目标不是重写通知框架，而是把通知策略迁移为配置驱动。

### 2.5 M1.8 与 M1.5-M1.7 的承接关系

- M1.5-M1.6 提供任务执行引擎与服务装配骨架
- M1.7 提供通知抽象与最小通知闭环
- M1.8 在此基础上统一配置来源，并将策略数据（尤其是通知路由）外置为配置

因此，M1.8 的实施重点应放在“配置治理”和“装配接线”，而不是重做已有运行能力。

## 3. M1.8 目标与边界

### 3.1 目标

M1.8 应完成以下能力：

1. 引入 `internal/config` 配置管理模块，统一 YAML / 环境变量 / CLI flags / 默认值
2. 在 `root` 命令层统一加载配置
3. 让 `serve` 从统一配置入口读取全部运行参数
4. 让 `task` 命令接入统一配置体系
5. 把通知路由从硬编码迁移为配置驱动
6. 提供仓库级配置覆盖能力
7. 提供配置热加载框架，并明确即时生效边界

### 3.2 非目标

本轮不做以下内容：

1. 不新增企业微信、钉钉、飞书等新通知渠道
2. 不重写 M1.5-M1.7 已经稳定的主流程逻辑
3. 不把 Phase 2/3 的业务能力提前落入 M1.8
4. 不尝试让 Redis、SQLite、Worker Pool、HTTP 监听地址等基础设施配置全部在线热切换
5. 不把 `task` / `serve` 做成一次性彻底架构重构

### 3.3 设计原则

1. **配置是数据，装配是行为**：`internal/config` 负责配置数据；`internal/cmd` 负责运行时装配
2. **先统一读取入口，再逐步清理历史结构**：M1.8 首先解决“配置从哪里读”，而不是一口气重构所有命令
3. **尽量复用现有抽象**：尤其是复用 `ServiceDeps`、`notify.Router`、`serveConfig`
4. **热加载边界可控**：先实现框架和少量即时生效能力，避免把 M1.8 膨胀成运行时依赖重建系统

## 4. 总体方案

### 4.1 统一配置链路

推荐采用以下主链路：

```text
配置文件 / 环境变量 / CLI flags
        ↓
internal/config.Manager
        ↓
root 命令统一加载配置
        ↓
serve / task / 后续 review-pr、fix-issue 命令读取配置
        ↓
BuildServiceDeps 使用配置装配 Store / Queue / Worker / Gitea / Notifier
```

关键点有两个：

- `root` 层统一加载配置，而不是每个命令自行加载 YAML
- `internal/config` 不参与依赖构造，真实依赖仍由 `internal/cmd` 装配

### 4.2 配置模型

M1.8 至少需要覆盖以下配置段：

- `server`
- `webhook`
- `redis`
- `database`
- `worker`
- `claude`
- `gitea`
- `notify`
- `repos`

其中 `worker` 需要显式承接当前仍在硬编码或 TODO 状态的字段：

- `concurrency`
- `image`
- `cpu_limit`
- `memory_limit`
- `network_name`

### 4.3 命令层接线方式

#### root

在 `rootCmd` 上增加统一的 `PersistentPreRunE`，完成：

- 初始化 `config.Manager`
- 处理 `--config`
- 绑定环境变量前缀
- 绑定 CLI flags 到 Viper
- 执行 `Load()` 与 `Validate()`
- 保存共享配置管理实例

#### serve

`serve` 应改为从 `cfgManager.Get()` 读取配置，再构造 `serveConfig`。保留 `serveConfig` 作为运行时快照结构，但不再以包级变量作为最终配置来源。

#### task

`task` 应先完成配置来源统一：继续保留现有 Store / QueueClient 初始化方式，但其所需参数改由统一配置入口提供。后续若需要进一步去包级状态化，可在后续迭代处理。

### 4.4 通知路由配置化

M1.7 已具备通知能力，本轮只将策略迁移为配置驱动：

- `buildNotifier()` 接收 `config.Config` 或对应通知配置视图
- 在 `internal/cmd` 层完成 `config.NotifyConfig -> []notify.RoutingRule` 映射
- 保持 `notify.Router` 核心行为不变

推荐在服务启动阶段就将全局 routes 与 repo 级 notify override 编译为最终规则列表，而不是侵入 `notify.Router` 运行时逻辑。

### 4.5 热加载边界

M1.8 首轮建议实现：

- `WatchConfig()` 监听配置变更
- 当前配置快照可更新
- 少量轻量配置可即时生效，如日志级别

本轮不要求立即热生效的配置包括：

- `server.host` / `server.port`
- `redis.*`
- `database.path`
- `gitea.*`
- `worker.image` / `worker.network_name` / 资源限制
- `asynq` 并发配置

这些配置变更可以记录日志，并要求重启服务生效。

## 5. 实施步骤（重点）

### 5.1 建立 `internal/config` 配置模型与 Manager

建议新增：

- `internal/config/config.go`
- `internal/config/errors.go`
- `internal/config/config_test.go`

第一阶段重点不是重构命令，而是先建立统一的配置能力：

1. 定义 `Config` 及子结构体
2. 封装 `config.Manager`
3. 集成默认值、环境变量绑定、配置文件读取
4. 输出统一配置快照

这一阶段完成后，系统第一次具备可复用的统一配置入口。

### 5.2 增加配置校验与仓库级覆盖

建议新增：

- `internal/config/validate.go`
- `internal/config/validate_test.go`
- `internal/config/repo_config.go`
- `internal/config/repo_config_test.go`

重点校验：

- `server.port` 范围
- `worker.concurrency >= 1`
- `webhook.secret` 必填
- `gitea.url` / `gitea.token` 必填
- `notify.default_channel` 合法
- `notify.routes[*].channels` 必须引用已定义渠道
- `repos[*].name` 合法

仓库级覆盖建议先支持：

- `notify` 路由覆盖
- `review` 配置结构预留

其中本轮真正接入现有主流程的是通知覆盖；review 相关覆盖先完成数据结构与解析能力，为后续 Phase 2 铺路。

### 5.3 把 `root` 变成统一配置入口

重点修改：

- `internal/cmd/root.go`

目标是让 `--config` 真正生效，并建立统一配置加载流程。建议在 `rootCmd` 上增加 `PersistentPreRunE`，完成：

1. 初始化 `config.Manager`
2. 处理 `--config`
3. 设置默认搜索路径
4. 绑定环境变量前缀
5. 绑定相关命令 flags
6. `Load()` + `Validate()`
7. 将 `cfgManager` 保存为共享变量

这一步完成后，CLI 层才真正拥有统一配置入口。

### 5.4 迁移 `serve` 到配置驱动

重点修改：

- `internal/cmd/serve.go`

当前 `serve` 的问题不在于依赖装配结构差，而在于配置来源仍是混合态。M1.8 应将 `runServe()` 改为从统一配置入口构造 `serveConfig`，并顺手解决以下问题：

1. `claude-api-key`、`gitea-token`、`webhook-secret` 不再在 `runServeWithConfig()` 中手工回读环境变量
2. `worker.cpu_limit` / `worker.memory_limit` 从配置读取，不再只依赖默认硬编码
3. `worker.network_name` 配置化，替代当前硬编码的 `dtworkflow-net`
4. `BuildServiceDeps()` 以统一配置作为依赖装配依据

完成后，`serve` 将从“flag/env 混合驱动”切换为“配置驱动装配”。

### 5.5 迁移 `task` 命令到统一配置体系

重点修改：

- `internal/cmd/task.go`

当前 `task` 命令还在自主管理一套配置初始化逻辑。M1.8 中建议先做最小承接：

1. 保留现有 Store / QueueClient 初始化路径
2. 将 `taskDBPath` 与 `taskRedisAddr` 的真实来源切换为统一配置入口
3. 保持 `status/list/retry` 的行为与输出不变

这样可以先把 `task` 纳入统一配置体系，而不把 M1.8 扩大成一次完整架构重构。

### 5.6 通知路由配置化

重点修改：

- `internal/cmd/serve.go`
- `internal/cmd/adapter.go`
- `configs/dtworkflow.example.yaml`

当前 `buildNotifier()` 仍使用最小硬编码规则。M1.8 应将其升级为配置驱动：

1. `buildNotifier()` 接收通知配置
2. 在 `internal/cmd` 层完成配置到 `notify.RoutingRule` 的映射
3. 接入全局默认路由与 fallback
4. 接入 repo 级 notify override

实施重点应是复用现有 `notify.Router`，把变化控制在装配层，而不是修改 `notify` 包的核心接口。

### 5.7 接入热加载与模板更新

建议新增：

- `internal/config/watcher.go`
- `internal/config/watcher_test.go`

并更新：

- `configs/dtworkflow.example.yaml`

热加载首轮重点是框架与边界：

1. 监听配置文件变更
2. 更新当前配置快照
3. 对少量轻量配置提供即时生效能力
4. 对基础设施配置明确要求“重启后生效”

示例配置文件需要完整覆盖 M1.8 引入的全部配置段，确保用户能看到可运行的模板。

## 6. 测试与验收

M1.8 的验收不应只停留在“配置文件能读出来”，而应同时证明：

- 配置入口已统一
- 现有运行链路未被破坏
- 通知策略已真正配置化

### 6.1 配置模块单元测试

验证：

- YAML 加载
- 默认值填充
- 环境变量覆盖
- 配置校验
- 仓库级覆盖
- 热加载更新与错误隔离

### 6.2 命令层集成测试

验证：

- `root` 成为统一配置入口
- `--config`、环境变量、CLI flags 的优先级正确
- `serve` 从统一配置读取运行参数
- `task` 命令接入统一配置体系

### 6.3 通知配置化测试

验证：

- 全局默认通知路由与当前最小硬编码行为等价
- 指定事件路由映射正确
- repo 级通知覆盖生效
- 非法通知配置在 `Validate()` 阶段即被拦截

### 6.4 回归与端到端验收

验证以下能力在 M1.8 接入后仍成立：

- `serve` 能正常启动
- `/healthz` 与 `/readyz` 行为正常
- Webhook -> Queue -> Worker -> Store 主链路不回退
- 通知闭环仍可用
- `task status/list/retry` 行为不变

### 6.5 建议验证命令

建议至少执行：

```bash
go test ./internal/config/...
go test ./internal/cmd/...
go test ./internal/notify ./internal/queue ./internal/cmd -v
go test ./...
```

并进行一轮手工验证：

1. 使用示例 YAML 启动 `dtworkflow serve --config <path>`
2. 检查 `/healthz` 与 `/readyz`
3. 执行一次 `dtworkflow task list`

## 7. 风险与实施顺序建议

### 7.1 主要风险

#### 风险 1：配置入口双轨并存

如果只是新增 `config.Manager`，但 `serve`、`task` 没有真正切换过去，就会形成“部分命令读 config、部分命令读旧变量”的双轨状态。

#### 风险 2：`internal/config` 职责膨胀

如果把依赖构造、路由映射、运行时重建都塞进 `config` 包，会使模块边界迅速变脏。

#### 风险 3：热加载范围失控

如果试图让 Redis、SQLite、Worker Pool、HTTP 监听地址等都在线热切换，M1.8 会迅速超出合理范围。

#### 风险 4：配置化侵入现有通知抽象

如果为配置驱动而修改 `notify.Router` 核心行为，会提高回归风险。应优先通过装配层映射实现配置化。

#### 风险 5：顺手重构过多导致范围失控

`serve` 和 `task` 当前确实仍有结构优化空间，但 M1.8 首轮目标应是完成配置治理，而不是同时解决所有历史结构问题。

### 7.2 推荐实施顺序

建议按以下顺序推进：

1. `internal/config/config.go`
2. `internal/config/validate.go`
3. `internal/config/repo_config.go`
4. `internal/cmd/root.go`
5. `internal/cmd/serve.go`
6. `internal/cmd/task.go`
7. `internal/config/watcher.go`
8. `configs/dtworkflow.example.yaml`

这个顺序可以保证先建立配置能力，再逐步接入现有命令与通知策略，降低改造风险。

### 7.3 完成定义

满足以下条件即可视为 M1.8 完成：

1. 已引入统一的 `internal/config` 配置管理模块
2. `root` 命令可统一加载配置文件、环境变量与 CLI flags
3. `serve` 已完全从统一配置入口读取运行参数
4. `task` 已接入统一配置体系
5. 通知路由由配置驱动，不再依赖硬编码默认规则
6. 支持仓库级配置覆盖
7. 支持配置热加载框架，且生效边界清晰
8. 相关单元测试、命令层测试、回归测试通过

## 8. 与 ROADMAP 的对应关系

本设计对应 `docs/ROADMAP.md` 中 M1.8 的三项要求：

- **Viper 集成，全局配置文件格式定义（YAML）**：通过 `internal/config.Manager + Config` 实现
- **仓库级配置支持**：通过 `repos` 配置段与 repo override 解析实现
- **配置热加载（Viper WatchConfig）**：通过 `Watcher` 与受控生效边界实现

同时，本设计也显式承接了今天 M1.5-M1.7 的已完成工作：

- 不重做任务执行引擎
- 不重做通知框架
- 仅将这些已有能力的配置来源统一治理并外置为配置
