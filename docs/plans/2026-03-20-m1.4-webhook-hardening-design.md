# M1.4 Webhook 接收器加固设计

**目标：** 修复代码审查中发现的所有阻塞与重要问题，使 Gitea Webhook 接收器在签名兼容性、请求体资源限制、启动配置约束、测试覆盖与文档交付上形成最小闭环。

## 背景

当前实现已完成 Gin 路由接收、事件解析与基础验签，但审查发现以下关键问题：

1. `X-Gitea-Signature` 当前仅接受 `sha256=<hex>`，与 Gitea 原生十六进制签名格式存在兼容性风险。
2. 请求体通过 `io.ReadAll` 无限制读取，存在资源耗尽风险。
3. `serve` 在未配置 `--webhook-secret` 时仍可启动，导致服务可用性表象与实际行为不一致。
4. 关键失败路径测试覆盖不足，且缺少 Gitea 侧配置指南交付物。

## 方案选择

采用“最小闭环修复”方案：

- 验签器兼容 Gitea 原生 hex 签名，并向后兼容 `sha256=` 前缀。
- Webhook 接收器增加固定请求体大小限制。
- `serve` 在 secret 为空时直接启动失败。
- 通过 TDD 补齐关键路径测试。
- 增加 Gitea Webhook 配置指南，并保持路线图与实际交付一致。

不做中间件重构、不引入额外配置项、不扩展开发模式开关，避免过度设计。

## 设计

### 1. 验签兼容性

修改 `internal/webhook/verifier.go`：

- 保持现有错误语义：`ErrMissingSignature`、`ErrSignatureMismatch`、`ErrSecretNotConfigured`。
- 新增签名规范化逻辑：
  - 如果签名以 `sha256=` 开头，则去掉此前缀后做十六进制解码。
  - 否则直接按原始十六进制字符串解码。
- 使用 `hmac.Equal` 对比计算出的 HMAC-SHA256 摘要与提供值。

这样既兼容 Gitea 原生格式，也兼容现有测试和潜在代理层传递的前缀格式。

### 2. 请求体大小限制

修改 `internal/webhook/receiver.go`：

- 引入固定常量，例如 `maxPayloadSize = 1 << 20`（1 MiB）。
- 在读取请求体前使用 `http.MaxBytesReader` 包装 `c.Request.Body`。
- 如果读取失败或请求体为空，则返回 `400 Bad Request`。
- 保持现有 content-type、验签、解析与 handler 分发逻辑不变。

这样可以在不扩大改动面的前提下，直接消除无界读取问题。

### 3. 启动配置约束

修改 `internal/cmd/serve.go`：

- 在 `runServe` 开始阶段检查 `serveWebhookSecret`。
- 若为空，直接返回错误，例如 `fmt.Errorf("webhook secret 未配置")`。
- 不再允许“服务启动成功但 webhook 永远 401”的隐式坏状态。

### 4. 测试策略

采用 TDD，先写失败测试，再实现最小代码。

#### `internal/webhook/verifier_test.go`
新增或调整以下用例：
- 原生 hex 签名可通过。
- `sha256=` 前缀签名可通过。
- 非法 hex 返回失败。
- secret 为空返回 `ErrSecretNotConfigured`。
- 缺失签名返回 `ErrMissingSignature`。

#### `internal/webhook/receiver_test.go`
新增以下用例：
- 超大请求体返回 `400`。
- 缺失签名返回 `401`。
- `issues.labeled` 正确分发到 `HandleIssueLabel`。
- `pull_request.synchronized` 正确分发到 `HandlePullRequest`。

#### `internal/cmd/serve_test.go`
新增或调整以下用例：
- `serveWebhookSecret` 为空时 `runServe` 直接返回错误。
- Webhook 路由测试改为断言明确状态码，而不是仅验证非 `404`。

### 5. 文档交付

新增 `docs/webhook-gitea.md`，内容至少包括：

- Webhook 接收路径 `/webhooks/gitea`
- 启动命令与 `--webhook-secret` 配置方式
- Gitea 端需要配置的 Secret
- 推荐订阅事件：`pull_request`、`issues`
- 当前支持动作：PR 的 `opened`、`synchronized`；Issue 的 `labeled`、`unlabeled`
- 成功与忽略行为概述

同时保留 `docs/ROADMAP.md` 中 M1.4 配置指南已完成状态，因为此交付物会被补齐。

## 验收标准

1. Gitea 原生十六进制签名请求能够通过验签。
2. 超出大小限制的请求体被 `400` 拒绝。
3. 未配置 `--webhook-secret` 时 `serve` 启动失败。
4. 新增测试覆盖上述行为并全部通过。
5. 仓库内存在可用的 Gitea Webhook 配置指南。
6. `docs/ROADMAP.md` 与实际交付保持一致。
