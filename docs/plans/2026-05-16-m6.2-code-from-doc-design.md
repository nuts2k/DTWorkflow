# M6.2 文档驱动自动编码（code_from_doc）设计文档

> 日期：2026-05-16
> 里程碑：M6.2（Phase 6 自动化闭环增强）
> 依赖：M3.4/M3.5（容器写权限 + PR 创建）、M6.1（迭代式评审修复闭环）

## 1. 目标

实现从设计文档/实施文档到具备合并资格的 PR 的全自动化能力。用户编写设计文档并 push 到仓库后，通过 CLI/API 触发自动编码任务，系统在容器中读取文档、实现功能、运行测试、创建 PR，随后自动接入已有的评审（M2）和迭代修复（M6.1）管线，形成 **文档 → 编码 → 评审 → 修复 → 可合并** 的完整闭环。

## 2. 关键设计决策

### D1：双路径统一模型

同时支持两种工作流，通过 `--branch` 参数自动路由：

| 路径 | 判断条件 | 行为 | 适用场景 |
|------|----------|------|----------|
| **路径 A** | `--branch` 省略或等于默认分支 | 从 main 派生 `auto-code/{doc-slug}` 分支 → 编码 → 创建新 PR | 设计文档已合并到主分支后触发 |
| **路径 B** | `--branch` 为非默认分支 | checkout 该分支 → 编码 → push 到同分支 → 创建 PR（如不存在） | 在功能分支上写完文档后直接触发 |

路径 B 的典型工作流：

```
git checkout -b feat/user-auth
# 编写设计文档
git add docs/plans/user-auth-design.md
git commit -m "docs: 用户认证模块设计"
git push origin feat/user-auth
dtw code-from-doc --repo myorg/backend --doc docs/plans/user-auth-design.md --branch feat/user-auth
```

Bot PR 过滤：路径 A 的 `auto-code/` 前缀加入过滤列表；路径 B 为用户分支，无需过滤（合并后触发 gen_tests/e2e 是正确行为）。

### D2：独立 API Key

code_from_doc 面向大规模代码生成，与 review/fix 的小规模操作使用不同的 Claude API 配额。新增 `claude.api_keys` 字典，按 TaskType 路由 API Key，未配置时回退 `claude.api_key`。

### D3：auto-iterate 标签灵活控制

code_from_doc 和 fix_issue 各自可配置创建 PR 时是否自动添加 `auto-iterate` 标签。开启后，PR 创建 → 评审 request_changes → M6.1 自动接管迭代修复，无需人工干预。

### D4：单容器全托管

一次容器调用完成全部操作：读文档 → 编码 → 测试 → commit → push。超时独立配置（默认 60 分钟）。与 fix_issue 的 D9（单容器全托管执行模型）保持一致。

### D5：单文档单 PR

一份设计文档对应一次容器执行、一个 PR。不做文档自动拆分。

### D6：触发方式

MVP 阶段以 **CLI/API** 为主要触发方式。Issue 标签触发作为后续扩展保留。

## 3. 整体架构与数据流

```
触发源（CLI / API）
        |
        v
  EnqueueCodeFromDoc          <-- 入队层（幂等、Cancel-and-Replace）
        |
        v
  queue.Processor             <-- 路由 code_from_doc -> codeService.Execute
        |
        v
  code.Service.Execute        <-- 核心编排
   |-- 1. 校验输入（文档路径非空、Ref 有效）
   |-- 2. 构造 prompt（stdin 传入，指引 Claude 读取仓库内文档）
   |-- 3. 容器执行（ImageFull，专用 API Key，超时 60 分钟）
   |      '-- Claude: 读文档 -> 编码 -> 测试 -> commit -> push
   |-- 4. 双层 JSON 解析 -> CodeFromDocOutput
   |-- 5. 创建 PR（Gitea API，幂等复用）
   |      '-- 可选：自动添加 auto-iterate 标签
   |-- 6. 结果持久化（code_from_doc_results 表）
   |-- 7. 可选：主动入队 review（同 gen_tests D6）
   '-- 8. 通知（飞书 + Gitea）
        |
        v
  Gitea Webhook (PR created / synchronized)
        |
        v
  review_pr                   <-- 已有管线
        |
        v
  [如有 auto-iterate 标签]
  fix_review 迭代闭环         <-- M6.1 已有
```

### 分支命名

- 路径 A：`auto-code/{doc-slug}`，doc-slug 从文档路径派生（`docs/plans/user-auth-design.md` -> `user-auth-design`）
- 路径 B：使用用户指定的分支名（如 `feat/user-auth`）

### Gitea Token 复用

| 操作 | 账号 |
|------|------|
| 容器内 push | fix 账号（与 fix_issue 一致） |
| 宿主侧创建 PR | fix 账号 PRClient |
| auto-iterate 标签操作 | review 账号（与 M6.1 一致） |

完全复用现有 `selectGiteaToken` + `buildContainerEnv` 路由机制，映射表增加 `code_from_doc -> fix token` 一行。

```yaml
  api_keys:                           # 按任务类型覆盖（可选）

```

### 路由逻辑

`internal/worker/container.go` 中 `buildContainerEnv` 新增 `resolveAPIKey(taskType)`：

1. 查 `api_keys[taskType]`，非空则使用
2. 回退 `api_key`

### 校验

- `api_keys` 中的值非空时与 `api_key` 共用相同校验规则
- 空值跳过（回退默认）

## 5. auto-iterate 标签自动添加

### 配置

```yaml
code_from_doc:
  auto_iterate: true                  # 创建 PR 时自动添加 auto-iterate 标签

fix:
  auto_iterate: false                 # fix_issue 默认不加
```

### 实现

PR 创建成功后，如果配置开启，调用 `AddIssueLabels` API 添加 `auto-iterate` 标签。使用 review 账号执行标签操作（与 M6.1 `IterateLabelManager` 一致）。

标签不存在时 API 返回错误，Service 记录 warn 继续，不影响 PR 创建。

## 6. Prompt 设计

四段式 prompt，通过 stdin 传入容器（同 Phase 2/3 安全实践）。

### 第一段：上下文

```
你正在仓库 {owner}/{repo} 的分支 {branch} 上工作。
当前代码基于 ref: {base_ref}。

你的任务是根据以下设计文档实现功能：
- 文档路径：{doc_path}

请先阅读该文档，理解需求、架构设计、接口定义和约束条件。
```

### 第二段：编码指令

```
执行步骤：
1. 阅读设计文档，理解完整需求
2. 阅读项目现有代码结构，理解编码规范和架构模式
3. 如存在 CLAUDE.md / .code-standards 等规范文件，遵循其约束
4. 按设计文档实施编码，包括：
   - 核心功能代码
   - 必要的单元测试
   - 如文档要求，补充集成测试
5. 运行测试验证（mvn test / npm test / go test ./...），失败时自动修正，最多 {max_retry_rounds} 轮
6. 每个逻辑单元完成后 commit（commit message 格式：feat: {简述}）
7. 全部完成后 git push
```

### 第三段：约束

```
- 严格按照设计文档的范围实现，不添加文档未描述的功能
- 不修改设计文档本身
- 不删除现有代码和测试（除非设计文档明确要求重构）
- 禁止访问外部网络 API
- 如果设计文档信息不足以完成实现，在输出中标记 info_sufficient=false 并列出缺失信息
```

### 第四段：输出格式

JSON schema，见第 7 节。

## 7. 输出 Schema

### CodeFromDocOutput

```go
type CodeFromDocOutput struct {
    Success        bool           `json:"success"`
    InfoSufficient bool           `json:"info_sufficient"`
    MissingInfo    []string       `json:"missing_info,omitempty"`
    BranchName     string         `json:"branch_name"`
    CommitSHA      string         `json:"commit_sha"`
    ModifiedFiles  []ModifiedFile `json:"modified_files"`
    TestResults    TestRunResults `json:"test_results"`
    Analysis       string         `json:"analysis"`
    Implementation string         `json:"implementation"`
    FailureCategory FailureCategory `json:"failure_category"`
    FailureReason   string          `json:"failure_reason,omitempty"`
}

type ModifiedFile struct {
    Path        string `json:"path"`
    Action      string `json:"action"`      // created / modified / deleted
    Description string `json:"description"`
}

type TestRunResults struct {
    Passed    int  `json:"passed"`
    Failed    int  `json:"failed"`
    Skipped   int  `json:"skipped"`
    AllPassed bool `json:"all_passed"`
}
```

### 失败分类

```go
type FailureCategory string

const (
    FailureCategoryNone             FailureCategory = "none"
    FailureCategoryInfoInsufficient FailureCategory = "info_insufficient"
    FailureCategoryTestFailure      FailureCategory = "test_failure"
    FailureCategoryInfrastructure   FailureCategory = "infrastructure"
)
```

| 分类 | 处理策略 |
|------|----------|
| `none` | 成功，`Success` 必须为 `true` |
| `info_insufficient` | SkipRetry，通知用户补充文档 |
| `test_failure` | 仍创建 PR（保留已有成果），通知包含失败详情 |
| `infrastructure` | 允许 asynq 重试 |

### 成功不变量

`Success=true` 时 `BranchName` 和 `CommitSHA` 必须非空，`FailureCategory` 必须为 `none`。

## 10. 通知

三事件（与 gen_tests/e2e 一致模式）：

| 事件 | 飞书卡片颜色 | 内容 |
|------|-------------|------|
| `EventCodeFromDocStarted` | 蓝色 | 仓库/分支/文档路径 |
| `EventCodeFromDocDone` | 绿色 | PR 链接 + 修改文件数 + 测试结果 |
| `EventCodeFromDocFailed` | 红色/橙色 | failure_category + failure_reason |

`failure_category` -> severity 映射：`infrastructure` = Warning，`test_failure` = Info，`info_insufficient` = Info。

code_from_doc 属 repo 级通知，复用 `repoScopedGiteaNotifier` 包装。

## 12. entrypoint.sh

新增 `code_from_doc)` case，复用 fix_issue 的凭证配置：

```bash
code_from_doc)
    # credential helper + git identity（同 fix_issue）
    setup_credential_helper
    setup_git_identity

    # 构建缓存重定向（同 fix_issue）
    redirect_build_cache

    # 分支策略
    if [ -n "${CODE_BRANCH}" ]; then
        # 路径 B：checkout 已有分支
        git fetch origin "${CODE_BRANCH}"
        git checkout "${CODE_BRANCH}"
    else
        # 路径 A：从 BASE_REF 派生新分支
        git checkout -b "auto-code/${DOC_SLUG}" "${BASE_REF}"
    fi

    # 环境变量清除（同 fix_issue）
    cleanup_sensitive_env
    ;;
```

环境变量：
- `CODE_BRANCH`：路径 B 的目标分支（路径 A 为空）
- `DOC_SLUG`：路径 A 的分支后缀（从 doc_path 派生）
- `BASE_REF`：基础 ref
- `DOC_PATH`：文档路径（prompt 中使用，非 entrypoint 直接使用）

## 14. 与现有管线的集成点

| 集成点 | 机制 | 说明 |
|--------|------|------|
| 自动评审 | Gitea Webhook PR created/synchronized | 自然触发 review_pr |
| 主动入队 review | `EnqueueManualReview`（同 gen_tests D6） | 减少 Webhook 延迟 |
| 迭代修复闭环 | `auto-iterate` 标签 + M6.1 | 配置控制是否自动添加标签 |
| Bot PR 过滤 | `auto-code/` 前缀（路径 A）| 防止 gen_tests/e2e 自触发 |
| Cancel-and-Replace | 按 (repo, branch) 聚合 | 同分支重复触发时取消旧任务 |

## 15. 安全考量

- **错误脱敏**：`CodeFromDocOutput` 自由文本字段经 `SanitizeErrorMessage` 处理，防 prompt-injection 内容泄露到飞书/PR
- **API Key 隔离**：code_from_doc 使用独立 Key，不影响 review/fix 配额
- **容器凭证**：复用 fix_issue 的 credential helper 脚本（权限 700，随容器销毁），origin URL 脱敏
- **Prompt 安全**：保留 prompt 层网络访问禁令；不加 `--disallowedTools` Bash 限制（需运行测试）
- **文档路径校验**：`doc_path` 必须为相对路径，禁止路径遍历（`..`）、绝对路径

## 16. 验证标准

### 路径 A（从主分支派生）

- 主分支上已有设计文档 -> `dtw code-from-doc --repo X --doc docs/plans/xxx.md` -> 创建 `auto-code/{slug}` 分支 -> 容器编码 -> 创建 PR -> 飞书通知
- PR 带 `auto-iterate` 标签 -> 评审 request_changes -> M6.1 自动修复 -> 迭代直到通过或超限

### 路径 B（在功能分支继续）

- 功能分支上有设计文档 -> `dtw code-from-doc --repo X --doc docs/plans/xxx.md --branch feat/xxx` -> checkout 该分支 -> 编码 -> push -> 创建 PR（如不存在）-> 飞书通知
- 分支已有 open PR -> push 触发 synchronized -> review_pr 重新评审

### 通用验证

- 设计文档信息不足 -> Claude 返回 `info_sufficient=false` -> 通知用户补充 -> SkipRetry
- 编码完成但测试失败 -> 仍创建 PR（保留成果）-> 通知包含失败详情
- 容器执行失败 -> asynq 重试（最多 3 次）
- 同分支重复触发 -> Cancel-and-Replace 取消旧任务
- 独立 API Key 生效 -> 容器内 `ANTHROPIC_API_KEY` 为 code_from_doc 专用 Key
- `auto_iterate: false` 时 PR 不添加标签 -> 不进入迭代闭环
