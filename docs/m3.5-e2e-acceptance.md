## 验收项目

### 验收 1：fix-to-pr 完整闭环（核心）

**场景**：在 `changyu/ai_workflow_test` 创建含明确 bug 描述的 Issue → 打 `fix-to-pr` 标签

**断言**：
- [ ] 容器启动使用 `worker-full` 镜像（查 `docker ps` 或服务日志）
- [ ] Claude 完成：分析 + 代码修改 + 测试运行 + git push
- [ ] `fix.Service` 创建 PR，`fixes #N` 关联 Issue
- [ ] Issue 收到 "✅ 已创建修复 PR #X" 评论
- [ ] PR 描述含根因分析、修复方案、修改文件列表、测试结果

### 验收 2：修复 PR 自动触发评审

**断言**：
- [ ] 验收 1 创建的 PR 被 Gitea Webhook 触发 `review_pr` 任务（Phase 2 零代码自动接入）

### 验收 3：无前序分析 → Claude 隐式分析后修复

**场景**：新 Issue 直接打 `fix-to-pr`（未先经过 `auto-fix` 分析）

**断言**：
- [ ] 容器正常启动，Claude 在 prompt 驱动下先理解问题再修复

### 验收 4：前序分析"信息不足" → 阻断

**场景**：Issue 先打 `auto-fix` 得到 `info_sufficient=false` 的分析 → 再打 `fix-to-pr`

**断言**：
- [ ] 容器**不**启动（查 `docker ps` 无新容器 + 任务日志输出 "前序分析标记为信息不足"）
- [ ] Issue 收到 "## DTWorkflow Issue 修复前置检查" 评论
- [ ] 任务状态 failed + `ErrInfoInsufficient`

### 验收 5：测试失败 → Claude 返回 success=false

**场景**：故意构造无法修复的 Issue（要求删除关键逻辑）

**断言**：
- [ ] Issue 收到 "## DTWorkflow Issue 修复失败" 评论，含 `failure_reason`
- [ ] 任务状态 failed + `ErrFixFailed`，不重试

### 验收 6：Push 成功但 PR 创建失败

**场景**：（难以自然复现）临时在 serve.go 中注入一个 mock `PRClient` 返回 error；或使用无 pulls 写权限的 Token

**断言**：
- [ ] Issue 收到 "⚠️ 修复分支 ... 已推送成功，但 PR 创建失败" 评论
- [ ] 任务允许 asynq 重试

### 验收 7：分支已存在 → force-with-lease 覆盖

**场景**：手动在目标仓库创建 `auto-fix/issue-X` 分支 → 触发 `fix-to-pr` 修复同一 Issue

**断言**：
- [ ] Claude 执行 `git push --force-with-lease` 覆盖，PR 正常创建

### 验收 8：Issue Ref 为 tag

**场景**：Issue 的 "Reference" 字段设为 tag（如 `v1.0.0`）→ 触发 `fix-to-pr`

**断言**：
- [ ] PR 的 Base 为仓库默认分支（非 tag）
- [ ] PR 描述注明 "原 Issue 指定的 ref 为 tag，Gitea PR 不支持 tag 作为 base 分支"

### 验收 9：飞书通知三场景

**断言**：
- [ ] 蓝色 "Issue 自动修复开始"（任务入队时）
- [ ] 绿色 "Issue 修复 PR 已创建"（成功时，带 "查看修复 PR" 按钮）
- [ ] 红色 "Issue 自动修复任务失败"（失败时）

### 验收 10：dtw fix-issue --fix 远程触发

**命令**：`./bin/dtw fix-issue --owner changyu --repo ai_workflow_test --issue N --fix`

**断言**：
- [ ] 服务端创建 `fix_issue` 任务，执行路径与标签触发完全一致

### 验收 11：并发修复 Cancel-and-Replace

**场景**：连续两次触发同一 Issue 的 `fix-to-pr`

**断言**：
- [ ] 旧任务被取消，只有最新任务执行

> 若 M3.4 未完整接入 Cancel-and-Replace 机制，此项标注延后。
