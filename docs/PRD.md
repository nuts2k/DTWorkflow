# DTWorkflow 产品需求文档 (PRD)

> 基于 Claude Code 的 Gitea 自动化工作流平台

## 1. 项目概述

### 1.1 背景

团队使用自部署 Gitea 进行代码管理，现有流程中 PR 评审依赖人工、Issue 修复效率受限于个人经验、测试覆盖不完整。希望借助 Claude Code 的代码理解和生成能力，构建一套自动化工作流平台，提升团队研发效能。

### 1.2 目标

- 自动化 PR 评审，作为人工 review 前的第一道关卡
- 自动化 Issue 根因分析与修复 PR 生成
- 补全并扩展测试体系，包含 AI 驱动的 E2E 模拟用户测试
- 统一通知机制，确保关键事件及时触达相关人员

### 1.3 项目范围

| 范围内 | 范围外 |
|--------|--------|
| Gitea 仓库的 PR 自动评审 | 跨仓库关联分析 |
| 基于标签触发的 Issue 自动修复 | Gitea 以外的代码托管平台 |
| 单元测试 / 集成测试补全 | 性能测试、压力测试 |
| AI 驱动的 E2E 模拟用户测试 | 移动端 App 测试 |
| 可扩展的通知框架 | 即时通讯机器人深度定制 |

### 1.4 团队与环境

- 团队规模：约 10+ 人
- 活跃仓库：不超过 10 个
- 技术栈：后端 Java，前端 Vue
- Gitea：内网部署，具备管理员权限，支持 Webhook
- 部署：专用机器（Debian），网络与 Gitea 互通
- 开发环境：macOS，需保证 macOS / Linux 双平台可运行

## 2. 系统架构

### 2.1 接入层设计

系统采用 **CLI-first，API 并行** 的接入设计。核心逻辑与接入层分离，CLI 和 API 都是薄壳调用层：

```
  其他 Agent / 脚本 / 人工          Gitea Webhook
        │            │                    │
        ▼            ▼                    ▼
       CLI          API            Webhook 接收器
        │            │                    │
        └─────┬──────┴────────────────────┘
              ▼
         核心逻辑层
```

**CLI 设计原则**：
- 所有核心能力均通过 CLI 命令暴露
- 支持 `--json` 标志输出结构化 JSON，方便 agent 解析；默认人类可读格式
- 退出码规范：0 成功 / 1 失败 / 2 部分成功
- 所有命令支持非交互模式，参数完全通过 flags 传入

**CLI 命令示例**：
```bash
# PR 评审
dtworkflow review-pr --repo myrepo --pr 42 --json

# Issue 分析修复
dtworkflow fix-issue --repo myrepo --issue 15

# 任务状态查询
dtworkflow task status <task-id> --json

# 手动触发测试补全
dtworkflow gen-tests --repo myrepo --module service
```

初期不做 Web 管理界面。配置通过 YAML 文件管理，状态查询通过 CLI 或 API。

### 2.3 隔离策略

采用 **Docker 容器 + Git Worktree** 双层隔离：

- **Docker 容器**：每个任务在独立容器中执行，预装 Claude Code CLI、Git、Java/Node.js 工具链
- **Git Worktree**：容器内使用 worktree 切换到目标分支，避免完整 clone 的开销
- **Ref 关联**：Issue 修复任务支持通过 `ISSUE_REF` 环境变量指定代码基线（分支或 tag），entrypoint 自动 checkout 到该 ref，确保分析和修复在正确的代码版本上进行
- **资源限制**：每个容器设置 CPU/内存上限，防止单任务耗尽宿主机资源

### 2.4 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 开发语言 | Go | 单二进制分发、毫秒级 CLI 启动、原生 Docker SDK、低资源占用 |
| CLI 框架 | Cobra | Go 生态最成熟的 CLI 框架，支持子命令、自动补全、帮助生成 |
| HTTP 服务 | Gin | 高性能 HTTP 框架，用于 API 和 Webhook 接收 |
| 任务队列 | asynq + Redis | Go 原生任务队列，基于 Redis，支持优先级/重试/定时任务 |
| 数据持久化 | SQLite（modernc.org/sqlite，纯 Go） | 零 CGO 依赖、支持交叉编译，后期可迁移至 PostgreSQL |
| 容器管理 | Docker SDK for Go | Docker 官方 Go SDK，一等公民支持 |
| Gitea 交互 | Gitea REST API | 官方 Swagger 文档完善，覆盖所有需要的操作 |
| 执行引擎 | Claude Code CLI | 内置代码读取/编辑/Git 操作能力 |
| 配置管理 | Viper | Go 生态标准配置库，支持 YAML、环境变量、热加载 |
| 通知框架 | 策略模式插件化 | 初期实现 Gitea 评论，后续插拔式扩展 |

## 3. 功能需求

### 3.1 PR 自动评审

#### 3.1.1 触发条件

- Gitea Webhook：PR 创建时触发
- Gitea Webhook：PR 更新（新 push）时重新触发

#### 3.1.2 评审维度

| 维度 | 说明 | 严重程度 |
|------|------|---------|
| 代码风格/规范 | 命名规范、格式、注释质量 | INFO / WARNING |
| 逻辑正确性 | 潜在 bug、边界条件、空指针、异常处理 | WARNING / ERROR |
| 安全漏洞 | SQL 注入、XSS、硬编码密钥、不安全的反序列化等 | ERROR / CRITICAL |
| 架构合理性 | 设计模式、分层、职责划分、耦合度 | INFO / WARNING |

严重程度分级：
- **CRITICAL**：安全漏洞、数据泄露风险 → 必须打回
- **ERROR**：逻辑错误、明显 bug → 打回
- **WARNING**：代码质量问题、潜在风险 → 建议修改，不强制打回
- **INFO**：风格建议、优化建议 → 仅供参考

#### 3.1.3 评审输出

1. **逐行 Comment**：针对 CRITICAL / ERROR / WARNING 级别的问题，在 PR diff 中对应行添加 review comment，标注严重程度和修改建议
2. **整体 Summary**：在 PR 中发布一条总结评论，包含：
   - 评审概览（检查了哪些文件、发现多少问题）
   - 按严重程度分类的问题列表
   - 总体评价与建议
3. **自动打回**：当存在 CRITICAL 或 ERROR 级别问题时，自动提交 "Request Changes" review

#### 3.1.4 评审上下文

- 读取 PR 的完整 diff
- 读取变更文件的上下文代码（非仅 diff 部分）
- 读取项目的编码规范文件（如存在）
- 对于大型 PR（文件数 > 阈值），分批处理并合并结果

#### 3.1.5 用户故事

> 作为开发者，我创建一个 PR 后，希望在 5 分钟内收到 Claude 的自动评审结果，包含逐行批注和整体总结。如果有严重问题，PR 被自动标记为需要修改，我可以根据评审意见进行修复后重新提交。

### 3.2 Issue 自动分析与修复

#### 3.2.1 触发条件

- Issue 被添加 `auto-fix` 标签时触发
- 同一 Issue 重复添加标签不重复触发（幂等性）

#### 3.2.2 分析流程

1. **信息采集**
   - 读取 Issue 标题、描述、评论
   - 读取 Issue 关联的 Ref（分支或 tag），作为代码基线；Ref 缺失或无效时自动在 Issue 中回复提醒用户设置
   - 如果信息不足（无错误日志、无复现步骤），自动在 Issue 中回复追问，并暂停等待补充
   - 关联已有的类似 Issue（同仓库内）

2. **根因定位**
   - 基于 Issue 描述中的关键词（类名、方法名、错误信息）搜索代码库
   - 分析相关代码的调用链
   - 输出根因分析报告（定位到文件、方法、行号级别）

3. **修复生成**
   - 基于根因分析，在 Issue 指定的 Ref 上创建修复分支并生成修复代码
   - 修复代码需通过现有测试（如有）
   - 生成修复说明，解释改动原因

4. **PR 创建**
   - 自动创建修复 PR，关联原 Issue
   - PR 描述中包含：根因分析、修复方案、影响范围
   - PR 标记为需要人工 review

#### 3.2.3 信息不足处理

当 Issue 描述过于简短或缺少关键信息时：
- 自动在 Issue 中评论，列出需要补充的信息（错误日志、复现步骤、影响版本等）
- 将 Issue 标记为 `waiting-info`
- 当有新评论补充信息后，自动恢复分析流程

#### 3.2.4 用户故事

> 作为测试人员，我在 Issue 上添加 `auto-fix` 标签后，Claude 会自动分析根因并创建修复 PR。如果我的 Issue 描述信息不足，Claude 会自动追问我需要补充什么。

### 3.3 集中化测试

#### 3.3.1 测试补全

**触发方式**：定时任务（如每周）或手动触发

**流程**：
1. 分析项目现有测试覆盖率
2. 识别缺失测试的关键模块/方法
3. 按优先级生成测试用例代码
   - Java 后端：JUnit 单元测试 + 集成测试
   - Vue 前端：Vitest/Jest 单元测试
4. 创建 PR 提交测试代码，需人工 review

#### 3.3.2 变更驱动测试

**触发方式**：PR 合并后自动触发

**流程**：
1. 分析合并的代码变更 diff
2. 判断是否有对应测试覆盖
3. 如无，自动生成针对变更的测试用例
4. 创建独立 PR 提交测试代码

#### 3.3.3 AI 驱动 E2E 模拟用户测试

**触发方式**：定时任务 / PR 合并到主分支后触发

**流程**：
1. 部署到测试环境（需对接现有部署流程）
2. AI 基于需求文档和页面结构，规划测试路径
3. 使用 Playwright 等工具执行 E2E 测试
4. 测试失败时：
   - 自动在 Gitea 创建 Issue，附带失败截图、操作路径、错误信息
   - 通知相关人员

#### 3.3.4 用户故事

> 作为技术负责人，我希望每周自动检查测试覆盖率并补全关键测试。当 PR 合并后，如果变更没有对应测试，系统能自动生成。同时 E2E 测试能在每次发版前自动跑一轮，发现问题自动报 Issue。

### 3.4 通知框架

#### 3.4.1 通知事件

| 事件 | 默认通知方式 | 通知对象 |
|------|------------|---------|
| PR 评审完成 | Gitea PR 评论 | PR 作者 |
| PR 被自动打回 | Gitea PR 评论 + 扩展渠道 | PR 作者 |
| Issue 分析完成 | Gitea Issue 评论 | Issue 创建者 |
| Issue 信息不足追问 | Gitea Issue 评论 | Issue 创建者 |
| 修复 PR 已创建 | Gitea Issue 评论 + PR 通知 | Issue 创建者 + 仓库维护者 |
| E2E 测试失败 | Gitea Issue + 扩展渠道 | 仓库维护者 |
| 系统异常 | 扩展渠道 | 系统管理员 |

#### 3.4.2 框架设计

- 采用策略模式，每种通知渠道为一个插件
- 初期实现：Gitea 评论（内置）
- 后续扩展：企业微信 / 钉钉 / 飞书 / 邮件
- 支持按仓库、按事件类型配置通知渠道和接收人

## 4. 非功能需求

### 4.1 性能

- PR 评审响应时间：小型 PR（<10 文件）5 分钟内，大型 PR（>30 文件）15 分钟内
- Issue 分析：30 分钟内输出根因报告
- 并发任务数：至少支持 3 个任务并行执行

### 4.2 可靠性

- 任务失败自动重试（最多 3 次）
- 重试间隔指数退避
- 任务超时自动终止并通知
- 所有任务执行记录可追溯

### 4.3 安全性

- Claude API Key 加密存储，不暴露在日志中
- Gitea Token 最小权限原则
- Docker 容器以非 root 用户运行
- 容器网络隔离，仅允许访问 Gitea 和 Claude API

### 4.4 可配置性

- 每个仓库可独立配置：评审规则、触发条件、通知渠道
- 全局配置：并发数、超时时间、重试策略
- 配置变更无需重启服务（热加载）

### 4.5 可观测性

- 任务执行日志持久化
- 关键指标监控：任务成功率、平均耗时、队列深度
- 异常告警
